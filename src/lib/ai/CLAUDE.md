# AI Integration Layer

Provider-agnostic LLM interface with tiered prompt architecture. The AI generates narrative and soft lore (Tier 2/3) while the rules engine handles hard mechanics (Tier 1).

## Structure

- `types.ts` — All AI-related type definitions: providers, responses, prompts, memory, game state. `Continent` (`"central"|"forsaken-land"|"southern-continent"` — the third added in world build-out 9, issue #138, the freely-reachable colonized Southern Continent across the Berserk Sea, gated by distance not a flag) and `AccessFlag` (`"dream-world-passage"` | `"silver-city-passage"` | `"moon-city-passage"`) + `ACCESS_FLAGS`/`isAccessFlag` (world build-out, issues #130/#133) model the access-gated continents and the per-city awareness gating within the Forsaken Land (a City-of-Silver native and a Moon City native are mutually unaware, so each holds only their own city's flag plus the shared dream-world passage); `GameState.accessFlags?: AccessFlag[]` is the NOT-AI-mutable list of held capability flags (optional + strictly validated + preserved on deserialize, exactly like `customLocations`; lives in the `game_sessions.data` jsonb — no migration). `GameState.customLocations?: CustomLocation[]` (Backlund location sync) is the engine-maintained, NOT-AI-mutable registry of narrator-named venues not in the static gazetteer, filed under their city so the map can render/pin them (the pure engine lives in `@/lib/game/location.ts`; the movement bullet in `prompts.ts` asks the narrator to lead a `location` with the city name so new venues file under the right city). `StateChange.involuntaryCause?: string` (issue #101) is the optional against-the-will relocation code, loosely typed at the boundary and carried through `parseAIResponse` only when a string — the engine (`isInvoluntaryMoveCause` in `@/lib/game/world-state`) narrows it to a known cause and lets an otherwise-refused cross-city `location` move through (the prompt's movement rule advertises it). `AIResponse.journalEntry` (issue #11) is the AI's optional journal-worthy flag — loosely typed at the boundary; `validateJournalFlag` in `@/lib/game/journal` is the single validation point. `AIResponse.proposedSelfChange` (character-info storage) is the same shape of loose flag for a true-self change the narrator detected the player declaring — whitelisted through `parseAIResponse`, validated by `validateSelfChangeProposal` (`@/lib/game/profile`), and NEVER applied without an explicit player confirm (it is deliberately separate from the AI-mutable `worldStateChanges` allowlist). `PromptInput.profileContext`/`recognitionContext` are the True-Self ground-truth and recognition-gap narrator layers. **Story-consistency Codex (history-context Codex):** `AIResponse.codexUpdates?: CodexUpdateInput[]` is the narrator's per-turn entity-DELTA channel (introduce/update a person/location/object/group/thread) — loosely typed at the boundary like `journalEntry`; `applyCodexUpdates` (`@/lib/game/codex`) is the single validation point. `PinnedCodexEntity` is the structural slice of a `CodexEntity` the prompt assembler consumes (declared here, like `RetrievedLoreChunk`, so the AI layer never imports the game layer); `PromptInput.pinnedEntities` carries the engine-selected, scene-relevant + pivotal subset into the `## Established Facts` history block. `AIResponse.fundsDiscovered` (issue #16 follow-up) is the optional pence-found-this-turn channel — whitelisted in `parseAIResponse` as a finite number, clamped + applied in `world-state.ts` (NOT the AI-mutable allowlist). **Hybrid sanity + acting-method discovery (issue #95):** `AIResponse.sanityImpact` is now only a small residual nuance tightened to **±5**; the bulk of per-turn sanity comes from `AIResponse.sanityEventTags?: string[]` (the engine computes magnitudes via `sanityDeltaForTags`), normalized in `parseAIResponse` to the known set (unknown tags dropped). `AIResponse.actingMethodTaught?: boolean` is the drop-not-throw flag (carried only when literally `true`) for the AI signalling the acting method was explicitly taught/revealed this turn — the only AI-driven discovery trigger, never a sanity scare. `AIResponse.pursuers?: string[]` (issue #101) is the narrator's channel for who is HUNTING the character — `parseAIResponse` normalizes it to clean, de-duplicated, non-empty names (drop-not-throw, empty dropped), and the game loop rosters each as a hostile follower via `joinRoster` (idempotent), so a pursuer reappears at the player's destination after travel until shaken off. Companions are the PLAYER's choice (the character sheet), never this field. **Narration verbosity (verbosity preset):** `NarrativeVerbosity` (`"concise"|"standard"|"rich"`) + the runtime `NARRATIVE_VERBOSITY_VALUES` and `isNarrativeVerbosity` guard are the single source of truth the game-layer `preferences.ts` imports for its `narrativeVerbosity` field; `PromptInput.verbosity`/`GenerateOptions.verbosity` carry the player's pick into the prompt (absent/"standard" = baseline).
- `errors.ts` — `AIError` class with unified error codes: AUTH_ERROR, RATE_LIMITED, QUOTA_EXCEEDED, PROVIDER_ERROR, MALFORMED_OUTPUT, VALIDATION_FAILED, NETWORK_ERROR, TOKEN_LIMIT_EXCEEDED. `AIError` also carries the originating HTTP `status` (when from `classifyHttpError`) so callers can distinguish 401 (bad key) from 403 (key fine, model/request refused) without string-matching, plus the already-extracted `reason` (the provider's bare message) so consumers read `err.reason` instead of re-parsing the body. `classifyHttpError` extracts the reason once and threads `status`/`reason` onto every error it builds. `classifyHttpError` maps **401** to the "invalid/expired key" message but **403** to a distinct model-unavailable/quota message (both `AUTH_ERROR`, so the UI keeps the Settings CTA) — a 403 means the key worked and the request was refused, so it must not blame the key. **400/404** map to a non-retryable `AUTH_ERROR` (model id invalid/unavailable, or wrong base URL) — deterministic config faults that must not burn the retry backoff, while keeping the Settings CTA without blaming the key. **Every** branch surfaces the provider's own wording via a `withReason` helper (so the message reads e.g. `Unexpected HTTP 400. Provider said: model: … not found`) — this is the only way to see the cause for browser-direct providers, which have no server-side log. The wording comes from `extractProviderMessage(body)` (exported) — e.g. ollama.com's "this model requires a subscription, upgrade for access: …"; `extractProviderMessage` reads `{error}` / `{error:{message}}` / `{message}` JSON shapes and falls back to capped raw text for non-JSON bodies.
- `providers.ts` — Provider adapters: Anthropic, OpenAI, OpenRouter, Ollama, Ollama Cloud, Custom. Each implements `LLMProviderAdapter` interface (incl. `listModels(apiKey)` for the live model catalog). Factory: `createAdapter(providerId, baseUrl?)`. The Anthropic adapter forces JSON via a system-prompt directive (no `response_format` support); it must NOT prefill the assistant turn — newer Claude models (Opus 4.6+, Sonnet 4.6, Fable 5) reject a trailing assistant turn with HTTP 400 ("does not support assistant message prefill"), and the forgiving parser + corrective retry loop cover the looser guarantee. `parseResponse` sets `truncated` from the provider's stop/finish reason. `inferModelTier(id)` guesses a tier for dynamically-listed models. `OllamaAdapter` also exposes `embed(model, texts, apiKey)` over `/api/embed` (issue #60) — the one HTTP embed call both embedding transports use. `OllamaAdapter.makeRequest` sets `options.num_ctx` via the pure `ollamaNumCtx(messages, maxTokens)` helper (prompt-token estimate + output cap + headroom, rounded up to a 1024 step, floored at `OLLAMA_MIN_NUM_CTX` 4096) — Ollama otherwise runs the model at its small default context, so a large RAG-retrieval prompt would fill the window and leave no room to generate (the model emits 2-3 tokens and stops with `done_reason "length"`, surfacing as "unparseable provider output"). Only Ollama needs this; the hosted providers manage context server-side.
- `embeddings.ts` — Embedding seam (issue #60). The `EmbeddingProvider` interface (`id`, `dims`, `model_id`, `embed(texts)`) is a sibling to `LLMProviderAdapter`: the embedding model is a fixed, approved choice locked per save (a query vector only compares to corpus vectors from the same model's "map"), not a free-form pick. `APPROVED_EMBEDDING_MODELS` = two 1024-dim models, `qwen3-embedding-0.6b` (**default**) + `bge-m3`, each pinned by an explicit `ollamaTag` AND a required `cloudflareModel` (`@cf/qwen/qwen3-embedding-0.6b`, `@cf/baai/bge-m3`) — **both** models are served by every transport, so any save resolves on any transport. `createEmbeddingProvider({id, modelId?, baseUrl?, apiKey?})` builds a provider for one transport (`embed()` validates batch count + 1024-dim shape via the shared `assertEmbeddingShape`): `ollama` = the player's own Ollama (BYO, browser-direct, mirrors BYOK) and `operator` = a self-hosted always-on CPU box at `NEXT_PUBLIC_OPERATOR_EMBEDDING_URL` both speak Ollama's `/api/embed` (`OllamaEmbeddingProvider`); `cloudflare` = the operator's **Cloudflare Workers AI** endpoint running our open models on its free tier (`CloudflareEmbeddingProvider`). The CF provider is model-aware: against the same-origin proxy (relative path) it POSTs `{text, model}` (the approved id) and the proxy maps id → CF model id + injects `CF_ACCOUNT_ID`/`CF_API_TOKEN`; against an absolute CF run base (CI) it appends `/<cloudflareModel>` to the URL and POSTs `{text}` with the token. Browser default endpoint is `CLOUDFLARE_EMBEDDING_PROXY_PATH` (`/api/proxy/cloudflare/embed`); no Worker to deploy — the REST API needs only an account id + a Workers-AI-scoped token. **Why Cloudflare, not Ollama Cloud:** ollama.com hosts only chat models, never embedding models, so its `/api/embed` 401s (web-researched); Cloudflare hosts BOTH our already-approved models (so no model change / no new map — the corpus already has both maps), 1024-dim, free tier, no training on inputs.
- `prompts.ts` — Layered prompt assembly: system prompt, sanity-narration directive, lore context, game state, history, instruction. **Backstory sequence-claim rule (issue #183):** `buildSystemPrompt()` includes a rule immediately after the sequence-advancement boundary: backstory sequence claims are UNRELIABLE NARRATIVE — the narrator treats any sequence number in the player's background text as a story detail or aspiration, never an established game fact; the character's actual sequence is always and only what the engine reports in `## Character & Origin`. This prevents a player from bootstrapping a higher-power level via a backstory claim. The UI layer enforces the same boundary via `validateBackstorySequence` (`@/lib/game/backstory-validation.ts`), blocking the "Begin Your Chronicle" button while violations exist. **Movement (issue #101):** the `worldStateChanges` schema advertises the optional `involuntaryCause`, and a Rules bullet (tone matching the "Sequence advancement is owned by the rules engine" boundary) tells the narrator to move within a city freely, never relocate cross-city for pacing, signal an against-the-will cross-city move with `involuntaryCause`, and that a cross-city change without one is refused (the character stays put). **Issue #95:** the schema/rules now advertise `sanityEventTags` and `actingMethodTaught`, cap `sanityImpact` at ±5 (the stale -16..-20 guidance is gone), and **soften the acting rule** — the AI scores acting honestly but is told NOT to name/explain the "digestion"/"acting method" mechanic (it is secret knowledge) and to set `actingMethodTaught` only for an explicit in-fiction teaching, never a sanity scare. The game state still feeds `potionDigestion: X%` to the AI. The system prompt carries a **Canon Fidelity** section (follow provided lore/retrieved material exactly; when uncovered, invent only mundane, world-consistent texture — never fabricate Beyonder-tier canon — and prefer ambiguity to a confident wrong answer) and a hardened **Running Summary** rule (record only what play/lore established — no new canonical names/orgs/deities/history harden into the durable synopsis). **Story-memory consistency:** the Running Summary rule now asks for a LABELLED, structured synopsis (`Who`/`Situation`/`Goals`/`Ties`/`Threads` lines, overwrite-don't-linger) rather than free prose — structured fields drift less — and tells the narrator the `## Ground Truth` history anchor wins on any conflict. `buildHistoryPrompt(memory, gameState?, pinnedEntities?)` pins that compact `## Ground Truth (authoritative)` block (current location + present NPCs) at the TOP of the history layer, right above `## Story So Far`, so the engine's truth overrides a drifted synopsis; alongside it `buildEstablishedFacts` renders the **`## Established Facts (canon — do not contradict)`** block from `input.pinnedEntities` (the engine's scene-relevant + pivotal Codex subset — history-context Codex), a stable per-entity index so a long story can't contradict an established person/place/thread; the system prompt's `## Codex` section tells the narrator to emit `codexUpdates` deltas that feed it. Both blocks share the "history exists" gate (absent on turn 0) and are bounded — the pinned subset is hard-capped by `selectPinnedEntities`, so the per-turn budget stays flat; `assemblePrompt` threads `input.gameState` in (the anchor only appears once there is history). Token budget rebalanced for continuity (**11,500 total**): `history` **1,000 → 4,500** (the continuity bottleneck — it now holds the widened 8-turn immediate window + the durable synopsis + recent bullets without re-trimming), `gameState` **1,000 → 1,200**, `lore` **4,000 → 3,000** (prompt-cached, not the bottleneck — partly offsets the rise), `system` 2,500, `instruction` 300; ~+2,700 net input tokens/turn, negligible against every provider window and cents/turn with caching ($0 on local Ollama). `buildLoreContext(loreContext, retrievedChunks?)` injects curated guardrail entries FIRST and in full (`narratorOnly`-aware), then packs gated retrieved chunks (`RetrievedLoreChunk`, threaded via `PromptInput.retrievedChunks`) into whatever lore budget remains via `selectRetrievedForBudget` (first-fit in retrieval-rank order) — authored lore is never crowded out. Retrieved text is framed `[NARRATOR ONLY]` with a no-verbatim-quoting directive. Optional system layers: `## Epoch` (issues #26/#29, `PromptInput.epochContext`), `## Active Identity` (issue #22, `identityContext`), `## True Self` (character-info storage, `profileContext` — ground-truth self facts), and `## Recognition` (character-info storage, `recognitionContext` — who knew the character before a drastic change and doesn't recognise them now). `buildGameStatePrompt` also appends a durable `## Character & Origin` section (the character's `characterBackground` + `prologueRecap`) inside the never-trimmed game-state layer — the prologue → story bridge, so the backstory and the prologue the character just lived through keep shaping narration for the whole chronicle instead of aging out of the history window. **Canon-character takeover (issue #92):** when `GameState.canonCharacterId` is set, `buildGameStatePrompt` appends a `## You Are This Character` directive ("the player character IS &lt;name&gt; … never portray them as a separate present character") — closing the residual third-person-narration gap left after the world-state `npcsPresent` suppression. When `GameState.canonPersonality` is also present, the directive adds a personality-bias clause telling the narrator to shape the choices it OFFERS toward how the figure would characteristically act (while the player stays free to act against type and an in-character action is never forced). `GameState.canonCharacterId?: string` / `GameState.canonPersonality?: string` (in `types.ts`) are rules-engine-only takeover markers (NEVER AI-mutable; preserved on the `...gs` spread). Per-turn cost quantified in `docs/rag-per-turn-budget.md`; note retrieved chunks DO reach the browser (BYOK browser-direct) — risk-accepted, never claim otherwise. `buildSanityDirective(gameState)` appends an unreliable-narration directive (from `sanity.ts`) when sanity is below the high tier; at high sanity it returns empty content and the layer is dropped. `buildDemigodDirective(gameState)` (issues #25, #35) mirrors that pattern: at Sequence ≤ `DEMIGOD_SEQUENCE_THRESHOLD` (4 — Saint and above) it adds a `## Demigod Stakes` system layer scaling narration to cosmic consequences, divine politics, and churches/organizations reacting; below the tier it returns empty content and the layer is dropped. At **Sequence 0** (issue #30) the same function returns a `## Godhood` layer instead — the narration shifts entirely from human-scale to a True God's cosmic existence (prayers as voices, action at continental range, divine politics, a human heart inside a god). Optional `## Setting Tone` system layer (issue #23, `PromptInput.cityNarration` from `cityNarrationDirective`) gives each city its own one-sentence narration tone; dropped (`null`/absent) for cities with no specific tone. **Narration length + pacing (verbosity preset):** `buildVerbosityDirective(verbosity?)` emits a `## Narration Length` system layer for `"concise"`/`"rich"` (empty for `"standard"`/absent → dropped, like `buildSanityDirective`) and applies to EVERY instruction incl. combat; its phrasing comes from the exported `VERBOSITY_GUIDANCE` map — the **single source** the prologue's `prologueVerbosityLine` (`prologue-client.ts`) ALSO consumes, so the main-turn and prologue length cues can't drift (a reconciliation test in `ai.test.ts` pins them). `buildPacingDirective(instruction)` emits a `## Pacing & Agency` rule ("advance one beat, stop at the next decision point, never play the character forward; brief downtime montage allowed") for the player-driven instructions, returning empty (dropped) for the ENGINE-COMMITTED turns in `PACING_EXEMPT_INSTRUCTIONS` — both `"combat"` (a multi-exchange state machine that already hands control back at each decision point) AND `"advancement"` (a single engine-decided climactic beat routed through ENGINE_RESOLUTION). Length is governed by this directive, NOT by lowering `MAX_OUTPUT_TOKENS` (that cap holds the whole JSON). `assemblePrompt` pushes both after the history layer (drop-if-empty).
- `sanity.ts` — Sanity tier classification + AI narration directives (issue #9). Lives in the AI layer because `GameState` and the prompt assembly that consume it do. `classifySanityTier(sanity, maxSanity)` maps to `high` (≥75%) / `medium` (≥40%) / `low` (≥15%) / `critical` (<15%) per the spec; `sanityNarrationDirective(tier)` returns the prompt text that makes the narrator progressively unreliable (subtle wrongness → contradictions + one irrational choice → hallucinations narrated as real + false choices). The game layer (`src/lib/game/sanity.ts`) builds its UI-effect and mechanics concerns on top of these.
- `memory.ts` — Tiered memory manager: immediate (last 3-8 turns full — `IMMEDIATE_WINDOW_MAX` raised 5 → 8 for story-memory consistency), recent (older turns as bullets), session facts. Rule-based summarization (no extra LLM call). **Story-memory consistency changes:** (1) `summarizeTurn`'s narrative bullet cap rose 120 → `SUMMARY_NARRATIVE_CHAR_CAP` (320) — once a turn ages out of the immediate window the bullet is all that remains, and 120 chars dropped nearly the whole beat. (2) **Provenance + salience-weighted fact eviction:** `SessionFact.source?: "engine" | "ai"` (`FactSource` in `types.ts`; absent → `engine`, the protected default) — `extractSessionFacts` tags its AI-derived facts `ai`, `addSessionFact` defaults engine-recorded facts to `engine`. The session-fact cap (and `trimMemoryForBudget`) now evict the lowest-`factSalience` fact (`ENGINE_SOURCE_BONUS` lifts every engine fact above every AI fact; then `FACT_TYPE_IMPORTANCE` quest>npc>event>item) via `lowestSalienceFactIndex` (ties break oldest-first → FIFO for a uniform set, preserving prior behaviour) instead of blind `shift()`, so ground-truth facts outlive AI-asserted ones. (3) **Running-summary regression guard:** `isSummaryRegression(prior, next)` (`SUMMARY_REGRESSION_FLOOR` 300 chars, `SUMMARY_COLLAPSE_CEILING` 120 chars) — `addTurn` rejects a new synopsis that near-empties a substantial prior one (a "fail-plausible" recursive-summarization wipe), keeping the prior. The ceiling is an ABSOLUTE char floor on `next`, NOT a fraction of `prior` — a fraction would livelock (a long prior would perpetually reject every legitimately terse rewrite, freezing the synopsis), whereas any real labelled summary clears 120 chars and only a genuine blanking falls below it. No fuzzy GameState string-matching — that's the `## Ground Truth` anchor's job. Plus the **durable rolling `runningSummary`** ("story so far"): the narrator maintains it inside its normal per-turn JSON (amortized — still no extra call), `addTurn` adopts it when present and non-collapsing (keeping the prior synopsis when a turn omits/blanks it), `formatMemoryForPrompt` pins it at the TOP of the history block under `## Story So Far` (fed back so the model recursively prunes/extends it), and `trimMemoryForBudget` never evicts it — so lasting context survives the whole chronicle while `recentSummaries` still FIFO out. `capRunningSummary`/`RUNNING_SUMMARY_CHAR_CAP` bound its growth (via the shared `capWithEllipsis` string helper, also used by the prologue recap); `estimateMemoryTokens` counts it (the shared `CHARS_PER_TOKEN` heuristic, exported here and reused by `prompts.ts`). Prior art: AI Dungeon auto Story Summary, NovelAI pinned Memory, MemGPT recursive summary. `buildTurnRecord(...)` accepts optional `retrievedChunkIds` (issue #63) so each turn records which `source_chunks` informed the narrator — determinism/debuggability for the RAG path.
- `scene-art.ts` — AI scene art (issue #20), **prompt + trigger helpers only** (generation moved to `image-providers.ts`): `SCENE_ART_STYLE` (one fixed Victorian-occult style prelude for visual coherence), `buildSceneArtPrompt` (style + capped moment description), `shouldGenerateSceneArt` (journal event types: advancement/death/combat/major-event), `sceneArtKey(sessionId, turn)` (the cache identity — a moment is never generated twice), `SceneArtContext`. No longer OpenAI-coupled: the old `generateSceneArt`/`sceneArtSupported`/`SCENE_ART_MODEL` are gone — scene art is now produced by an independently-configured image provider.
- `image-providers.ts` — **Image model selection (decoupled scene-art provider).** A provider/model choice for illustrations that is INDEPENDENT of the text narrator: the player may run text on Ollama and images on OpenAI (or a local Ollama image model, or local Stable Diffusion, or nothing). `ImageProviderId` = `openai` | `ollama` | `local-sd`; `ImageProviderConfig {providerId, apiKey, baseUrl?, model}` is persisted SEPARATELY (`IMAGE_PROVIDER_CONFIG_KEY`, never synced — BYOK boundary, like the text config). Two transports: **`openai-images`** (`openai`/`ollama`) POSTs the OpenAI-compatible `/images/generations` (Ollama added an experimental image-gen endpoint of this shape in Jan 2026 — z-image / flux2-klein) and reads `b64_json` (→ `data:` URL) or a hosted `url`; **`sd-txt2img`** (`local-sd`) POSTs Automatic1111/Forge `/sdapi/v1/txt2img` and wraps the base64 `images[0]`. **Ollama _Cloud_ is deliberately NOT an image backend** — ollama.com hosts only chat/vision models, and Ollama's image-generation models (z-image / flux2-klein) are local-on-device only per its docs, so a cloud key has no image endpoint to call (image gen runs via local Ollama / OpenAI / local SD). Pure helpers `imageProviderRequiresKey`/`imageProviderNeedsBaseUrl`/`imageProviderNeedsModel`/`defaultImageBaseUrl` drive the Settings UI; `IMAGE_PROVIDER_MODELS` is the curated per-provider catalog (the UI also allows a free-form model id for the diffusion backends). `imageArtSupported(config)` is the new gate (known provider + a key when the provider needs one + a model when the transport needs one — `local-sd` runs against its loaded checkpoint with no model id). `generateImage(config, prompt, fetchFn?)` is browser-direct with shared-AIError mapping (`classifyHttpError`/`createNetworkError`). The OpenAI catalog is the **gpt-image** family (`gpt-image-2` default / `gpt-image-1.5` / `gpt-image-1` / `gpt-image-1-mini`) — dall-e-2/3 were removed from the API on 2026-05-12; we send `size` but **never** `response_format` (gpt-image rejects it with HTTP 400 and returns base64 by default, which the reader wraps as a `data:` URL — required since the CSP `img-src` is `'self' data: blob:`, so a hosted image URL would not render).
- `usage.ts` — Session token-usage estimates (issue #15): `TurnUsage`/`SessionUsage`, `addUsage`, `estimateSessionCost` (generic `DEFAULT_TOKEN_RATES` — a rough order-of-magnitude figure, the player's BYOK key pays the real bill), `formatUsage`/`formatTokenCount`, strict `serializeUsage`/`deserializeUsage`. `generate()` returns `usage` (prompt side from the assembly estimate, output side from the parsed JSON) on every `ValidatedAIResponse`; the game loop tallies and displays it. Pure functions — localStorage stays in the React layer.
- `validation.ts` — Parse and validate AI output: JSON parsing with code-block extraction (plus a forgiving first-`{`…last-`}` fallback when prose surrounds the JSON), structural validation (sanity bounds — now **±5**, issue #95; choice limits, alignment range, non-empty choice text), `sanitizeAIResponse` for auto-correction (clamps ranges, drops blank-text choices). `parseAIResponse` also normalizes `sanityEventTags` to the known set (drop-not-throw, empty array dropped) and carries `actingMethodTaught` only when literally `true` (issue #95). Item categories are validated against `VALID_ITEM_CATEGORIES` (the five `Item["category"]` kinds incl. `mundane` and `uniqueness`); an unknown/missing category now defaults to **`mundane`** (issue #90) — never a reagent — so a garbled category becomes harmless loot rather than something the engine-only block must strip. `parseAIResponse` extracts the optional `runningSummary` (whitelisted, like every field), stored **trimmed** and only when non-blank; length-capping is owned solely by the persistence boundary (`addTurn`), the one path into durable memory — no redundant capping in parse/sanitize. `parseAIResponse` also carries the optional `codexUpdates` (history-context Codex) through loosely — keep only well-formed items (string kind + non-empty name), coerce optional fields, drop the rest, and cap at `MAX_CODEX_UPDATES` (6); the game-layer `applyCodexUpdates` is the single real validation point. Exports `ensureUniqueChoiceIds(choices)` — back-fills empty/duplicate choice ids with a stable `choice-${i}` so ids stay unique (they double as React keys and the selection handle); shared with `prologue-client.ts` so both AI paths get the same guarantee.
- `client.ts` — Main orchestrator. `generateCodexRebuild(config, input)` (history-context Codex) is the rebuild shell: builds the archivist prompt (`codex-rebuild.ts`), calls the **premium** model through the shared `executeWithRetry` seam with `json_object`, and returns `parseCodexRebuild`'d `CodexUpdateInput[]`. It runs a **corrective parse-retry loop** (`MAX_PARSE_ATTEMPTS`, mirroring `requestAndParse`) — the first attempt is deterministic (temperature 0) for a stable/repeatable result, and a failed parse re-prompts with the corrective instruction at a small temperature (0.4) so a degenerate temp-0 reply isn't reproduced verbatim; each unparseable attempt is `console.error`'d (`logUnparseableOutput`, key never logged) since browser-direct BYOK has no server log. Returns `[]` only after exhausting the retries. `generate()` assembles prompt (threading `retrievedChunks` for the RAG path, issue #64), selects model by tier, calls provider with retry, parses/validates response, and attaches the rough `usage` estimate (issue #15). JSON parsing retries up to `MAX_PARSE_ATTEMPTS` with a corrective prompt (truncation-aware) and lowered temperature. **Observability:** `executeWithRetry` is the single seam both the main loop and the prologue funnel through, so it `console.error`s every failed provider attempt (`provider`/`model`/`attempt`/`code`/`status`/`reason`/body snippet) for **any** provider — the intentional `probeModelAccess`/`validateKey` paths call adapters directly and bypass it, so expected probe errors don't add noise; the API key is never logged. `requestAndParse` likewise logs unparseable 2xx output (`truncated` flag + content head/tail) — the client-side complement to the ollama-cloud proxy's server-side diagnostic, covering every provider for the "Failed to parse structured JSON" (gpt-oss/Harmony) failure mode. `classifyCall()` determines routine vs premium. `validateProviderConfig()` checks API key validity. `listProviderModels()` fetches the provider's live model catalog. `findUnservedModels(config, catalog)` is a pure check that returns any configured `routineModel`/`premiumModel` absent from a live catalog — the Settings UI warns on it to pre-empt the mid-game 403 an unserved model (e.g. an Ollama-Cloud account that doesn't serve the premium model) would otherwise cause. `probeModelAccess(config, modelIds)` goes a step further for the case `findUnservedModels` can't see: ollama.com's `/v1/models` lists its **whole** cloud catalog regardless of plan, so a listed (selectable) model can still 403 ("requires a subscription") only at chat time. It test-calls each model with a minimal 1-token chat request and returns `ModelAccessResult[]` (`{model, accessible, reason?}`), flagging a model inaccessible **only** on a definitive per-model refusal (403 gated / 404 missing) — a 401 (bad key), 429, 5xx, or network error is inconclusive and left accessible. Never throws; blank ids drop and duplicates collapse.
- `codex-rebuild.ts` — **Codex rebuild from history (history-context Codex), pure prompt + parse.** A pre-feature long save's Codex can only be heuristically backfilled from its structured sub-states; the heavy entity record lives in PROSE (running summary + journal + facts). `buildCodexRebuildPrompt(CodexRebuildInput)` builds a one-shot archivist extraction prompt (system + a digest user message: character, synopsis, journal beats with `[present: …]`, recorded facts); `parseCodexRebuild(raw)` is the forgiving parser (fence/embedded-JSON extraction, accepts a bare array or `{entities:[…]}`, drops malformed items, caps at `MAX_REBUILD_ENTITIES` = 80, never throws → `[]`) returning `CodexUpdateInput[]` — `applyCodexUpdates` stays the single real validation point. `CodexRebuildJournalEntry`/`CodexRebuildInput` are the shapes; the digest is assembled by `codexRebuildDigest` (`@/lib/game/codex`). The network shell is `generateCodexRebuild(config, input)` in `client.ts` (reuses the `executeWithRetry`+`selectModel` seam, premium model, low temperature, json_object) — the React Codex tab folds the result into a fresh `codexState`.
- `prologue-client.ts` — AI-driven prologue generation. **Neighborhood affinities (issue #119):** the prologue's affinities are the canonical pathway **regions** (Sefirah groups) over **all twenty-two playable pathways** (every id now carries full becoming-scene + potion/sequence content, issue #120), defined in the self-contained `PROLOGUE_AFFINITY_REGIONS` catalog spanning all nine canon groups (`mysteries` {1,7,8}, `god-almighty` {2,3,6,9,10}, `eternal-darkness` {4,5,11}, `order` {12,13}, `combat` {14,15}, `life` {16,17}, `knowledge` {18,19}, `wheel-of-fortune` {20}, `abyss` {21,22}) — kept free of a rules-engine import, with a reconciliation test (`ai.test.ts`) holding the ids/groups against `@/lib/rules` `PATHWAY_GROUPS` so they can't drift. `generatePrologueScene()` runs an interactive multi-turn prologue where each scene presents `PROLOGUE_MIN_CHOICES`..`PROLOGUE_MAX_CHOICES` (3–5) choices, each LEANING toward 1–2 NEIGHBORING pathways **within one region** (a blended `affinities` weight map, never a clean 1:1 "the X option"), and the scene must span at least `PROLOGUE_MIN_REGIONS_PER_SCENE` (2) regions so the player chooses between temperaments. The AI only narrates and tags choices — it never judges or scores the pathway; `readyToConclude` is advisory and the engine enforces MIN/MAX (now `MIN_PROLOGUE_SCENES` = 6, typical 6–8). `parseChoice` is **forgiving**: it drops weights that aren't positive numbers on a playable id, then snaps the choice to the dominant id's region and caps it at `PROLOGUE_MAX_AFFINITIES_PER_CHOICE` (2) neighbors — so a stray cross-region weight is normalized away rather than thrown, while a choice with no usable signal or a scene that fails the count/region-span contract still throws `AIError("MALFORMED_OUTPUT")`. The specific pathway EMERGES from the accumulated pattern (the engine's generic `tallyAffinities`/`selectTopCandidates` in `@/lib/game`), so the guided prologue can now lead to any of the twenty-two playable pathways (issue #120). Choice ids do not constrain affinities, so both generators run the parsed choices through `ensureUniqueChoiceIds()` (from `validation.ts`) to keep React keys unique even if the model repeats/omits an id. `generatePrologueFinale()` is a separate, player-decided step: the engine computes the candidate pathway ids (the only place a pathway id enters a prompt) and the AI renders exactly those as evocatively-described potion choices; the player's pick IS the final pathway. **Epoch-aware setting (character epoch isolation):** the prologue is the ONE narration surface that runs on its OWN system prompt (not `assemblePrompt`), so the epoch tone the main turn/combat get from `epochNarrationDirective` never reached it — the setting was hardcoded to Fifth-Epoch Tingen, narrating a pre-Iron-Age "becoming" with gaslamps/churches that don't exist in that era. `PROLOGUE_EPOCH_SETTINGS` (a self-contained per-epoch table keyed to the five canon epochs, like `PROLOGUE_AFFINITY_REGIONS` — no lore/rules import; data-integrity tested) supplies the era's `setting`/`concealment` (the Fifth's "the supernatural is hidden" frame is wrong for e.g. the Dark Epoch, where the gods rule openly)/`dailyLife`/`atmosphere`/`defaultBackground`; `resolvePrologueSetting(epoch?)` resolves it (absent/unknown → Fifth, `PROLOGUE_DEFAULT_EPOCH`); `buildPrologueSystemPrompt(epoch?)`/`buildPrologueFinaleSystemPrompt(epoch?)` weave it into the prompts while the mechanical contract (affinity regions, choice rules, response format) stays identical for every era. Both generators take a trailing optional `epoch` (threaded from `character-creation.tsx`'s epoch selector); the exported `PROLOGUE_SYSTEM_PROMPT`/`PROLOGUE_FINALE_SYSTEM_PROMPT` consts are the Fifth-Epoch baselines. **Verbosity preset:** since the prologue runs on its own system prompt (not `assemblePrompt`), the main `## Narration Length` directive never reaches it — instead a trailing optional `verbosity` arg (`buildPrologueSystemPrompt(epoch?, verbosity?)`/`buildPrologueFinaleSystemPrompt(epoch?, verbosity?)`, and `CanonPrologueContext.verbosity`) weaves a per-scene length line (`prologueVerbosityLine`) into the WRITING GUIDELINES for `"concise"`/`"rich"` (empty for `"standard"`/absent — the existing "2–4 paragraphs" baseline holds). The PACING rule is deliberately NOT applied to the prologue (it is its own structured multi-scene flow, like combat). `character-creation.tsx` reads the saved `narrativeVerbosity` and threads it in. **Canon-character takeover prologue (issue #92):** when the player takes over a known novel figure the pathway is FIXED (no affinity discovery), so a separate canon-faithful path runs — `generateCanonPrologueScene(ctx, config, history)` narrates the figure's established story toward their introduction with PLAIN (affinity-free) choices that LEAN toward the figure's `personality`, and `generateCanonPrologueFinale(ctx, config, history)` writes the closing beat: a becoming-potion threshold when `ctx.becomesOnScreen`, else an arrive-at-first-appearance hand-off with NO potion. The fixed canon facts (`CanonPrologueContext`: characterName/background/pathwayName/personality/becomesOnScreen/epoch) enter as PLAIN STRINGS, so the AI layer stays lore/rules-free (mirrors the rest of the file); both reuse `executePrologueRequest`/`buildHistoryMessages`/`resolvePrologueSetting` and `ensureUniqueChoiceIds` (via `parseCanonChoices`). `CanonPrologueScene`/`CanonPrologueFinale`/`CanonPrologueChoice` are the return shapes. **JSON resilience (prologue-json-error):** `executePrologueRequest` now mirrors the main loop's `requestAndParse` — it caps output at `PROLOGUE_MAX_TOKENS` (**reuses** the exported `MAX_OUTPUT_TOKENS` = 3072 so the two caps can't drift; the old **800** cap truncated the heavier finale mid-JSON, surfacing as "Prologue AI returned invalid JSON" and blocking the end of character creation) and wraps the provider call + `parseObject`/`requireNarrative` in a corrective parse-retry loop (`MAX_PROLOGUE_PARSE_ATTEMPTS` = 3; first attempt at `PROLOGUE_TEMPERATURE` 0.85, retries halved). It reuses `client.ts`'s exported `correctiveMessage` (truncation-aware) + `logUnparseableOutput`. The retry is scoped to the PROVIDER-OUTPUT parse (bad JSON / missing narrative — the truncation failure mode); the per-generator structural contract (choice count, region span, candidate coverage) is still validated by the callers AFTER `executePrologueRequest` returns and fails fast (that JSON is well-formed, so a generic "respond with valid JSON" retry would not address it).

## Key Design Decisions

- **Stateless single-turn calls.** Every AI call is self-contained — no multi-turn conversation threads.
- **BYOK (Bring Your Own Key).** API keys stored client-side only (localStorage). Never sent to our backend.
- **Direct browser-to-provider calls.** No proxy server needed for supported providers.
- **Model tiering.** Calls classified as `routine` or `premium` based on instruction type. Advancement and combat are premium; everything else is routine.
- **Token budget enforcement.** History is trimmed to fit `TOKEN_BUDGET` (11,500 total; history 4,500). System prompt and game state are never trimmed.
- **Robust JSON output.** Output cap is `MAX_OUTPUT_TOKENS` (3072) to avoid mid-JSON truncation; Anthropic forces JSON with a system-prompt directive (assistant prefill 400s on current Claude models), OpenAI-style providers use `response_format: json_object`. Malformed/truncated output triggers a corrective retry loop (`MAX_PARSE_ATTEMPTS`).
- **Live model lists.** `listModels()` reuses each provider's catalog endpoint (`/models`, `/api/tags`); the config UI caches results in localStorage and falls back to the static `PROVIDER_MODELS` map.

## Conventions

- All provider adapters implement `LLMProviderAdapter` interface — swap providers with config change only.
- Validation functions return `ValidationResult` (same pattern as rules engine) — never throw.
- `sanitizeAIResponse` clamps out-of-range values rather than rejecting — player always gets a response.
- Error handling uses exponential backoff retry (2s, 4s, 8s) for retryable errors.
