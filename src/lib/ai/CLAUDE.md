# AI Integration Layer

Provider-agnostic LLM interface with tiered prompt architecture. The AI generates narrative and soft lore (Tier 2/3) while the rules engine handles hard mechanics (Tier 1).

## Structure

- `types.ts` ‚Äî All AI-related type definitions: providers, responses, prompts, memory, game state. `AIResponse.journalEntry` (issue #11) is the AI's optional journal-worthy flag ‚Äî loosely typed at the boundary; `validateJournalFlag` in `@/lib/game/journal` is the single validation point.
- `errors.ts` ‚Äî `AIError` class with unified error codes: AUTH_ERROR, RATE_LIMITED, QUOTA_EXCEEDED, PROVIDER_ERROR, MALFORMED_OUTPUT, VALIDATION_FAILED, NETWORK_ERROR, TOKEN_LIMIT_EXCEEDED.
- `providers.ts` ‚Äî Provider adapters: Anthropic, OpenAI, OpenRouter, Ollama, Ollama Cloud, Custom. Each implements `LLMProviderAdapter` interface (incl. `listModels(apiKey)` for the live model catalog). Factory: `createAdapter(providerId, baseUrl?)`. The Anthropic adapter prefills the assistant turn with `{` to force JSON (no `response_format` support). `parseResponse` sets `truncated` from the provider's stop/finish reason. `inferModelTier(id)` guesses a tier for dynamically-listed models. `OllamaAdapter` also exposes `embed(model, texts, apiKey)` over `/api/embed` (issue #60) ‚Äî the one HTTP embed call both embedding transports use.
- `embeddings.ts` ‚Äî Embedding seam (issue #60), **local-only** (no vendor, no operator key ‚Äî resolved in #57's thread). The `EmbeddingProvider` interface (`id`, `dims`, `model_id`, `embed(texts)`) is a sibling to `LLMProviderAdapter`: the embedding model is a fixed, approved choice locked per save (a query vector only compares to corpus vectors from the same model's "map"), not a free-form pick. `APPROVED_EMBEDDING_MODELS` = two 1024-dim CPU models, `qwen3-embedding-0.6b` (default) + `bge-m3`, each pinned by an explicit Ollama tag. `createEmbeddingProvider({id, modelId?, baseUrl?, apiKey?})` builds a provider for one transport: `ollama` = the player's own Ollama (BYO, browser-direct, mirrors BYOK); `operator` = the always-on CPU box at `NEXT_PUBLIC_OPERATOR_EMBEDDING_URL` (the zero-setup default/fallback ‚Äî **not** a Vercel serverless function; no resident model). Both speak Ollama's `/api/embed`; `embed()` validates the batch count and 1024-dim shape.
- `prompts.ts` ‚Äî Layered prompt assembly: system prompt, sanity-narration directive, lore context, game state, history, instruction. Token budget enforcement (8,800 total; lore 4,000 ‚Äî raised for retrieval, issue #64). `buildLoreContext(loreContext, retrievedChunks?)` injects curated guardrail entries FIRST and in full (`narratorOnly`-aware), then packs gated retrieved chunks (`RetrievedLoreChunk`, threaded via `PromptInput.retrievedChunks`) into whatever lore budget remains via `selectRetrievedForBudget` (first-fit in retrieval-rank order) ‚Äî authored lore is never crowded out. Retrieved text is framed `[NARRATOR ONLY]` with a no-verbatim-quoting directive. Per-turn cost quantified in `docs/rag-per-turn-budget.md`; note retrieved chunks DO reach the browser (BYOK browser-direct) ‚Äî risk-accepted, never claim otherwise. `buildSanityDirective(gameState)` appends an unreliable-narration directive (from `sanity.ts`) when sanity is below the high tier; at high sanity it returns empty content and the layer is dropped.
- `sanity.ts` ‚Äî Sanity tier classification + AI narration directives (issue #9). Lives in the AI layer because `GameState` and the prompt assembly that consume it do. `classifySanityTier(sanity, maxSanity)` maps to `high` (‚â•75%) / `medium` (‚â•40%) / `low` (‚â•15%) / `critical` (<15%) per the spec; `sanityNarrationDirective(tier)` returns the prompt text that makes the narrator progressively unreliable (subtle wrongness ‚Üí contradictions + one irrational choice ‚Üí hallucinations narrated as real + false choices). The game layer (`src/lib/game/sanity.ts`) builds its UI-effect and mechanics concerns on top of these.
- `memory.ts` ‚Äî Tiered memory manager: immediate (last 3-5 turns full), recent (turns 6-20 bullets), session facts. Rule-based summarization (no extra LLM call). `buildTurnRecord(...)` accepts optional `retrievedChunkIds` (issue #63) so each turn records which `source_chunks` informed the narrator ‚Äî determinism/debuggability for the RAG path.
- `usage.ts` ‚Äî Session token-usage estimates (issue #15): `TurnUsage`/`SessionUsage`, `addUsage`, `estimateSessionCost` (generic `DEFAULT_TOKEN_RATES` ‚Äî a rough order-of-magnitude figure, the player's BYOK key pays the real bill), `formatUsage`/`formatTokenCount`, strict `serializeUsage`/`deserializeUsage`. `generate()` returns `usage` (prompt side from the assembly estimate, output side from the parsed JSON) on every `ValidatedAIResponse`; the game loop tallies and displays it. Pure functions ‚Äî localStorage stays in the React layer.
- `validation.ts` ‚Äî Parse and validate AI output: JSON parsing with code-block extraction (plus a forgiving first-`{`‚Ä¶last-`}` fallback when prose surrounds the JSON), structural validation (sanity bounds, choice limits, alignment range, non-empty choice text), `sanitizeAIResponse` for auto-correction (clamps ranges, drops blank-text choices). Exports `ensureUniqueChoiceIds(choices)` ‚Äî back-fills empty/duplicate choice ids with a stable `choice-${i}` so ids stay unique (they double as React keys and the selection handle); shared with `prologue-client.ts` so both AI paths get the same guarantee.
- `client.ts` ‚Äî Main orchestrator: `generate()` assembles prompt (threading `retrievedChunks` for the RAG path, issue #64), selects model by tier, calls provider with retry, parses/validates response, and attaches the rough `usage` estimate (issue #15). JSON parsing retries up to `MAX_PARSE_ATTEMPTS` with a corrective prompt (truncation-aware) and lowered temperature. `classifyCall()` determines routine vs premium. `validateProviderConfig()` checks API key validity. `listProviderModels()` fetches the provider's live model catalog.
- `prologue-client.ts` ‚Äî AI-driven prologue generation. `generatePrologueScene()` runs an interactive multi-turn prologue where each scene presents EXACTLY `PROLOGUE_AFFINITY_COUNT` (4) choices, one per Beyonder affinity, each tagged with an `affinities` weight map (`{pathwayId: weight}`). The AI only narrates and tags choices ‚Äî it never judges or scores the pathway (no `inferredPathwayId`); `readyToConclude` is advisory and the engine enforces MIN/MAX. Validation is strict (exactly 4 choices, each with a positive-int/positive-weight affinity map, the four dominant affinities all present) and throws `AIError("MALFORMED_OUTPUT")` rather than silently defaulting to a pathway. Choice ids do not constrain affinities, so both generators run the parsed choices through `ensureUniqueChoiceIds()` (from `validation.ts`) to keep React keys unique even if the model repeats/omits an id. `generatePrologueFinale()` is a separate, player-decided step: the engine computes the candidate pathway ids (the only place a pathway id enters a prompt) and the AI renders exactly those as evocatively-described potion choices; the player's pick IS the final pathway.
- `index.ts` ‚Äî Public exports.
- `ai.test.ts` ‚Äî Comprehensive test suite.

## Key Design Decisions

- **Stateless single-turn calls.** Every AI call is self-contained ‚Äî no multi-turn conversation threads.
- **BYOK (Bring Your Own Key).** API keys stored client-side only (localStorage). Never sent to our backend.
- **Direct browser-to-provider calls.** No proxy server needed for supported providers.
- **Model tiering.** Calls classified as `routine` or `premium` based on instruction type. Advancement and combat are premium; everything else is routine.
- **Token budget enforcement.** History is trimmed if prompt exceeds 8,800 tokens. System prompt and game state are never trimmed.
- **Robust JSON output.** Output cap is `MAX_OUTPUT_TOKENS` (3072) to avoid mid-JSON truncation; Anthropic uses assistant prefill, OpenAI-style providers use `response_format: json_object`. Malformed/truncated output triggers a corrective retry loop (`MAX_PARSE_ATTEMPTS`).
- **Live model lists.** `listModels()` reuses each provider's catalog endpoint (`/models`, `/api/tags`); the config UI caches results in localStorage and falls back to the static `PROVIDER_MODELS` map.

## Conventions

- All provider adapters implement `LLMProviderAdapter` interface ‚Äî swap providers with config change only.
- Validation functions return `ValidationResult` (same pattern as rules engine) ‚Äî never throw.
- `sanitizeAIResponse` clamps out-of-range values rather than rejecting ‚Äî player always gets a response.
- Error handling uses exponential backoff retry (2s, 4s, 8s) for retryable errors.
