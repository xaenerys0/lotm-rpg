# AI Integration Layer

Provider-agnostic LLM interface with tiered prompt architecture. The AI generates narrative and soft lore (Tier 2/3) while the rules engine handles hard mechanics (Tier 1).

## Structure

- `types.ts` — All AI-related type definitions: providers, responses, prompts, memory, game state. `AIResponse.journalEntry` (issue #11) is the AI's optional journal-worthy flag — loosely typed at the boundary; `validateJournalFlag` in `@/lib/game/journal` is the single validation point. `AIResponse.proposedSelfChange` (character-info storage) is the same shape of loose flag for a true-self change the narrator detected the player declaring — whitelisted through `parseAIResponse`, validated by `validateSelfChangeProposal` (`@/lib/game/profile`), and NEVER applied without an explicit player confirm (it is deliberately separate from the AI-mutable `worldStateChanges` allowlist). `PromptInput.profileContext`/`recognitionContext` are the True-Self ground-truth and recognition-gap narrator layers. `AIResponse.fundsDiscovered` (issue #16 follow-up) is the optional pence-found-this-turn channel — whitelisted in `parseAIResponse` as a finite number, clamped + applied in `world-state.ts` (NOT the AI-mutable allowlist). **Hybrid sanity + acting-method discovery (issue #95):** `AIResponse.sanityImpact` is now only a small residual nuance tightened to **±5**; the bulk of per-turn sanity comes from `AIResponse.sanityEventTags?: string[]` (the engine computes magnitudes via `sanityDeltaForTags`), normalized in `parseAIResponse` to the known set (unknown tags dropped). `AIResponse.actingMethodTaught?: boolean` is the drop-not-throw flag (carried only when literally `true`) for the AI signalling the acting method was explicitly taught/revealed this turn — the only AI-driven discovery trigger, never a sanity scare.
- `errors.ts` — `AIError` class with unified error codes: AUTH_ERROR, RATE_LIMITED, QUOTA_EXCEEDED, PROVIDER_ERROR, MALFORMED_OUTPUT, VALIDATION_FAILED, NETWORK_ERROR, TOKEN_LIMIT_EXCEEDED. `AIError` also carries the originating HTTP `status` (when from `classifyHttpError`) so callers can distinguish 401 (bad key) from 403 (key fine, model/request refused) without string-matching, plus the already-extracted `reason` (the provider's bare message) so consumers read `err.reason` instead of re-parsing the body. `classifyHttpError` extracts the reason once and threads `status`/`reason` onto every error it builds. `classifyHttpError` maps **401** to the "invalid/expired key" message but **403** to a distinct model-unavailable/quota message (both `AUTH_ERROR`, so the UI keeps the Settings CTA) — a 403 means the key worked and the request was refused, so it must not blame the key. **400/404** map to a non-retryable `AUTH_ERROR` (model id invalid/unavailable, or wrong base URL) — deterministic config faults that must not burn the retry backoff, while keeping the Settings CTA without blaming the key. **Every** branch surfaces the provider's own wording via a `withReason` helper (so the message reads e.g. `Unexpected HTTP 400. Provider said: model: … not found`) — this is the only way to see the cause for browser-direct providers, which have no server-side log. The wording comes from `extractProviderMessage(body)` (exported) — e.g. ollama.com's "this model requires a subscription, upgrade for access: …"; `extractProviderMessage` reads `{error}` / `{error:{message}}` / `{message}` JSON shapes and falls back to capped raw text for non-JSON bodies.
- `providers.ts` — Provider adapters: Anthropic, OpenAI, OpenRouter, Ollama, Ollama Cloud, Custom. Each implements `LLMProviderAdapter` interface (incl. `listModels(apiKey)` for the live model catalog). Factory: `createAdapter(providerId, baseUrl?)`. The Anthropic adapter forces JSON via a system-prompt directive (no `response_format` support); it must NOT prefill the assistant turn — newer Claude models (Opus 4.6+, Sonnet 4.6, Fable 5) reject a trailing assistant turn with HTTP 400 ("does not support assistant message prefill"), and the forgiving parser + corrective retry loop cover the looser guarantee. `parseResponse` sets `truncated` from the provider's stop/finish reason. `inferModelTier(id)` guesses a tier for dynamically-listed models. `OllamaAdapter` also exposes `embed(model, texts, apiKey)` over `/api/embed` (issue #60) — the one HTTP embed call both embedding transports use. `OllamaAdapter.makeRequest` sets `options.num_ctx` via the pure `ollamaNumCtx(messages, maxTokens)` helper (prompt-token estimate + output cap + headroom, rounded up to a 1024 step, floored at `OLLAMA_MIN_NUM_CTX` 4096) — Ollama otherwise runs the model at its small default context, so a large RAG-retrieval prompt would fill the window and leave no room to generate (the model emits 2-3 tokens and stops with `done_reason "length"`, surfacing as "unparseable provider output"). Only Ollama needs this; the hosted providers manage context server-side.
- `embeddings.ts` — Embedding seam (issue #60), **local-only** (no vendor, no operator key — resolved in #57's thread). The `EmbeddingProvider` interface (`id`, `dims`, `model_id`, `embed(texts)`) is a sibling to `LLMProviderAdapter`: the embedding model is a fixed, approved choice locked per save (a query vector only compares to corpus vectors from the same model's "map"), not a free-form pick. `APPROVED_EMBEDDING_MODELS` = two 1024-dim CPU models, `qwen3-embedding-0.6b` (default) + `bge-m3`, each pinned by an explicit Ollama tag. `createEmbeddingProvider({id, modelId?, baseUrl?, apiKey?})` builds a provider for one transport: `ollama` = the player's own Ollama (BYO, browser-direct, mirrors BYOK); `operator` = the always-on CPU box at `NEXT_PUBLIC_OPERATOR_EMBEDDING_URL` (the zero-setup default/fallback — **not** a Vercel serverless function; no resident model). Both speak Ollama's `/api/embed`; `embed()` validates the batch count and 1024-dim shape.
- `prompts.ts` — Layered prompt assembly: system prompt, sanity-narration directive, lore context, game state, history, instruction. **Issue #95:** the schema/rules now advertise `sanityEventTags` and `actingMethodTaught`, cap `sanityImpact` at ±5 (the stale -16..-20 guidance is gone), and **soften the acting rule** — the AI scores acting honestly but is told NOT to name/explain the "digestion"/"acting method" mechanic (it is secret knowledge) and to set `actingMethodTaught` only for an explicit in-fiction teaching, never a sanity scare. The game state still feeds `potionDigestion: X%` to the AI. The system prompt carries a **Canon Fidelity** section (follow provided lore/retrieved material exactly; when uncovered, invent only mundane, world-consistent texture — never fabricate Beyonder-tier canon — and prefer ambiguity to a confident wrong answer) and a hardened **Running Summary** rule (record only what play/lore established — no new canonical names/orgs/deities/history harden into the durable synopsis). Token budget enforcement (8,800 total; lore 4,000 — raised for retrieval, issue #64). `buildLoreContext(loreContext, retrievedChunks?)` injects curated guardrail entries FIRST and in full (`narratorOnly`-aware), then packs gated retrieved chunks (`RetrievedLoreChunk`, threaded via `PromptInput.retrievedChunks`) into whatever lore budget remains via `selectRetrievedForBudget` (first-fit in retrieval-rank order) — authored lore is never crowded out. Retrieved text is framed `[NARRATOR ONLY]` with a no-verbatim-quoting directive. Optional system layers: `## Epoch` (issues #26/#29, `PromptInput.epochContext`), `## Active Identity` (issue #22, `identityContext`), `## True Self` (character-info storage, `profileContext` — ground-truth self facts), and `## Recognition` (character-info storage, `recognitionContext` — who knew the character before a drastic change and doesn't recognise them now). `buildGameStatePrompt` also appends a durable `## Character & Origin` section (the character's `characterBackground` + `prologueRecap`) inside the never-trimmed game-state layer — the prologue → story bridge, so the backstory and the prologue the character just lived through keep shaping narration for the whole chronicle instead of aging out of the history window. Per-turn cost quantified in `docs/rag-per-turn-budget.md`; note retrieved chunks DO reach the browser (BYOK browser-direct) — risk-accepted, never claim otherwise. `buildSanityDirective(gameState)` appends an unreliable-narration directive (from `sanity.ts`) when sanity is below the high tier; at high sanity it returns empty content and the layer is dropped. `buildDemigodDirective(gameState)` (issues #25, #35) mirrors that pattern: at Sequence ≤ `DEMIGOD_SEQUENCE_THRESHOLD` (4 — Saint and above) it adds a `## Demigod Stakes` system layer scaling narration to cosmic consequences, divine politics, and churches/organizations reacting; below the tier it returns empty content and the layer is dropped. At **Sequence 0** (issue #30) the same function returns a `## Godhood` layer instead — the narration shifts entirely from human-scale to a True God's cosmic existence (prayers as voices, action at continental range, divine politics, a human heart inside a god). Optional `## Setting Tone` system layer (issue #23, `PromptInput.cityNarration` from `cityNarrationDirective`) gives each city its own one-sentence narration tone; dropped (`null`/absent) for cities with no specific tone.
- `sanity.ts` — Sanity tier classification + AI narration directives (issue #9). Lives in the AI layer because `GameState` and the prompt assembly that consume it do. `classifySanityTier(sanity, maxSanity)` maps to `high` (≥75%) / `medium` (≥40%) / `low` (≥15%) / `critical` (<15%) per the spec; `sanityNarrationDirective(tier)` returns the prompt text that makes the narrator progressively unreliable (subtle wrongness → contradictions + one irrational choice → hallucinations narrated as real + false choices). The game layer (`src/lib/game/sanity.ts`) builds its UI-effect and mechanics concerns on top of these.
- `memory.ts` — Tiered memory manager: immediate (last 3-5 turns full), recent (turns 6-20 bullets), session facts. Rule-based summarization (no extra LLM call). Plus a **durable rolling `runningSummary`** ("story so far"): the narrator maintains it inside its normal per-turn JSON (amortized — still no extra call), `addTurn` adopts it when present (keeping the prior synopsis when a turn omits/blanks it), `formatMemoryForPrompt` pins it at the TOP of the history block under `## Story So Far` (fed back so the model recursively prunes/extends it), and `trimMemoryForBudget` never evicts it — so lasting context survives the whole chronicle while `recentSummaries` still FIFO out. `capRunningSummary`/`RUNNING_SUMMARY_CHAR_CAP` bound its growth (via the shared `capWithEllipsis` string helper, also used by the prologue recap); `estimateMemoryTokens` counts it (the shared `CHARS_PER_TOKEN` heuristic, exported here and reused by `prompts.ts`). Prior art: AI Dungeon auto Story Summary, NovelAI pinned Memory, MemGPT recursive summary. `buildTurnRecord(...)` accepts optional `retrievedChunkIds` (issue #63) so each turn records which `source_chunks` informed the narrator — determinism/debuggability for the RAG path.
- `scene-art.ts` — AI scene art (issue #20): research summary (OpenAI Images chosen — browser-CORS friendly, rides the player's existing key, ~$0.04/image; hence OPT-IN, default off), `SCENE_ART_STYLE` (one fixed Victorian-occult style prelude for visual coherence), `buildSceneArtPrompt` (style + capped moment description), `shouldGenerateSceneArt` (journal event types: advancement/death/combat/major-event), `sceneArtSupported` (OpenAI-only gate), `generateSceneArt` (browser-direct dall-e-3 → data URL, AIError mapping), `sceneArtKey(sessionId, turn)` (the cache identity — a moment is never paid for twice).
- `usage.ts` — Session token-usage estimates (issue #15): `TurnUsage`/`SessionUsage`, `addUsage`, `estimateSessionCost` (generic `DEFAULT_TOKEN_RATES` — a rough order-of-magnitude figure, the player's BYOK key pays the real bill), `formatUsage`/`formatTokenCount`, strict `serializeUsage`/`deserializeUsage`. `generate()` returns `usage` (prompt side from the assembly estimate, output side from the parsed JSON) on every `ValidatedAIResponse`; the game loop tallies and displays it. Pure functions — localStorage stays in the React layer.
- `validation.ts` — Parse and validate AI output: JSON parsing with code-block extraction (plus a forgiving first-`{`…last-`}` fallback when prose surrounds the JSON), structural validation (sanity bounds — now **±5**, issue #95; choice limits, alignment range, non-empty choice text), `sanitizeAIResponse` for auto-correction (clamps ranges, drops blank-text choices). `parseAIResponse` also normalizes `sanityEventTags` to the known set (drop-not-throw, empty array dropped) and carries `actingMethodTaught` only when literally `true` (issue #95). Item categories are validated against `VALID_ITEM_CATEGORIES` (the five `Item["category"]` kinds incl. `mundane` and `uniqueness`); an unknown/missing category now defaults to **`mundane`** (issue #90) — never a reagent — so a garbled category becomes harmless loot rather than something the engine-only block must strip. `parseAIResponse` extracts the optional `runningSummary` (whitelisted, like every field), stored **trimmed** and only when non-blank; length-capping is owned solely by the persistence boundary (`addTurn`), the one path into durable memory — no redundant capping in parse/sanitize. Exports `ensureUniqueChoiceIds(choices)` — back-fills empty/duplicate choice ids with a stable `choice-${i}` so ids stay unique (they double as React keys and the selection handle); shared with `prologue-client.ts` so both AI paths get the same guarantee.
- `client.ts` — Main orchestrator: `generate()` assembles prompt (threading `retrievedChunks` for the RAG path, issue #64), selects model by tier, calls provider with retry, parses/validates response, and attaches the rough `usage` estimate (issue #15). JSON parsing retries up to `MAX_PARSE_ATTEMPTS` with a corrective prompt (truncation-aware) and lowered temperature. **Observability:** `executeWithRetry` is the single seam both the main loop and the prologue funnel through, so it `console.error`s every failed provider attempt (`provider`/`model`/`attempt`/`code`/`status`/`reason`/body snippet) for **any** provider — the intentional `probeModelAccess`/`validateKey` paths call adapters directly and bypass it, so expected probe errors don't add noise; the API key is never logged. `requestAndParse` likewise logs unparseable 2xx output (`truncated` flag + content head/tail) — the client-side complement to the ollama-cloud proxy's server-side diagnostic, covering every provider for the "Failed to parse structured JSON" (gpt-oss/Harmony) failure mode. `classifyCall()` determines routine vs premium. `validateProviderConfig()` checks API key validity. `listProviderModels()` fetches the provider's live model catalog. `findUnservedModels(config, catalog)` is a pure check that returns any configured `routineModel`/`premiumModel` absent from a live catalog — the Settings UI warns on it to pre-empt the mid-game 403 an unserved model (e.g. an Ollama-Cloud account that doesn't serve the premium model) would otherwise cause. `probeModelAccess(config, modelIds)` goes a step further for the case `findUnservedModels` can't see: ollama.com's `/v1/models` lists its **whole** cloud catalog regardless of plan, so a listed (selectable) model can still 403 ("requires a subscription") only at chat time. It test-calls each model with a minimal 1-token chat request and returns `ModelAccessResult[]` (`{model, accessible, reason?}`), flagging a model inaccessible **only** on a definitive per-model refusal (403 gated / 404 missing) — a 401 (bad key), 429, 5xx, or network error is inconclusive and left accessible. Never throws; blank ids drop and duplicates collapse.
- `prologue-client.ts` — AI-driven prologue generation. `generatePrologueScene()` runs an interactive multi-turn prologue where each scene presents EXACTLY `PROLOGUE_AFFINITY_COUNT` (4) choices, one per Beyonder affinity, each tagged with an `affinities` weight map (`{pathwayId: weight}`). The AI only narrates and tags choices — it never judges or scores the pathway (no `inferredPathwayId`); `readyToConclude` is advisory and the engine enforces MIN/MAX. Validation is strict (exactly 4 choices, each with a positive-int/positive-weight affinity map, the four dominant affinities all present) and throws `AIError("MALFORMED_OUTPUT")` rather than silently defaulting to a pathway. Choice ids do not constrain affinities, so both generators run the parsed choices through `ensureUniqueChoiceIds()` (from `validation.ts`) to keep React keys unique even if the model repeats/omits an id. `generatePrologueFinale()` is a separate, player-decided step: the engine computes the candidate pathway ids (the only place a pathway id enters a prompt) and the AI renders exactly those as evocatively-described potion choices; the player's pick IS the final pathway.
- `index.ts` — Public exports.
- `ai.test.ts` — Comprehensive test suite.

## Key Design Decisions

- **Stateless single-turn calls.** Every AI call is self-contained — no multi-turn conversation threads.
- **BYOK (Bring Your Own Key).** API keys stored client-side only (localStorage). Never sent to our backend.
- **Direct browser-to-provider calls.** No proxy server needed for supported providers.
- **Model tiering.** Calls classified as `routine` or `premium` based on instruction type. Advancement and combat are premium; everything else is routine.
- **Token budget enforcement.** History is trimmed if prompt exceeds 8,800 tokens. System prompt and game state are never trimmed.
- **Robust JSON output.** Output cap is `MAX_OUTPUT_TOKENS` (3072) to avoid mid-JSON truncation; Anthropic forces JSON with a system-prompt directive (assistant prefill 400s on current Claude models), OpenAI-style providers use `response_format: json_object`. Malformed/truncated output triggers a corrective retry loop (`MAX_PARSE_ATTEMPTS`).
- **Live model lists.** `listModels()` reuses each provider's catalog endpoint (`/models`, `/api/tags`); the config UI caches results in localStorage and falls back to the static `PROVIDER_MODELS` map.

## Conventions

- All provider adapters implement `LLMProviderAdapter` interface — swap providers with config change only.
- Validation functions return `ValidationResult` (same pattern as rules engine) — never throw.
- `sanitizeAIResponse` clamps out-of-range values rather than rejecting — player always gets a response.
- Error handling uses exponential backoff retry (2s, 4s, 8s) for retryable errors.
