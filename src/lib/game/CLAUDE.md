# Game Loop Engine

Pure-function game loop state machine and world state management. The engine handles phase transitions, world state mutations, and session persistence. AI calls and UI rendering are handled by components in `src/components/game/`.

## Structure

- `types.ts` — Game loop type definitions: phases, actions, session, pillars, transition maps.
- `state-machine.ts` — Pure reducer: `transition(session, action)` produces the next session state. Phase transitions validated against `VALID_TRANSITIONS`. `InvalidTransitionError` for illegal transitions.
- `world-state.ts` — World state mutation: `applyResolution()` orchestrates sanity, location, inventory, digestion, and memory updates from a validated AI response. Only AI-mutable fields (`location`, `activeQuests`, `npcsPresent`) are applied from `worldStateChanges`; pathway/sequence/maxSanity are rules-engine-only. `applyDigestion()` advances/reverses potion digestion from the AI's acting evaluation. A normal turn also calls `tickInjuries()` (from `combat.ts`) so combat injuries heal over play.
- `combat.ts` — Combat engine (issue #10). Pure, deterministic functions for hybrid combat: a mechanical **preparation** phase, an AI-narrated **exchange** of mid-fight decision points, and a **resolution**. `scorePreparation()` rates preparation 0-1 (capped per dimension; `AMBUSH_PREP_CAP` for ambushes); `computeSequenceGap()`, `computePathwayMatchup()` (Beyonder-vs-mundane edge, God-Almighty-vs-Eternal-Darkness opposition, kindred groups, same-pathway penalty), and `computeInjuryPenalty()` feed `computeBaseAdvantage()` together with a single injected `randomFactor`. `createEncounter()` → `applyPreparation()` → `chooseOption()` (×2-3) → `resolveEncounter()` walk the `CombatEncounter` mini state machine; `generateDecisionPoints()` builds pathway-flavoured options whose context-aware modifiers reward knowing your enemy and holding an edge. `resolveOutcome()` maps the final advantage + tactics to `victory`/`defeat`/`escape`/`stalemate`; `computeConsequences()` produces injuries, sanity drain (via `sanityDelta`), items gained/lost, and dropped characteristics. `applyCombatResult()` writes the result to `GameState` (sanity, inventory, `injuries`); `tickInjuries()` heals injuries each turn. **Design principle: preparation helps but never decides** — clever mid-fight choices can overturn a poor start and a single random factor keeps no fight certain. The AI layer only narrates (instruction `"combat"`).
- `inventory.ts` — Shared inventory helpers. `removeItemsByName(inventory, items)` is the single pure "drop these items by name" operation reused by combat (`applyCombatResult`), advancement (consuming the next potion's ingredients), and any other subsystem that removes carried items — one place so name-vs-id matching stays consistent.
- `digestion.ts` — Acting Method digestion engine (issue #8). Pure functions mapping the AI's acting `alignment` score (0-1) to 0-100 digestion progress for the current potion: `computeProgressDelta()`, `applyDigestionProgress()`, `createDigestionState()`, `isDigestionComplete()`, and deterministic in-world `digestionFeedback()`. In-character acting (alignment ≥ 0.5) advances digestion; contradictory acting (< 0.35) reverses it; non-contradictory acting always yields at least `MIN_PROGRESS_PER_SESSION` (anti-stagnation guardrail).
- `session.ts` — Session lifecycle: `createSession()`, `createDefaultGameState()` (seeds a fresh `digestion` state; accepts an optional `prologueRecap` pinned onto `GameState` for the prologue → story bridge, and an optional `startScenario` from `@/lib/lore/start-scenarios` for **varied story openings** — sets `GameState.location` to the scenario's (randomly varied) starting place and `GameState.openingBeat` to its first-turn beat; both fall back to the epoch default when absent, so legacy callers/tests are unaffected), serialization/deserialization with shape validation (`isValidDigestionShape()`; digestion is seeded on deserialize for legacy sessions saved without it; the durable `memory.runningSummary` is optional but rejected when present and not a string). Pure functions — no localStorage dependency. Accepts optional `initialMemory` to pre-populate memory (used for prologue facts). Anchor state (issues #35, #25): `GameSession.anchorState` is optional and strictly validated (`isValidAnchorStateShape`), exactly like `identityState` — absent on saves that never consecrated an anchor, rejected when malformed. RAG fields (issue #63): `GameSession.canonPosition` (the player's shared-timeline position feeding the retrieval timeline gate; starts at `DEFAULT_CANON_POSITION`, advanced only forward via `advanceCanonPosition()` — monotonic) and `GameSession.embeddingModelId` (the embedding model **locked at character creation** — validated against `APPROVED_EMBEDDING_MODELS`, no mid-save switching; deserialize rejects unapproved locks and seeds defaults for legacy saves).
- `lore-retrieval.ts` — The pure half of wiring RAG retrieval into the game loop (issues #63/#64): `buildRetrievalQuery(session)` composes the per-turn query from the current location, the most recent narrative beat (or the opening beat on turn 0), active quests, and present NPCs (capped at `RETRIEVAL_QUERY_CHAR_CAP`); `toRetrievedLoreChunks(chunks)` narrows retrieval rows to the `RetrievedLoreChunk` slice the prompt assembler consumes. The network glue (embedder + Supabase RPC, best-effort) lives in `components/game/lore-retrieval-client.ts`, outside the coverage mandate.
- `travel.ts` — Travel mechanics (issue #23). Pure module: `CITIES` (Tingen + Backlund, Trier, Bayam, plus the varied-start regions Pritz Harbor, Enmat Harbor, and Feysac — each with id/name/kingdom/public blurb; the id is the lowercase first word so it matches the lore `city` field and `cityNarrationDirective`), a symmetric `travelDays` table, `getCity`, `cityIdFromLocation` (resolve a `location` string by leading word), `canTravelTo(state, cityId)` (known destination, not the current city; an unknown current location may set out for any city), and `travelTo(state, cityId, turnNumber?)` returning a NEW `GameState` (location set to the city name, `npcsPresent` cleared) plus a memory-fact payload — or `null` when not permitted. Location is on the AI-mutable allowlist; this is the deliberate _player_ travel path. The React layer (`map-panel.tsx`) folds the fact in via `addSessionFact` and persists.
- `sanity.ts` — Sanity/madness engine (issue #9). Re-exports the tier classifier from `@/lib/ai` (the single source of truth, since `GameState` and the prompt assembly live there) and builds the game-facing concerns on top: `SANITY_EFFECTS` (per-tier UI-effect profiles — CSS class, distortion intensity, and which narrative degradations are active) consumed by the React layer via `sanityEffects()`; `sanityDelta(event)` mapping a `SanityEvent` (ability-use, horror-encounter scaled to sequence gap, advancement, outer-deity, acting-success, rest, human-connection, routine, acting-neglect) to a deterministic sanity delta (engine-owned, not AI); and loss-of-control evaluation at zero sanity — `isLossOfControl(state)` and `evaluateLossOfControl(ctx)` returning `setback | transformation | fatal` (higher sequences fail more catastrophically; `highRisk` escalates one step). Ready to integrate with the death/failure system (issue #12). The caller applies deltas via `applySanityImpact` (world-state.ts), which clamps to `[0, maxSanity]`.
- `death.ts` — Death & failure engine (issue #12). The rules engine — never the AI — owns the consequence: `evaluateFailure({cause, sequenceLevel, highRisk?, catastrophic?})` resolves a failure (`loss-of-control` via the sanity ladder; `combat-defeat`/`ritual-failure` are setbacks unless `catastrophic`, which escalates on the same sequence-scaled ladder) into `setback | permadeath`. `applySetback(state, random?, turn?)` = wake at 25% sanity, lose ~half the inventory, displaced to an unknown corner of the city, reputation memory fact. Permadeath: `buildLegacy(session, severity)` (transformation → loose **monster**; death → **legend** for Seq ≤6 else **cautionary tale**, with epitaph), `endSession()` sets `GameSession.ended` (the session is preserved as a historical record — inventory, memory, journal stay readable), `legaciesToFacts()` seeds a new same-timeline character's memory with the fallen (the world remembers), `fallbackDescentScene()` is the deterministic descent narration when no AI is configured, and strict legacy (de)serialization under `LEGACIES_KEY`. **Full restart** = wiping the legacy list (canonical baseline restored).
- `advancement.ts` — Sequence advancement (Seq 9 → 2), the rungs below the apotheosis endgame. The rules engine — never the AI — owns the climb. `advancementRequirements(session)`/`canAdvance` build a checklist from the **target** sequence's canon data (`getSequence`): a fully digested potion (Acting Method), the target potion's `prerequisiteItems` carried, a **mandatory Advancement Ritual from Sequence 5 onward** (a hard gate — the pathway must define one; it is performed and narrated during the attempt), anchors at the Saint tier (`anchorsRelevant`), and a sanity floor (`ADVANCEMENT_SANITY_RATIO`). All requirements are hard gates (`meetsRequirements`/`canAdvance`). `advancementSuccessChance` is high at low Sequences and falls per rung, **never reaching certainty** (a permanent loss-of-control chance), steadied by anchor surplus + sanity. `attemptAdvancement(session, random?)` returns `advanced` (engine decrements `sequenceLevel`, consumes the ingredients via `removeItemsByName`, re-seeds digestion via `createDigestionState`, drains sanity through `applySanityImpact`, seeds memory facts) or `lost-control` (an `evaluateFailure({cause:"loss-of-control"})` verdict — `highRisk` from an under-anchored high form / frayed mind — routed through the same setback/permadeath machinery as any failure). Canon source: novel + wiki (rituals begin at Seq 5). Pure + deterministic; the React layer narrates via the `"advancement"` instruction and persists.
- `potion-preparation.ts` — Next-potion acquisition (issue #84), the missing step that makes `advancement.ts`'s strict ingredient gate reachable in normal play (inventory is not AI-mutable, so nothing else grants the next potion's reagents). Pure + engine-owned like advancement. `potionPreparationPlan(session)` builds the target potion's checklist (each `prerequisiteItem` with `owned`, allowed `methods`, and purchase `cost`); `acquisitionMethodsFor(item, targetSeq)` makes the Beyonder Characteristic (main ingredient) **hunt-or-buy at shallow rungs but hunt-only from `MAIN_INGREDIENT_HUNT_ONLY_AT_OR_BELOW` (6) onward**, while the formula and supplementary reagents are purchase-only; `acquisitionCost`/`huntLootReward` scale with `acquisitionDepthFactor` (deeper potions cost far more — "higher Sequences require extra effort"). `purchasePotionItem(session, name)` spends funds (`marketplace` economy) and delivers the item (refusing `not-required`/`already-owned`/`not-purchasable`/`unaffordable`); `deliverHuntedItem(session, name)` grants a hunted Characteristic plus depth-scaled spoils (the React layer runs the combat encounter and calls this only on victory). Both append memory facts; the caller journals + persists. The combination is what lets the engine actually commit a sequence change (status bar updates) instead of the AI narrating an advancement the engine never made.
- `apotheosis.ts` — Sequence 0 endgame (issue #30). The True God ascent, owned by the rules engine. `apotheosisRequirements(session)`/`canAttemptApotheosis` gate the ritual on session-carried data only (Seq 1, fully-digested potion, the pathway `uniquenessItemFor` item in inventory, anchor `effectiveSupport ≥ requiredSupport(0)`, ≥ half sanity); `apotheosisSuccessChance` is strong-but-capped (anchor surplus + sanity), `attemptApotheosis(session, random?)` returns `ascended` (engine writes Seq 0 — consumes the Uniqueness, resets digestion, seeds memory facts + the one-time `ABOVE_SEQUENCE_TEASE`) or `unmade` (a catastrophic `ritual-failure` verdict from the death engine — Seq 1 = fatal — routed through the same permadeath machinery). Post-apotheosis: `drawPetition` (a `PETITION_CHANCE` per-turn worshipper memory fact), `trueGodName`/`TRUE_GOD_NAMES`, and `TRUE_GOD_ABILITIES`/`TRUE_GOD_ACTING` (the narrator framing that stands in for a rules `Sequence`, which Seq 0 has none of). Pure + deterministic under injected randomness; the godhood narration shift lives in `buildDemigodDirective` (ai/prompts.ts).
- `echoes.ts` — Cross-epoch timeline echoes (issue #31). When a chronicle ends, `mintArtifact(session, legacy)` mints a `TimelineArtifact` from the fallen character's signature possession (potion formula > first carried item > their journal as fallback), persisted under `ECHOES_KEY` next to the legacy list. A fresh character may discover echoes: `discoverableArtifacts` enforces the **paradox guard** (`originEpoch <= currentEpoch` — the past never inherits from the future; echoes are resonances, not unique objects, so multiple timelines may each find one), `pickStartingEcho` (≤1, ~50% chance, deterministic under injected randomness) seeds the inventory via `artifactToItem`, and `echoFacts` gives the narrator up to three foreshadow facts plus a concrete carried-item fact. Strict `serializeArtifacts`/`deserializeArtifacts`. Full timeline restart wipes `ECHOES_KEY` together with `LEGACIES_KEY`.
- `journal.ts` — Story journal engine (issue #11). Pure functions only: `validateJournalFlag` (the single validation point for the AI's optional `AIResponse.journalEntry` flag — malformed flags are dropped, never thrown), `deriveJournalEntries` (the AI flag plus deterministic detections from the state delta — advancement, travel, new-NPC encounters — deduped by event type with the AI's summary winning), annotation CRUD (`addAnnotation`/`editAnnotation`/`deleteAnnotation`; **player-only — never fed into prompt assembly**), `filterJournal` (text/type/npc/location/character), `journalToMarkdown` (arc-grouped export with annotations as blockquotes), and strict `serializeJournal`/`deserializeJournal`. Arc = the character's sequence label.
- `journal-sync.ts` — Journal persistence to Supabase (issue #11): `toEntryRow`/`toAnnotationRow` mappers and `syncEntries` (batched, after a turn) / `syncAnnotation` + `deleteAnnotationRemote` (immediate on edit) / `syncJournal`, over a minimal structural `JournalSyncClient`. `deleteSessionEntriesRemote(client, sessionId)` deletes every durable journal row for one character on removal (RLS scopes to the caller; annotations cascade via `entry_id`). Best-effort from the UI — the localStorage journal never blocks on the network. Tables: `journal_entries` / `journal_annotations` (migration `20260612160000`, owner-only RLS).
- `character-admin.ts` — Character management (deletion). `characterDeletionPlan(sessionId, sessionIndex)` is a PURE function returning the localStorage keys to remove (session save, journal, in-progress combat `COMBAT_KEY_PREFIX`, usage estimate `USAGE_KEY_PREFIX`) and the session index with that id stripped; the React layer executes the plan and the best-effort `deleteSessionEntriesRemote` (journal-sync). Cross-timeline `LEGACIES_KEY`/`ECHOES_KEY` are world memory and intentionally NOT removed — the separate full-timeline restart wipes those.
- `preferences.ts` — Player display preferences (issues #9, #13). `GamePreferences` (`sanityMeterVisible` default `false`; `highContrast` default `false` — legacy payloads without it remain valid), `DEFAULT_PREFERENCES`, strict shape validation, and pure `serializePreferences`/`deserializePreferences`/`mergePreferences` (fall back to defaults on missing/invalid data). localStorage access lives in the React layer (`PREFERENCES_KEY`).
- `achievements.ts` — Achievements & timeline divergence (issue #18). Pure evaluation over a `GameSession`: `ACHIEVEMENTS` (11 milestones derived ONLY from data the session carries — turns, sequence, digestion, injuries, identities, legacies in memory facts, funds, ended chronicles), `evaluateAchievements`, `divergenceScore` (an **honest heuristic** 0-100: story length, fallen predecessors remembered, leaving the canon start, identity webs, an ended chronicle — documented in-code, not a literal canon diff), `showcaseStats` (turns, combats won via memory facts, potions = rungs climbed, funds), `legacyCount`.
- `showcase-sync.ts` — Supabase glue for showcases/leaderboards (injectable structural client): `toShowcaseRow`, `publishShowcase` (explicit player action, upsert by user), `fetchLeaderboard` (public rows only — RLS hides private ones — ranked by sequence/achievements/divergence with a pathway filter). Table `player_showcases` (migration `20260613010000`).
- `marketplace.ts` — Item marketplace engine (issue #16). Introduces currency: `GameState.funds` in pence (optional for old saves; `getFunds` seeds `STARTING_FUNDS`), `adjustFunds` (floored at 0) and `canAfford`. `validateListing` is the anti-exploit gate (item must actually be carried with a rules-engine category; integer price under a sanity cap), `removeItemForListing` escrows the item out of the save at listing time (cannot list twice or keep while listed), `addItemToInventory` delivers purchases/returns, `PRICE_GUIDANCE` per category, `filterListings` for browsing (active + unexpired only). Honest framing: the server guarantees marketplace-row legitimacy and atomic purchase; funds/escrow live client-side with the rest of the save, by architecture.
- `marketplace-sync.ts` — Supabase glue (injectable structural client): `fetchActiveListings`, `fetchOwnHistory` (seller+buyer ledger), `createListing`, `delist`, and `purchase` via the `purchase_listing` SECURITY DEFINER RPC (row-locked, self-purchase and stale listings rejected server-side; the ONLY path to 'sold'). Table `market_listings` (migration `20260613000000`).
- `world-messages.ts` — Shared world messages (issue #17). Safety is structural: a message is a TEMPLATE id + slot fills validated against a fixed vocabulary (`composeMessage` throws on anything else — arbitrary text cannot exist). `MESSAGE_TEMPLATES` covers warnings/tips/lore/humor; `selectMessagesForScene` does helpfulness-weighted sampling capped at `MAX_MESSAGES_PER_SCENE` (2) with heavily-downvoted spam buried, deterministic under an injected random source.
- `world-messages-sync.ts` — Supabase glue (injectable structural client): `fetchLocationMessages`, `placeMessage`, `rateMessage` (the `rate_world_message` SECURITY DEFINER RPC — one vote per player, repeats are silent no-ops). Tables `world_messages` / `world_message_votes` (migration `20260612230000`): shared authenticated read, owner insert/delete.
- `identity.ts` — Identity management (issue #22). Pure engine: `identityCapability(pathwayId, seq)` gates the **full** tier by pathway ability thresholds (`FULL_IDENTITY_THRESHOLDS` — Fool ≤6 "Faceless", Error ≤8 Swindler line) with a one-disguise **basic** tier for everyone else (`activeDisguise`, frays 2×); `createIdentity`/`discardIdentity`/`switchIdentity` (deliberate, never automatic; `null` = true face); per-identity `reputation` (clamped −100..100) and `knownBy` NPC awareness (`recordIdentityUse` — NPCs learn the active face only, never cross-reference); exposure: risk accrues per use/new witness, and past `EXPOSURE_EVENT_THRESHOLD` a witness who knows two faces may connect them (`checkExposure`, injected randomness) — `applyExposure` links them, dents both reputations with that NPC, resets risk to a simmer. `identityPromptContext` renders the one narrator-facing presentation line (threaded through `GenerateOptions.identityContext` → an `## Active Identity` system layer); `GameSession.identityState` is optional + strictly validated.
- `society.ts` — Secret society / Tarot Club system (issue #32). `societyKindForPathway` (Fool founds the Tarot Club proper; Death/Darkness a Nighthawks squad, Sun a church division, Tyrant a pirate crew, others a scholars' circle), founding gated at Seq ≤7 (`canFoundSociety`). AI-NPC members carry code names, pathway hints, dispositions (−100..100) and their own slow arcs; `holdGathering` ("above the gray fog", `GATHERING_COOLDOWN_TURNS` apart) yields MECHANICAL consequences — intel **memory facts** the narrator/investigation pillar consume, occasional traded items, disposition drift, arc creep — deterministic under injected randomness; `resolveMemberArc` settles a fully-advanced arc for trust. `GameSession.societyState` optional + strictly validated.
- `anchors.ts` — Anchors system (issues #35, #25). The tier-gated stabilising resource that opposes a high-Sequence Beyonder's godhood/mythical-form pressure. **Pure + additive — imports the sanity and death/failure ladders, never duplicates them.** `Anchor {id, kind: object|place|congregation, name, integrity 0-100}` + `AnchorState` (on `GameSession.anchorState`, optional + strictly validated via `isValidAnchorStateShape`, mirroring `identityState`). `requiresAnchors(seq)` (mandatory at Seq ≤ 2 — Angels) and `anchorsRelevant(seq)` (Saint tier, Seq ≤ 4). `effectiveSupport` (Σ integrity × `ANCHOR_KIND_WEIGHT`, congregation > place > object — "the faith of the mass" is strongest) vs `requiredSupport(seq)` (0 above Saint, constant for Saints, rising for Angels) → `supportShortfall`. **Integration helpers the game loop calls (game-loop.tsx is NOT modified here):** `anchorPressure(state, seq)` returns a non-positive per-turn sanity delta (fold into the same accumulation as `sanityDelta(event)`, then `applySanityImpact`); `anchorHighRisk(state, seq)` maps an under-anchored Beyonder onto the failure engine's `highRisk` flag (thread into `evaluateFailure`/`evaluateLossOfControl`) — this is canon anchor theft tipping an advancement into loss of control. `consecrateAnchor`/`damageAnchor`/`repairAnchor`/`loseAnchor` (pure, integrity clamped 0-100; `damageAnchor` keeps a zeroed anchor re-growable, `isLost` reports it). `DEMIGOD_TRAITS` = per-tier (Seq 4-1) Saint/Angel/King-of-Angels lifespan + mythical-form + prayer-response flavour data for the character sheet (pure data, not fed to the AI). Design doc: `docs/anchors-design.md`.
- `input-modes.ts` — Context-dependent input modes (issue #24). `detectInputMode(choices, narrative)` infers the scene's mode from the AI's own choice mix (plurality; narrative keywords only as tiebreaker): exploration/ritual/dialogue/investigation (combat is exogenous — an active encounter takes the surface). Guided modes COMPOSE one free-text action (`composeRitualAction` with `RITUAL_STEPS` + material selection, `composeDialogueAction`, `composeDeduction` over `gatherClues`) and submit through the validated issue-#19 pipeline — one resolution path serves every mode.
- `free-text.ts` — Free-text player actions (issue #19). Pure pipeline: `validateFreeText` (normalize; reject empty/over-280-chars/exploit declarations like "I become Sequence 0" — the hard guard is the engine's existing clamps: AI-mutable allowlist, sanity clamp, rules-engine-only advancement), `freeTextRejection` (in-world narration, never an error message), and `freeTextToChoice` (wraps valid text as the synthetic `FREE_TEXT_CHOICE_ID` choice with a pillar inferred from its verbs, so the unchanged SELECT_CHOICE → resolution machinery — sanity, acting, journal — handles it).
- `tutorial.ts` — Tutorial prologue data (issue #14): `TUTORIAL_SCENES`, four hand-authored scenes that teach Beyonders/secrecy, the potion risk, the acting method, and pathway choice **through fiction** with one plain lesson line each; `TUTORIAL_REQUIRED_TOPICS` is the coverage contract tests enforce. Pure data — skippable, no gameplay state, never fed to the AI.
- `prologue.ts` — Prologue engine: 4 static narrative scenes, pathway affinity scoring, recommendation algorithm, and `createPrologueMemory()` to seed session facts from character creation choices. Also the **live, deterministic decision path** for the AI prologue (issue #53), generic over any number of pathways (never assumes 4): `tallyAffinities(weightMaps)` sums per-pathway affinity weights across the player's picked choices; `rankPathways(tally)` sorts into descending `PathwayScore[]` (ties break by ascending id); `selectTopCandidates(tally, {count=3, min=2})` returns the ranked candidate ids for the finale (top `count`, all ties at the cutoff, with a `min` floor); `dominantAffinity(map)` is the argmax pathway id of a single affinity map. The engine — not the LLM — decides the candidate set; the player picks the final potion. `recommendPathway` is de-biased: with no affinity signal it returns `pathwayId: 0` (none) rather than defaulting to Seer/pathway 1. **`buildPrologueRecap({choices, finaleNarrative?, chosenPotion?})`** composes the compact, durable recap of the AI prologue (the defining choices, the finale encounter, the potion drunk; finale capped at `RECAP_FINALE_CHAR_CAP`) that character creation pins into `GameState.prologueRecap` — the prologue → story bridge, since the story narrator runs on a separate prompt that never sees the prologue scenes. Returns `""` when there is nothing to recap (the manual path). Exported via `index.ts`.

## Game Loop Phases

```
idle → situation → choices → resolution → consequences → situation (loop)
any active phase → error (on failure)
error → situation (retry) | idle (abandon)
```

## Key Design Decisions

- **Pure state machine.** `transition()` is a pure function — no side effects, no async. The UI orchestrates AI calls and storage.
- **Separation of concerns.** Phase transitions and world state mutations are independent operations composed by the caller.
- **AI-mutable field allowlist.** Only `location`, `activeQuests`, `npcsPresent` can be changed via AI `worldStateChanges`. Pathway, sequence, and max sanity are rules-engine-only.
- **Session serialization is pure.** `serializeSession`/`deserializeSession` operate on strings. localStorage access is in the UI layer.

## Conventions

- All functions are pure and synchronous (except AI calls in UI).
- Test patterns follow `src/lib/rules/rules.test.ts`.
- Session shape validation is strict — unknown JSON is rejected rather than coerced.
