# Game Loop Engine

Pure-function game loop state machine and world state management. The engine handles phase transitions, world state mutations, and session persistence. AI calls and UI rendering are handled by components in `src/components/game/`.

## Structure

- `types.ts` ‚Äî Game loop type definitions: phases, actions, session, pillars, transition maps.
- `state-machine.ts` ‚Äî Pure reducer: `transition(session, action)` produces the next session state. Phase transitions validated against `VALID_TRANSITIONS`. `InvalidTransitionError` for illegal transitions. **`ENGINE_RESOLUTION`** is the action for an engine-decided turn (advancement / apotheosis): the engine has already committed the outcome onto `gameState`, so it carries a narration-only `ValidatedAIResponse` straight from `choices` ‚Üí `consequences` (the one extra edge in `VALID_TRANSITIONS.choices`), setting `currentNarrative`/`lastResolution`/`pendingPlayerAction` and clearing `selectedChoiceId`. The normal `handleContinue` then commits it like any turn ‚Äî so the climb lands in the transcript AND a memory turn record (the narrator stays aware), with no side-channel banner. `pendingPlayerAction` (the player-action text for that turn) is cleared on `APPLY_CONSEQUENCES`.
- `world-state.ts` ‚Äî World state mutation: `applyResolution()` orchestrates sanity, location, inventory, **funds**, digestion, acting-method discovery, and memory updates from a validated AI response. **Hybrid sanity (issue #95):** per-turn sanity = engine-scored `sanityDeltaForTags(response.sanityEventTags, pre-mutation sequence)` + the AI's residual `sanityImpact` clamped to ¬±`SANITY_RESIDUAL_CAP` (5); both fold into one `applySanityImpact`. It also takes a trailing `actingMethodState` param (caller passes `resolveActingMethodState(session.actingMethodState)`), runs `evaluateActingDiscovery` from the acting eval + `actingMethodTaught` flag, appends a diegetic `actingMethodDiscoveryFact` on discovery, and returns the extended shape `{ gameState, memory, digestionDelta, sanity: {tagDelta, residual, total}, actingMethodState, discovery: {discoveredThisTurn, trigger} }`. Only AI-mutable fields (`location`, `activeQuests`, `npcsPresent`) are applied from `worldStateChanges`; pathway/sequence/maxSanity are rules-engine-only. **Movement gate (issue #101):** `applyWorldStateChanges(state, changes, opts)` now returns `{state, facts}` (`opts` = `{epoch, gateEnabled, trackedNpcState, turnNumber}`) and routes every `location` change through `gateLocationChange` ‚Äî a cross-city teleport without a valid `involuntaryCause` (`isInvoluntaryMoveCause` over `INVOLUNTARY_MOVE_CAUSES` = abduction/forced-passage/capability-gated-teleport) is **refused** (location unchanged, an in-world redirect fact emitted ‚Äî narration, never a throw, like `discoveredItemLeadFact`/`freeTextRejection`); a within-city/provisional move passes; an against-the-will move passes with an "against your will" fact. The gate anchors the origin city to the engine-tracked `GameState.currentCity` (passed as `fromCity`) when the `location` STRING is a bare district that names no city ‚Äî so a teleport out of a district-level location ("a medical facility" ‚Üí "Bayam") is still caught instead of slipping through as `provisional`. **Access-gated continent (world build-out, issue #130):** `gateLocationChange` also takes the `sequenceLevel`/`accessFlags` slice and, when the destination resolves to a city on a non-central, access-gated continent (the Forsaken Land), **refuses** the move unless `meetsAccessGate` passes ‚Äî checked FIRST, so it bites **even on `involuntaryCause:"capability-gated-teleport"` AND even when the movement-realism gate is off** (a hard canon boundary, not a pacing preference); the narrator can never route an unqualified character into the sealed continent. `applyWorldStateChanges` threads `next.sequenceLevel`/`next.accessFlags` into the call. **In-play flag grant (world build-out 3, issue #132):** after the `location` is applied, `applyWorldStateChanges` checks `reachedDreamWorldGate(next.location)` and, when true and the flag isn't held, `grantAccessFlag`s `dream-world-passage` + emits an `event` fact ‚Äî the path by which a NON-origin character earns entry to the Forsaken Land by reaching the Dream-World shadow of Giant King's Court through the story (the physical forsaken city stays access-gated; only the dream threshold grants). `gateLocationChange` returns a `crossCity` flag: on a **cross-city** relocation the scene cast is reset to the roster's followers (`reassertFollowersAt([], ‚Ä¶)` ‚Äî the origin's incidental NPCs are left behind), while a **within-city/provisional** nudge PRESERVES the existing cast and only re-asserts the followers into it (`reassertFollowersAt(next.npcsPresent, ‚Ä¶)`) ‚Äî a local move no longer wipes mid-scene NPCs. An explicit `npcsPresent` write likewise re-asserts the followers, so the AI can never drop a follower (engine-truth-over-AI-string, like `advanceActiveHunts`). An applied location change also updates the engine-tracked `GameState.currentCity` when the new location names a known city, and PRESERVES it for a bare district string (so the map's per-city atlas stays oriented ‚Äî `cityForLocation` is the resolver, `travelTo`/`createDefaultGameState` seed it too), and then calls `registerCustomLocation` (`location.ts`, Backlund location sync) so a narrator-named venue that matches no known district is filed under the resolved city as a `GameState.customLocations` entry the map can render and pin. `applyResolution` gained trailing `epoch`, `trackedNpcState`, `movementGateEnabled` params (precedent: the trailing `actingMethodState` param) and folds the gate's returned facts into memory via `addSessionFact`. Deliberate map travel (`travelTo`) is the trusted player path and is NOT run through the gate. Money found/lost in the fiction arrives via the dedicated `AIResponse.fundsDiscovered` channel (NOT `worldStateChanges`, like `itemsDiscovered`): clamped to ¬±`FUNDS_DISCOVERED_CAP` per turn and applied through `adjustFunds` (floors the wallet at 0) ‚Äî the wallet updates so found money is spendable on ingredients, bounded so a fortune can't be conjured in one turn. `applyDigestion()` advances/reverses potion digestion from the AI's acting evaluation. A normal turn also calls `tickInjuries()` (from `combat.ts`) so combat injuries heal over play. **AI-discovered items are partitioned (issue #90):** `partitionDiscoveredItems()` splits `itemsDiscovered` into the items the player may carry (`mundane` loot + the narrator-grantable `uniqueness` artifact) and the advancement-critical reagents (`main-ingredient`/`supplementary-ingredient`/`potion-formula`, via `isReagentCategory`) the AI may NOT mint ‚Äî only the carried kinds reach inventory, and each blocked reagent becomes a `quest-progress` story lead via `discoveredItemLeadFact()` instead (so narration cannot bypass the purchase/hunt gates in `potion-preparation.ts`). The turn record is built from a response carrying only the carried items, so the memory fact extractor / recent-summary never report a blocked reagent as "discovered". The consequences panel (`game-loop.tsx`) renders the same partition. **`narrationOnly(response)`** strips an `AIResponse` down to its narrative (and optional journal flag) for an engine-decided turn (advancement / apotheosis) routed through `ENGINE_RESOLUTION` ‚Äî the engine already applied every mechanical effect, so feeding the full response back through `applyResolution` would double-apply sanity/digestion/items/world changes; narration-only keeps the turn-loop integration (transcript + memory) without re-deciding anything.
- `hunt.ts` ‚Äî Hunt quests (advancement/combat streamline). Hunting a Beyonder Characteristic for the next potion no longer jumps straight to combat: it starts a tracked, multi-turn QUEST to find the creature. `HuntState {targetItemName, targetSeq, turnsRemaining, totalTurns}` lives as a **list** on `GameSession.hunts` (several hunts can run at once), optional + strictly validated (`isValidHuntStateShape`/`isValidHuntsShape`) and preserved on the deserialize `...s` spread, exactly like `actingMethodState`. `huntTrackingTurns(targetSeq)` scales the search length with the same `acquisitionDepthFactor` as the potion economy (deeper rungs take longer) off `HUNT_BASE_TRACKING_TURNS`. `startHunt(session, itemName)` validates like `deliverHuntedItem` (missing, huntable prerequisite for the target rung; honors `MAIN_INGREDIENT_HUNT_ONLY_AT_OR_BELOW`) and refuses `not-required`/`already-owned`/`not-huntable`/`already-hunting`; on success it records the hunt, seeds a `quest-progress` fact, and adds the AI-visible `huntQuestLabel` to `activeQuests`. `advanceHunt` decrements one hunt (floored at 0 = ready); `advanceActiveHunts(session)` advances all each turn and idempotently re-adds dropped quest labels (engine truth is `hunts`, never the AI-mutable string). `isHuntReady` gates the engage step; `clearHunt(session, itemName)` removes a hunt + its label on a victorious kill or abandonment; `findHunt` looks one up. Pure; the React layer runs the combat encounter (existing `CombatEncounter.huntTarget` plumbing ‚Üí `deliverHuntedItem` on victory) and persists.
- `combat.ts` ‚Äî Combat engine (issue #10). Pure, deterministic functions for hybrid combat: a mechanical **preparation** phase, an AI-narrated **exchange** of mid-fight decision points, and a **resolution**. `scorePreparation()` rates preparation 0-1 (capped per dimension; `AMBUSH_PREP_CAP` for ambushes); `computeSequenceGap()`, `computePathwayMatchup()` (Beyonder-vs-mundane edge, God-Almighty-vs-Eternal-Darkness opposition, kindred groups, same-pathway penalty), and `computeInjuryPenalty()` feed `computeBaseAdvantage()` together with a single injected `randomFactor`. `createEncounter()` ‚Üí `applyPreparation()` ‚Üí `chooseOption()` (√ó2-3) ‚Üí `resolveEncounter()` walk the `CombatEncounter` mini state machine; `generateDecisionPoints()` builds pathway-flavoured options whose context-aware modifiers reward knowing your enemy and holding an edge. `resolveOutcome()` maps the final advantage + tactics to `victory`/`defeat`/`escape`/`stalemate`; `computeConsequences()` produces injuries, sanity drain (via `sanityDelta`), items gained/lost, and dropped characteristics. `applyCombatResult()` writes the result to `GameState` (sanity, inventory, `injuries`); `tickInjuries()` heals injuries each turn. **Design principle: preparation helps but never decides** ‚Äî clever mid-fight choices can overturn a poor start and a single random factor keeps no fight certain. The AI layer only narrates (instruction `"combat"`). **Dynamic mid-fight options:** `createEncounter` captures the player's `availableAbilities` (from `sequenceAbilities`) and `availableArtifacts` (inventory minus reagents) onto the encounter, and `generateDecisionPoints` appends, beyond the templated trio, up to `MAX_DYNAMIC_ABILITY_OPTIONS` named ability options and `MAX_DYNAMIC_ARTIFACT_OPTIONS_PER_POINT` carried-artifact options per point ‚Äî selected deterministically by point index (no new randomness, so a serialized fight regenerates identical points). A chosen dynamic artifact carries its `artifactItem`; `computeConsequences` adds those to `itemsLost` distinctly from the sealed-prep `consumesArtifact` slice (no double-count). **`enemyIntel(enemy, intel, playerSequence)`** is the pure, intel-gated opponent presenter (`none` ‚Üí name/description only; `partial` ‚Üí coarse strength + pathway; `thorough` ‚Üí exact sequence + `knownAbilities`); the combat UI formats it. **`combatNarrationContext(encounter)`** is a pure plain-language summary of the choices the player has committed ‚Äî preparation (intelligence/terrain/readied abilities/sealed artifacts/ritual materials) plus the chosen tactical options in order ‚Äî appended to the combat-narration `playerAction` (`combat-encounter.tsx`) so the narrated fight matches what the player actually did (and, with the threaded `identityContext`/`profileContext`, the face they wear) rather than a generic clash; `""` before anything is committed.
- `inventory.ts` ‚Äî Shared inventory helpers. `removeItemsByName(inventory, items)` is the single pure "drop these items by name" operation reused by combat (`applyCombatResult`), advancement (consuming the next potion's ingredients), and any other subsystem that removes carried items ‚Äî one place so name-vs-id matching stays consistent. `hasItem(inventory, name)` is the name-only carried check; `hasItemMatching(inventory, item)` matches **both category and name** ‚Äî the advancement, potion-preparation, and apotheosis gates use it (issue #90) so a `mundane` item that merely shares a reagent's name can neither satisfy a prerequisite nor soft-lock its acquisition, staying consistent with the canonical category+name gate in `@/lib/rules` (`laws.ts`). `REAGENT_CATEGORIES`/`isReagentCategory(category)` is the single source of truth for "is this an advancement reagent?" ‚Äî `world-state` blocks exactly these from AI item discovery and `marketplace` lists exactly these, so the engine-block and the market-sellable set can never drift.
- `digestion.ts` ‚Äî Acting Method digestion engine (issue #8). Pure functions mapping the AI's acting `alignment` score (0-1) to 0-100 digestion progress for the current potion: `computeProgressDelta()`, `applyDigestionProgress()`, `createDigestionState()`, `isDigestionComplete()`, and deterministic in-world `digestionFeedback()`. In-character acting (alignment ‚â• 0.5) advances digestion; contradictory acting (< 0.35) reverses it; non-contradictory acting always yields at least `MIN_PROGRESS_PER_SESSION` (anti-stagnation guardrail). **Pace slowed in issue #95** (`MAX_PROGRESS_PER_EVAL` 12‚Üí10, `MIN_PROGRESS_PER_SESSION` 2‚Üí1, flat across Sequences). `digestionFeedback(role, state, delta, knowsMethod)` takes a trailing **`knowsMethod`** flag (no default ‚Äî the compiler flags every caller): pre-discovery it returns vague, cryptic prose that names no mechanic/number and does NOT branch on the progress band (one line per direction, so magnitude never leaks); post-discovery it returns the clearer progress-tiered prose.
- `acting-method.ts` ‚Äî Acting-method discovery (issue #95). The Acting Method is LOTM _secret knowledge_ a Beyonder must earn through play; `ActingMethodState {knowsMethod, alignedStreak}` (on `GameSession.actingMethodState`, optional + strictly validated via `isValidActingMethodStateShape`, lazily resolved via `resolveActingMethodState`, mirroring `profileState` ‚Äî NOT seeded by `createSession`). `evaluateActingDiscovery({state, alignment?, taughtFlag})` short-circuits once known, then discovers it via **taught** (an AI-flagged NPC teaching / lore reading ‚Äî immediate) or **repetition** (the streak grows on aligned acting ‚â• `ACTING_ALIGNED_THRESHOLD` 0.5, resets on a lapse, and crossing `ACTING_DISCOVERY_STREAK` 6 flips it). Pre-discovery the digestion meter/number stays hidden and feedback is vague; the tutorial does NOT grant it and a neglect scare is NOT a trigger. Pure.
- `session.ts` ‚Äî Session lifecycle: `createSession()`, `createDefaultGameState()` (seeds a fresh `digestion` state; accepts an optional `prologueRecap` pinned onto `GameState` for the prologue ‚Üí story bridge, and an optional `startScenario` from `@/lib/lore/start-scenarios` for **varied story openings** ‚Äî sets `GameState.location` to the scenario's (randomly varied) starting place and `GameState.openingBeat` to its first-turn beat; both fall back to the epoch default when absent, so legacy callers/tests are unaffected ‚Äî and an optional `startArchetype` from `@/lib/lore/start-archetypes` for **start archetypes** (issue #131, begin embedded in an NPC's circle): the archetype's `location`/`openingBeat` take precedence and its `archetypeGrounding` line is folded into the durable `characterBackground` (issue #92's never-trimmed-context insight). **Access-gated origin (world build-out 3, issue #132):** when the chosen start (scenario OR archetype) carries `origin: "forsaken-land"`, `createDefaultGameState` seeds `accessFlags: ["dream-world-passage"]` (and `currentCity` from the location) so a Forsaken-native begins able to move within their sealed continent; a central start grants nothing. **`seedArchetype(session, archetype, now?)`** then applies the GameSession-level seeds AFTER `createSession`: tracked-ally roster (`joinRoster` ally+follows per `seeds.trackedAllies`), an optional society pre-membership (`seedSocietyMembership` from `society.ts`, bypassing the Seq-gated `foundSociety`), and `seeds.facts` relationship facts (via `addSessionFact`). A plain location start passes no archetype and is unchanged.), serialization/deserialization with shape validation (`isValidDigestionShape()`; digestion is seeded on deserialize for legacy sessions saved without it; `GameState.currentCity` is backfilled on deserialize from `cityIdFromLocation(location)` when absent ‚Äî so the map's per-city atlas opens on the right city for legacy saves instead of reverting to Tingen ‚Äî and left unresolved for a bare-district location that names no city; deserialize also runs `registerCustomLocation` (`location.ts`) so a save parked at an off-map narrator-named venue files it under its city on load; `GameState.customLocations` is optional, strictly validated via `isValidCustomLocationsShape`, and preserved on the `...gs` spread; `GameState.accessFlags` (world build-out, issue #130 ‚Äî capability flags like `dream-world-passage`) is optional, strictly validated via `isValidAccessFlagsShape` (from `travel.ts`), and likewise rides the `...gs` spread (no seeding ‚Äî a legacy save without it stays valid); the durable `memory.runningSummary` is optional but rejected when present and not a string). Pure functions ‚Äî no localStorage dependency. Accepts optional `initialMemory` to pre-populate memory (used for prologue facts). Anchor state (issues #35, #25): `GameSession.anchorState` is optional and strictly validated (`isValidAnchorStateShape`), exactly like `identityState` ‚Äî absent on saves that never consecrated an anchor, rejected when malformed. True-self profile (character-info storage): `GameSession.profileState` is optional and strictly validated (`isValidProfileStateShape`), same pattern ‚Äî lazily seeded for legacy saves via `resolveProfileState`, preserved on the deserialize `...s` spread. Acting-method discovery (issue #95): `GameSession.actingMethodState` is optional and strictly validated (`isValidActingMethodStateShape`), same pattern ‚Äî lazily resolved via `resolveActingMethodState`, preserved on the `...s` spread, never seeded by `createSession`. Hunt quests: `GameSession.hunts` (a list ‚Äî multiple concurrent hunts) is optional and strictly validated (`isValidHuntsShape` from `hunt.ts`), same pattern ‚Äî preserved on the `...s` spread, absent on saves with no active hunts. Tracked-NPC roster (issue #101): `GameSession.trackedNpcState` is optional and strictly validated (`isValidTrackedNpcStateShape` from `tracked-npcs.ts`), same pattern ‚Äî preserved on the `...s` spread, absent on saves that never rostered anyone. RAG fields (issue #63): `GameSession.canonPosition` (the player's shared-timeline position feeding the retrieval timeline gate; starts at `DEFAULT_CANON_POSITION`, advanced only forward via `advanceCanonPosition()` ‚Äî monotonic) and `GameSession.embeddingModelId` (the embedding model **locked at character creation** ‚Äî validated against `APPROVED_EMBEDDING_MODELS`, no mid-save switching; deserialize rejects unapproved locks and seeds defaults for legacy saves).
- `lore-retrieval.ts` ‚Äî The pure half of wiring RAG retrieval into the game loop (issues #63/#64): `buildRetrievalQuery(session)` composes the per-turn query from the current location, the most recent narrative beat (or the opening beat on turn 0), active quests, and present NPCs (capped at `RETRIEVAL_QUERY_CHAR_CAP`); `toRetrievedLoreChunks(chunks)` narrows retrieval rows to the `RetrievedLoreChunk` slice the prompt assembler consumes. The network glue (embedder + Supabase RPC, best-effort) lives in `components/game/lore-retrieval-client.ts`, outside the coverage mandate.
- `travel.ts` ‚Äî Travel mechanics (issue #23). Pure module: `CITIES` (Tingen + Backlund, Trier, Bayam, plus the varied-start regions Pritz Harbor, Enmat Harbor, and Feysac ‚Äî each with id/name/kingdom/public blurb; the id is the lowercase first word so it matches the lore `city` field and `cityNarrationDirective`), a symmetric `travelDays` table **built once from a `{a,b,days}` leg list** (`TRAVEL_LEGS`, the #85 idea) so both halves can never be hand-broken, `getCity`, `cityIdFromLocation` (resolve a `location` string by its leading word(s) ‚Äî **longest leading slug first**, so a multi-word Forsaken id resolves while a single-word central id still wins), `canTravelTo(state, cityId)` (known destination, **meets its access gate**, not the current city; an unknown current location may set out for any reachable city), and `travelTo(state, cityId, turnNumber?, trackedNpcState?)` returning a NEW `GameState` (location set to the city name, `npcsPresent` set to the roster's **followers** via `reassertFollowersAt` ‚Äî the default empty roster reproduces the legacy "clear `npcsPresent`" behaviour, issue #101) plus a memory-fact payload ‚Äî or `null` when not permitted. Location is on the AI-mutable allowlist; this is the deliberate _player_ travel path (the trusted path ‚Äî NOT run through the movement gate). The React layer (`map-panel.tsx`) passes `session.trackedNpcState`, folds the fact in via `addSessionFact`, and persists. **Access-gated continents (world build-out, issue #130):** `City.continent?: "central" | "forsaken-land"` (absent = central; the 7 existing cities untouched) and `City.accessGate?: {minSequence?; requiresFlag?: AccessFlag}` (absent = freely reachable; sequences count DOWN so "high enough" = `state.sequenceLevel <= minSequence`, and `requiresFlag` demands a held capability). The Forsaken Land of the Gods (the sealed Eastern Continent) adds two **content-less, gated** cities ‚Äî `silver-city` + `giant-kings-court`, both `accessGate: {requiresFlag:"dream-world-passage"}` ‚Äî so no character can see or reach them until #3 grants the flag. `continentOf(city)` (absent ‚Üí central), `crossesContinent(a,b)`, and the fixed `CONTINENT_CROSSING_DAYS` (cross-continent journeys are not a matter of distance ‚Äî `travelDays()` returns `null` for a cross-continent pair and `travelTo` applies the constant instead). `meetsAccessGate(state, city)` (sequence AND flag, reused by `canTravelTo` and the AI-move gate), `hasAccessFlag(state, flag)`, and `isValidAccessFlagsShape` (the strict persistence guard for `GameState.accessFlags`, mirroring `isValidCustomLocationsShape`). **In-play flag grant (world build-out 3, issue #132):** `grantAccessFlag(state, flag)` (idempotent add ‚Äî the engine grants flags, the AI never can) and `reachedDreamWorldGate(location)` (detects the **Dream-World shadow** of Giant King's Court ‚Äî requires BOTH "dream" and "giant king" in the string, so the PHYSICAL forsaken city never triggers it). `world-state.ts` wires these so a NON-origin character earns `dream-world-passage` by reaching the dream gate through the story.
- `place-graph.ts` ‚Äî Reachability graph (issue #101). Pure, derived on call. Keys on the ONE unambiguous signal ‚Äî a location whose **leading word names a known city** (`cityForLocation` = `cityIdFromLocation`, Fifth-Epoch only). `isReachable(from, to, epoch, fromCityId?)` ‚Üí `{reachable, reason: "same-city"|"provisional"}` or `{reachable:false, reason:"cross-city"}`: a move is **blocked only when BOTH endpoints resolve to a known, DIFFERENT city** (e.g. "Tingen City" ‚Üí "Bayam"). The origin city is the `from` string's leading word, falling back to `fromCityId` ‚Äî the engine-tracked `GameState.currentCity` ‚Äî when the string names no city, so a teleport OUT of a district-level location ("a medical facility" ‚Üí "Bayam") is still caught (the common real-world case: players sit at districts/landmarks, not bare city names). An unknown/stale `fromCityId` is ignored (validated via `getCity`) rather than false-blocking. A district/landmark destination, an unrecognised place, or a non-Fifth epoch ‚Üí `provisional` (reachable). This deliberately does NOT attribute a bare district string to a city by keyword: the per-city gazetteers share keywords, so mapping "the docks"/"Hillston Borough" to a city would FALSE-block legitimate within-city moves ‚Äî only the unambiguous tracked `currentCity` anchors the origin. Fail-open (an unresolved endpoint is reachable) remains the safe default ‚Äî the residual cost is that a teleport to a destination phrased without a city-naming leading word slips through, never that a real within-city move is refused. **Fifth-Epoch only** (the CITIES/travel model is, like `gazetteer.travelEnabled`/`regions.ts`): earlier epochs never gate (Risk 5).
- `location.ts` ‚Äî Location resolution (Backlund location sync). The ONE pure place that turns `GameState.location` + the engine-tracked `currentCity` into a single answer the character sheet, the map atlas, AND the "you are here" marker all read, so they can never disagree (the bug: a Backlund chapel rendered on Tingen's map, pinned to the Cathedral of Serenity). `resolveLocation(state, epoch)` ‚Üí `ResolvedLocation {venue, place, cityId, cityName, cityUnknown, districtSlug}`: Fifth-Epoch resolves the city (tracked `currentCity` first, then the location's leading word via `resolveCityId`) and matches the **city-prefix-stripped** venue against that city's districts PLUS its custom venues; earlier epochs match the era region list directly (no city model). `mapAtlasFor(state, epoch)` builds the map's atlas ‚Äî merges the character's custom venues into the current city's districts, and returns the neutral `uncertainFifthGazetteer()` (no districts, all central cities travel-reachable) when a Fifth-Epoch city can't be resolved instead of silently defaulting to Tingen, and threads the character's `accessFlags` so the farther-afield list is continent-filtered (issue #130). `matchDistrictSlug` pins by the LONGEST distinctive keyword at **word boundaries** (`containsWord`) and refuses to pin on a `GENERIC_KEYWORDS` place-noun alone (a curated set of building-type words ‚Äî chapel/church/cathedral/shrine/sanctuary/basilica/abbey/monastery/temple ‚Äî exactly the substring class that mis-pinned the chapel; no gazetteer district relies on one as its sole distinctive keyword). `stripCityPrefix` strips a leading city ONLY when a real separator (‚Äî, ‚Äì, -, :, ,) follows it ‚Äî a bare space ("Backlund Bridge") means the city word is part of the place's own name and is left intact (otherwise the real Backlund Bridge district would be lost), which also rejects a city word that is merely a prefix of a longer word ("Trierston"). `registerCustomLocation(state, epoch)` files a narrator-named venue that matches no known district (and whose leading word is not itself a known city) under the resolved city as a `customLocation` (`@/lib/ai` `CustomLocation {cityId, slug, name}`, slug via `customLocationSlug`) ‚Äî idempotent, **capped at `MAX_CUSTOM_LOCATIONS_PER_CITY` per city** (oldest of that city dropped) so a long chronicle can't grow the atlas/save unbounded. It is called from `applyWorldStateChanges` after an applied location change AND from `deserializeSession` on load (so a legacy/older save parked at an off-map venue pins immediately). `fifthResolved` is the shared builder so `resolveLocation` and `mapAtlasFor` compute the merged district list once (no double pass per render). `isFifthEpoch` is imported from `@/lib/lore` (the single shared epoch rule, also used by `place-graph.ts`). `customToDistrict`/`locationLabel`/`resolveCityId` are supporting pure helpers; `isValidCustomLocationsShape` is the strict persistence guard (`session.ts` rejects malformed, preserves valid on the `...gs` spread). Imports the lore gazetteer (game‚Üílore is allowed, like `session.ts`).
- `tracked-npcs.ts` ‚Äî Generalized tracked-NPC roster (issue #101). The durable cast the player is bound to ‚Äî allies who follow (a party) and hostiles who follow (pursuers) ‚Äî distinct from the transient, scene-scoped `gameState.npcsPresent`. The chosen NAME is NOT stored here; it mirrors `GameState.characterName` (single source of truth), and `renameTrueSelf` swaps it while pushing the old name onto `formerNames` (reputation carries over untouched ‚Äî it is keyed to the stable `characterId`, never the name). `TrueSelfProfile` holds gender/pronouns/appearance/epithet/age/marks, `demeanor` trait tags (allure etc.), a player-authored `notes` log (NEVER fed to the AI), and `formerNames`. `ProfileState.recognition` is the **inverse of identity exposure**: a drastic change snapshots the present NPCs into a "must re-recognise" pool (`openRecognitionGap`) and the narrator is told they see a stranger; `recordRecognition` is a deliberate reveal, `pierceRecognition` a per-turn deduction beat (deterministic under injected randomness) ‚Äî both close the gap when the pool empties. `isDrasticChange` is the hybrid pre-tick (gender changed OR demeanor changed OR `flawlessCapable`). `applyProfileChange` is the one edit path (rename + edits + optional gap) used by the sheet editor, the in-turn confirm, and the ritual scene; `proposalToChange` maps an AI `proposedSelfChange` (validated by `validateSelfChangeProposal`, the single drop-not-throw point). `profilePromptContext` (ground-truth self facts) and `recognitionPromptContext` thread into the prompt via `GenerateOptions.profileContext`/`recognitionContext` (the game loop suppresses recognition while a persona is worn). `resolveProfileState` is the lazy `?? createProfileState()` boundary; `GameSession.profileState` is optional + strictly validated (`isValidProfileStateShape`), exactly like `identityState`.
- `transformation-rites.ts` ‚Äî Pure data: canon-weighty advancement moments that remake who a Beyonder is. `TRANSFORMATION_RITES` (seed: Demoness Seq 7 "Witch" ‚Äî new name + allure + pre-ticked recognition gap); `transformationRiteFor(pathwayId, targetSeq)` returns the optional rite the character-sheet True-Self editor offers (never forces), pre-filling the editor with its suggestions.
- `society.ts` ‚Äî Secret society / Tarot Club system (issue #32). `societyKindForPathway` (Fool founds the Tarot Club proper; Death/Darkness a Nighthawks squad, Sun a church division, Tyrant a pirate crew, others a scholars' circle), founding gated at Seq ‚â§7 (`canFoundSociety`). AI-NPC members carry code names, pathway hints, dispositions (‚àí100..100) and their own slow arcs; `holdGathering` ("above the gray fog", `GATHERING_COOLDOWN_TURNS` apart) yields MECHANICAL consequences ‚Äî intel **memory facts** the narrator/investigation pillar consume, occasional traded items, disposition drift, arc creep ‚Äî deterministic under injected randomness; `resolveMemberArc` settles a fully-advanced arc for trust. **Member prose is never persisted**: a member stores stable indices (`pathwayHintId`, `arcId` ‚Äî `RESOLVED_ARC_ID` for a settled arc) into the append-only `PATHWAY_HINTS`/`MEMBER_ARCS` catalogs, and the UI derives the text via `memberPathwayHint`/`memberArc`, so a copy/grammar edit reaches every existing save instead of being frozen in at recruit time. `migrateSocietyState` (called by `deserializeSession`) converts legacy saves that stored the rendered prose ‚Äî including the pre-fix singular-verb arcs ‚Äî back to ids; `isValidSocietyShape` accepts both the legacy prose and id member shapes. `GameSession.societyState` optional + strictly validated. **`seedSocietyMembership(orgSlug)`** (issue #131) builds a valid `SocietyState` for a start-archetype pre-membership ‚Äî mapping a known org slug (`ORG_MEMBERSHIPS`, e.g. `nighthawks-tingen-team` ‚Üí `nighthawk-squad` "The Tingen Nighthawks", else a neutral scholars' circle) to the kind+name, with no members yet ‚Äî bypassing the Seq-gated `foundSociety` (a fresh Seq-9 Beyonder could never found one, yet a junior Nighthawk plainly belongs to a squad).
- `anchors.ts` ‚Äî Anchors system (issues #35, #25). The tier-gated stabilising resource that opposes a high-Sequence Beyonder's godhood/mythical-form pressure. **Pure + additive ‚Äî imports the sanity and death/failure ladders, never duplicates them.** `Anchor {id, kind: object|place|congregation, name, integrity 0-100}` + `AnchorState` (on `GameSession.anchorState`, optional + strictly validated via `isValidAnchorStateShape`, mirroring `identityState`). `requiresAnchors(seq)` (mandatory at Seq ‚â§ 2 ‚Äî Angels) and `anchorsRelevant(seq)` (Saint tier, Seq ‚â§ 4). `effectiveSupport` (Œ£ integrity √ó `ANCHOR_KIND_WEIGHT`, congregation > place > object ‚Äî "the faith of the mass" is strongest) vs `requiredSupport(seq)` (0 above Saint, constant for Saints, rising for Angels) ‚Üí `supportShortfall`. **Integration helpers the game loop calls (game-loop.tsx is NOT modified here):** `anchorPressure(state, seq)` returns a non-positive per-turn sanity delta (fold into the same accumulation as `sanityDelta(event)`, then `applySanityImpact`); `anchorHighRisk(state, seq)` maps an under-anchored Beyonder onto the failure engine's `highRisk` flag (thread into `evaluateFailure`/`evaluateLossOfControl`) ‚Äî this is canon anchor theft tipping an advancement into loss of control. `consecrateAnchor`/`damageAnchor`/`repairAnchor`/`loseAnchor` (pure, integrity clamped 0-100; `damageAnchor` keeps a zeroed anchor re-growable, `isLost` reports it). `DEMIGOD_TRAITS` = per-tier (Seq 4-1) Saint/Angel/King-of-Angels lifespan + mythical-form + prayer-response flavour data for the character sheet (pure data, not fed to the AI). Design doc: `docs/anchors-design.md`.

- `tutorial.ts` ‚Äî Tutorial prologue data (issue #14): `TUTORIAL_SCENES`, four hand-authored scenes that teach Beyonders/secrecy, the potion risk, the acting method, and pathway choice **through fiction** with one plain lesson line each; `TUTORIAL_REQUIRED_TOPICS` is the coverage contract tests enforce. ## Game Loop Phases

```
idle ‚Üí situation ‚Üí choices ‚Üí resolution ‚Üí consequences ‚Üí situation (loop)
any active phase ‚Üí error (on failure)
error ‚Üí situation (retry) | idle (abandon)
```

## Key Design Decisions

- **Pure state machine.** `transition()` is a pure function ‚Äî no side effects, no async. The UI orchestrates AI calls and storage.
- **Separation of concerns.** Phase transitions and world state mutations are independent operations composed by the caller.
- **AI-mutable field allowlist.** Only `location`, `activeQuests`, `npcsPresent` can be changed via AI `worldStateChanges`. Pathway, sequence, and max sanity are rules-engine-only. `location` is additionally **reachability-gated** (issue #101): the narrator may move within a city but a cross-city teleport is refused unless flagged with an `involuntaryCause` (or the player turns the gate off). It is also **access-gated** (issue #130): a destination on an access-gated continent (the Forsaken Land) is refused unless the character `meetsAccessGate` (sequence + held capability flag) ‚Äî a hard canon rule that holds even with an `involuntaryCause` and even when the movement gate is off. `GameState.accessFlags` (the held capabilities) is NOT AI-mutable.
- **Session serialization is pure.** `serializeSession`/`deserializeSession` operate on strings. localStorage access is in the UI layer.

## Conventions

- All functions are pure and synchronous (except AI calls in UI).
- Test patterns follow `src/lib/rules/rules.test.ts`.
- Session shape validation is strict ‚Äî unknown JSON is rejected rather than coerced.
