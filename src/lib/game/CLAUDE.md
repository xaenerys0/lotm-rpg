# Game Loop Engine

Pure-function game loop state machine and world state management. The engine handles phase transitions, world state mutations, and session persistence. AI calls and UI rendering are handled by components in `src/components/game/`.

## Structure

- `types.ts` — Game loop type definitions: phases, actions, session, pillars, transition maps.
- `constants.ts` — Shared localStorage key constants (prefixed namespacing to prevent collisions). The single source of truth every persistence boundary uses for sessions, prologue draft, provider config, preferences, journal, legacies, echoes, hints, the cloud-sync ledger, and transient combat/usage state (`SESSION_KEY_PREFIX`, `ACTIVE_SESSION_KEY`, `PROLOGUE_DRAFT_KEY`, `PREFERENCES_KEY`, `JOURNAL_KEY_PREFIX`, `LEGACIES_KEY`, `ECHOES_KEY`, `COMBAT_KEY_PREFIX`, `USAGE_KEY_PREFIX`, …).
- `math.ts` — Shared numeric helpers, kept dependency-free and pure so any engine module can reuse without re-declaring: `clamp(value, min, max)` constrains to the inclusive range. Used by combat, sanity, digestion, anchors.
- `state-machine.ts` — Pure reducer: `transition(session, action)` produces the next session state. Phase transitions validated against `VALID_TRANSITIONS`. `InvalidTransitionError` for illegal transitions. **`PRESENT_NEXT_CHOICES`** is the normal-turn loop edge `resolution → choices` (the action-assumption fix): a player turn resolves the action AND presents the resolution's OWN choices as the next decision in one beat — no separate forward-looking `generateSituation` scene that could assume actions the player never chose. The React layer (`resolveChoice`) commits the turn via `applyAndCommitTurn` (every `applyResolution` + bookkeeping side-effect) BEFORE this transition; the action CARRIES the resolution `result` and the transition sets `phase:"choices"`, `currentChoices` from it, **`lastResolution: action.result`** (so the choices screen renders the outcome + consequences recap above the choices), `currentNarrative: null` (that slot is reserved for a fresh scene, which the recap supersedes), increments `turnCount`, and clears `selectedChoiceId`/`activePillar`/`pendingPlayerAction`/`pendingTurnKind`. `lastResolution` must be carried on the action because `SELECT_CHOICE` clears it first (when the player picks the next action, so the resolution-loading screen shows no stale recap) — if `PRESENT_NEXT_CHOICES` did not re-set it, the recap would be blank. **`ENGINE_RESOLUTION`** is the action for an engine-decided turn (advancement / apotheosis / combat): the engine has already committed the outcome onto `gameState`, so it carries a narration-only `ValidatedAIResponse` straight from `choices` → `consequences` (the one extra edge in `VALID_TRANSITIONS.choices`), setting `currentNarrative`/`lastResolution`/`pendingPlayerAction` and clearing `selectedChoiceId`. The `handleContinue` "Continue" — now reached ONLY by an engine turn — commits it via the same `applyAndCommitTurn` and resumes with a fresh scene (`APPLY_CONSEQUENCES` → `situation`), so the climb lands in the transcript AND a memory turn record (the narrator stays aware), with no side-channel banner. `pendingPlayerAction` (the player-action text for that turn) is cleared on `APPLY_CONSEQUENCES`.
- `world-state.ts` — World state mutation: `applyResolution()` orchestrates sanity, location, inventory, **funds**, digestion, acting-method discovery, and memory updates from a validated AI response. **Hybrid sanity (issue #95):** per-turn sanity = engine-scored `sanityDeltaForTags(response.sanityEventTags, pre-mutation sequence)` + the AI's residual `sanityImpact` clamped to ±`SANITY_RESIDUAL_CAP` (5); both fold into one `applySanityImpact`. It also takes a trailing `actingMethodState` param (caller passes `resolveActingMethodState(session.actingMethodState)`), runs `evaluateActingDiscovery` from the acting eval + `actingMethodTaught` flag, appends a diegetic `actingMethodDiscoveryFact` on discovery, and returns the extended shape `{ gameState, memory, digestionDelta, sanity: {tagDelta, residual, total}, actingMethodState, discovery: {discoveredThisTurn, trigger} }`. Only AI-mutable fields (`location`, `activeQuests`, `npcsPresent`) are applied from `worldStateChanges`; pathway/sequence/maxSanity are rules-engine-only. **Movement gate (issue #101):** `applyWorldStateChanges(state, changes, opts)` now returns `{state, facts}` (`opts` = `{epoch, gateEnabled, trackedNpcState, turnNumber}`) and routes every `location` change through `gateLocationChange` — a cross-city teleport without a valid `involuntaryCause` (`isInvoluntaryMoveCause` over `INVOLUNTARY_MOVE_CAUSES` = abduction/forced-passage/capability-gated-teleport) is **refused** (location unchanged, an in-world redirect fact emitted — narration, never a throw, like `discoveredItemLeadFact`/`freeTextRejection`); a within-city/provisional move passes; an against-the-will move passes with an "against your will" fact. The gate anchors the origin city to the engine-tracked `GameState.currentCity` (passed as `fromCity`) when the `location` STRING is a bare district that names no city — so a teleport out of a district-level location ("a medical facility" → "Bayam") is still caught instead of slipping through as `provisional`. **Access-gated continent (world build-out, issue #130):** `gateLocationChange` also takes the `sequenceLevel`/`accessFlags` slice and, when the destination resolves to a city on a non-central, access-gated continent (the Forsaken Land), **refuses** the move unless `meetsAccessGate` passes — checked FIRST, so it bites **even on `involuntaryCause:"capability-gated-teleport"` AND even when the movement-realism gate is off** (a hard canon boundary, not a pacing preference); the narrator can never route an unqualified character into the sealed continent. **Crossing chokepoint (issue #132):** after the access gate, `gateLocationChange` also refuses a cross-CONTINENT move whose endpoints don't include the crossing city `CROSSING_CITY` ("giant-kings-court") — so even a QUALIFIED (flagged) character can't be teleported STRAIGHT from the mainland to the City of Silver; every crossing routes through the Giant King's Court dream threshold. Like the access gate it bites regardless of an involuntary cause and regardless of the realism toggle, and (origin anchored to the tracked `fromCity` for a bare district) only fires when both endpoints resolve to known cities — an ordinary within-city/unresolved move is untouched. `applyWorldStateChanges` threads `next.sequenceLevel`/`next.accessFlags` into the call. **In-play flag grant (world build-out 3, issue #132):** after the `location` is applied, `applyWorldStateChanges` checks `reachedDreamWorldGate(next.location)` and, when true and the flag isn't held, `grantAccessFlag`s `dream-world-passage` + emits an `event` fact — the path by which a NON-origin character earns entry to the Forsaken Land by reaching the Dream-World shadow of Giant King's Court through the story (the physical forsaken city stays access-gated; only the dream threshold grants). `gateLocationChange` returns a `crossCity` flag: on a **cross-city** relocation the scene cast is reset to the roster's followers (`reassertFollowersAt([], …)` — the origin's incidental NPCs are left behind), while a **within-city/provisional** nudge PRESERVES the existing cast and only re-asserts the followers into it (`reassertFollowersAt(next.npcsPresent, …)`) — a local move no longer wipes mid-scene NPCs. An explicit `npcsPresent` write likewise re-asserts the followers, so the AI can never drop a follower (engine-truth-over-AI-string, like `advanceActiveHunts`). **Doppelganger suppression (issue #92):** the `npcsPresent` write also runs `stripSelfFromNpcs(names, next.canonCharacterId)` (from `canon-takeover.ts`) FIRST, so a canon-takeover player's own name/aliases can never be added as a separate present NPC (a no-op for an ordinary save); because `buildRetrievalQuery` reads `npcsPresent`, the RAG-query self-reference is suppressed for free. An applied location change also updates the engine-tracked `GameState.currentCity` when the new location names a known city, and PRESERVES it for a bare district string (so the map's per-city atlas stays oriented — `cityForLocation` is the resolver, `travelTo`/`createDefaultGameState` seed it too), and then calls `registerCustomLocation` (`location.ts`, Backlund location sync) so a narrator-named venue that matches no known district is filed under the resolved city as a `GameState.customLocations` entry the map can render and pin. `applyResolution` gained trailing `epoch`, `trackedNpcState`, `movementGateEnabled` params (precedent: the trailing `actingMethodState` param) and folds the gate's returned facts into memory via `addSessionFact`. Deliberate map travel (`travelTo`) is the trusted player path and is NOT run through the gate. Money found/lost in the fiction arrives via the dedicated `AIResponse.fundsDiscovered` channel (NOT `worldStateChanges`, like `itemsDiscovered`): clamped to ±`FUNDS_DISCOVERED_CAP` per turn and applied through `adjustFunds` (floors the wallet at 0) — the wallet updates so found money is spendable on ingredients, bounded so a fortune can't be conjured in one turn. `applyDigestion()` advances/reverses potion digestion from the AI's acting evaluation. A normal turn also calls `tickInjuries()` (from `combat.ts`) so combat injuries heal over play. **AI-discovered items are partitioned (issue #90):** `partitionDiscoveredItems()` splits `itemsDiscovered` into the items the player may carry (`mundane` loot + the narrator-grantable `uniqueness` artifact) and the advancement-critical reagents (`main-ingredient`/`supplementary-ingredient`/`potion-formula`, via `isReagentCategory`) the AI may NOT mint — only the carried kinds reach inventory, and each blocked reagent becomes a `quest-progress` story lead via `discoveredItemLeadFact()` instead (so narration cannot bypass the purchase/hunt gates in `potion-preparation.ts`). The turn record is built from a response carrying only the carried items, so the memory fact extractor / recent-summary never report a blocked reagent as "discovered". The consequences panel (`game-loop.tsx`) renders the same partition. **Sealed Artifacts (Sealed Artifacts subsystem):** the `"sealed-artifact"` category is ALSO blocked from AI discovery (church-catalogued relics are not "found" in a scene) — `isDiscoveryBlockedCategory` adds it alongside the reagents, and `discoveredItemLeadFact` gives it its own "catalogued and locked away by the churches — earned through the story, never simply found" lead. The trusted acquisition path is `grantSealedArtifact(state, code)` (looks up the `@/lib/lore` catalogue, mints the carried `Item` via `mintArtifactItem`, appends it — a no-op for an unknown code or one already carried, since artifacts are singular), the engine counterpart to combat spoils (`Enemy.loot`). **Custody-aware acquisition (Sealed Artifacts subsystem):** `discoveredItemLeadFact`'s artifact lead now branches on custody (`custodyForArtifactItem`) — a church relic reads "locked away by the churches" (stolen or earned by affiliation), an individually-owned one names its holder, an unowned one reads findable; `isChurchAffiliated(session)` (a `church-division`/`nighthawk-squad` society) gates `grantChurchArtifact(session, code)` — the "working with the church" path that entrusts a church-custodied relic (refuses `unknown-code`/`not-church-custody`/`not-affiliated`/`already-held`). **`narrationOnly(response)`** strips an `AIResponse` down to its narrative (and optional journal flag) for an engine-decided turn (advancement / apotheosis) routed through `ENGINE_RESOLUTION` — the engine already applied every mechanical effect, so feeding the full response back through `applyResolution` would double-apply sanity/digestion/items/world changes; narration-only keeps the turn-loop integration (transcript + memory) without re-deciding anything.
- `canon-takeover.ts` — **Canon-character takeover seeding + doppelganger suppression (issue #92).** `createCanonCharacterSession(preset, initialMemory, startScenario?, options?, prologueRecapOverride?)` turns a matched `CanonCharacterPreset` (`@/lib/lore/canon-characters`) into a fully seeded `GameSession` by COMPOSING `createDefaultGameState`/`createSession` — then overriding the canon-specific fields the generic builders hardcode: `sequenceLevel` → the canon START sequence, `location` → the canon `startLocation` (unless a compatible explicit `startScenario` is passed), `digestion` re-seeded via `createDigestionState` for that start sequence, `currentCity` re-resolved, `canonCharacterId` set, **`canonPersonality` seeded from `preset.personalityTraits`** (so the narrator biases presented choices to the figure's canon disposition), and the timeline gate seeded at the character's introduction chapter (`canonPosition`). The optional **`prologueRecapOverride`** (the canon-faithful GUIDED prologue's recap) REPLACES the static `preset.openingRecap` in the durable `prologueRecap` slot when present + non-blank; absent/blank falls back to `openingRecap` (the manual-path takeover). DURABILITY: the figure's identity/relationships ride the never-trimmed `characterBackground` (+ the distinct `prologueRecap` "how you arrive" bridge), NOT trimmable session facts — `earlySalienceFacts` are seeded as ordinary (ageable) facts for turn-0 immediacy only. `stripSelfFromNpcs(npcs, canonCharacterId)` removes the player's OWN canon identity (display name + aliases, via `getCanonCharacter` + `normalizeCanonName`) from a narrator `npcsPresent` list — a no-op when not a canon takeover — and is reused by `world-state.ts`'s `npcsPresent` case so the narrator can never spawn the player as a separate present NPC. Pure.
- `hunt.ts` — Hunt quests (advancement/combat streamline). Hunting a Beyonder Characteristic for the next potion no longer jumps straight to combat: it starts a tracked, multi-turn QUEST to find the creature. `HuntState {targetItemName, targetSeq, turnsRemaining, totalTurns}` lives as a **list** on `GameSession.hunts` (several hunts can run at once), optional + strictly validated (`isValidHuntStateShape`/`isValidHuntsShape`) and preserved on the deserialize `...s` spread, exactly like `actingMethodState`. `huntTrackingTurns(targetSeq)` scales the search length with the same `acquisitionDepthFactor` as the potion economy (deeper rungs take longer) off `HUNT_BASE_TRACKING_TURNS`. `startHunt(session, itemName)` validates like `deliverHuntedItem` (missing, huntable prerequisite for the target rung; honors `MAIN_INGREDIENT_HUNT_ONLY_AT_OR_BELOW`) and refuses `not-required`/`already-owned`/`not-huntable`/`already-hunting`; on success it records the hunt, seeds a `quest-progress` fact, and adds the AI-visible `huntQuestLabel` to `activeQuests`. `advanceHunt` decrements one hunt (floored at 0 = ready); `advanceActiveHunts(session)` advances all each turn and idempotently re-adds dropped quest labels (engine truth is `hunts`, never the AI-mutable string). `isHuntReady` gates the engage step; `clearHunt(session, itemName)` removes a hunt + its label on a victorious kill or abandonment; `findHunt` looks one up. Pure; the React layer runs the combat encounter (existing `CombatEncounter.huntTarget` plumbing → `deliverHuntedItem` on victory) and persists.
- `acquired-powers.ts` — **Acquired powers — copying / stealing / borrowing another Beyonder's power.** Abilities are otherwise DERIVED from `(pathwayId, sequenceLevel)` (`apotheosis.ts` `sequenceAbilities`), so this is the ONLY place a power taken from someone ELSE is recorded on the character. `AcquiredPower {id, name, description, sourceName?, method, permanence, turnsRemaining?, acquiredAtTurn}` lives as a **list** on `GameSession.acquiredPowers` (optional + strictly validated via `isValidAcquiredPowerShape`/`isValidAcquiredPowersShape`, preserved on the deserialize `...s` spread, NEVER AI-mutable — exactly the `hunts` pattern). **Capability-gated:** `powerAcquisitionCapabilities(session)` derives the means available now — the pathway rungs the character has climbed to (corpus-grounded `PATHWAY_ACQUISITIONS`, every permanence VERIFIED against the wiki `<Pathway>/Abilities` pages, NOT memory: Error Seq 6 **Prometheus** "Steal a target's Beyonder power and use it for about ten minutes" = **TEMPORARY** (the victim recovers it over hours-days); Error Seq 4 **Parasite** "always Steal something when emerging" = permanent; Door Seq 6 **Scribe** "Record" a witnessed ACTIVE power "for a single use" = temporary/single-use; White Tower Seq 6 **Imitation** = temporary; Hanged Man Seq 5 **Shepherd** "Grazing" swallows a soul and wields its powers, but the soul "will gradually wear away … can't be permanently kept" = TEMPORARY-long, so only Parasite is truly permanent) PLUS any carried copy/steal Sealed Artifact (`ARTIFACT_ACQUISITIONS`, a matcher list keyed EITHER by church catalogue number — **2-081 Ring of Mimicry**, **2-105 Blood Vessel Thief**, recovered via `@/lib/lore` `sealedArtifactNumberFromItemName` — OR, for the famous relics with NO canon church number, by an item-name substring: **Leymano's Travels** (the Tarot Club's rental notebook, "copies Beyonder abilities it has seen … reused only once" — the Door Scribe's Record) and **Creeping Hunger** (the glove that "uses the stored souls' Beyonder abilities" — the Hanged Man Shepherd's Grazing, likewise temporary-long since the souls wear away/are consumed)) — so the "take a power" affordance never appears for a Beyonder who could not canonically do it (`canAcquirePower`/`capabilityForMethod`). The corpus mechanic lives at the PATHWAY level; the artifacts are in-fiction embodiments (so any pathway's carrier gets the means). **ADD** `acquirePower(session, input, idFactory?, now?)` (validates the method is a held capability — `no-capability`/`invalid-method`/`missing-name`/`at-capacity` (`MAX_ACQUIRED_POWERS` 24), defaults permanence/duration from the capability but allows overrides, seeds a memory fact); **UPDATE** `updateAcquiredPower(session, id, changes)` (rename/re-describe/re-attribute, refresh a copy's `turnsRemaining`, or convert permanent↔temporary — a Prometheus binding a copy for good); **REMOVE** `releasePower(session, id)` (drops it + seeds a fact). `tickAcquiredPowers(session)` fades every temporary power by one turn each play turn and releases the expired ones (a memory fact per expiry) — called in `game-loop.tsx`'s `handleContinue` alongside `advanceActiveHunts`/`advanceFormulaPursuit`. `acquiredPowerAbilityLabels(session)` surfaces the powers (a temporary one notes its turns left, a permanent one is flagged "stolen") into combat's `availableAbilities` AND the narrator prompt (`buildAICallParams`), so a stolen/copied power is actually USABLE. `permanentPowers`/`temporaryPowers`/`findAcquiredPower` are display helpers. The UI is the character-sheet `AcquiredPowersSection`. Pure + deterministic (injected `idFactory`/`now`). **Crafted-artifact capability (Sealed Artifacts subsystem):** a player-CRAFTED artifact whose derived effect carries the `acquired-power` hook (`customArtifactForItem`) also grants the generic `imitation` means, so artifice can produce a power-copy relic just as the canon Ring of Mimicry does.
- `custom-artifacts.ts` — **Player-crafted (artifice) Sealed Artifact registry.** A crafted artifact has no `@/lib/lore` catalogue entry, so its rich descriptor (`CustomArtifact {code, name, grade, sourcePathwayId, sourceSequence, effects, drawback, flavor?, ownerName?, craftedAtTurn}`) lives on the session keyed by a SYNTHETIC catalogue code `C<grade>-<NNN>`; `CustomArtifactState {artifacts, nextOrdinal}` on `GameSession.customArtifactState` (optional + strictly validated via `isValidCustomArtifactStateShape`, preserved on the deserialize `...s` spread, NEVER AI-mutable — the `acquiredPowers` pattern; no DB migration). The lightweight carried `Item` (`mintCustomArtifactItem`) embeds the code in its name exactly like `mintArtifactItem`, so every name-based helper keeps working — `gradeForArtifactItem` reads the grade straight from the `C<d>-` prefix (no lore→game import). `resolveCustomArtifactState`/`findCustomArtifact`/`customArtifactForItem` (code-keyed lookup), `registerCustomArtifact` (allocate code, bump `nextOrdinal`, `at-capacity`/`missing-name` refusals, `MAX_CUSTOM_ARTIFACTS` 24), `forgetCustomArtifact` (on trade/loss), `craftedArtifactFact`. Pure.
- `artifact-effects.ts` — **Artifact effects — making the powers REAL.** An artifact's effects come from the source pathway+sequence (authored in `@/lib/lore` `ARTIFACT_EFFECTS` for catalogue, DERIVED for crafted). `deriveArtifactEffects(pathwayId, sequence, grade)` distils the rung's canon abilities (`getSequence`) into `ArtifactEffect`s, each routed to an `EffectHook` by `effectHookForAbility` (the `EFFECT_HOOK_KEYWORDS` table → identity/sanity/combat/acquired-power/funds/access, else the combat-role fallback, else `narrator`); `deriveArtifactDrawback(pathwayId, grade)` is the grade-scaled loss-of-control line. `carriedArtifactEffects(session)` resolves every carried artifact's effects (catalogue + crafted); `carriedEffectsByHook` + the per-hook surfacers (`artifactIdentityCapabilities`/`artifactSanityActions`/`artifactCombatEffects`/`artifactPowerAcquisitions`) feed the engine subsystems; **`artifactNarratorContext(session)`** is the binding `## Artifact Effects` prompt block (threaded via `GenerateOptions.artifactEffectsContext` → `prompts.ts`) that makes EVERY effect real in the fiction — the only enforcement for an effect with no engine subsystem. Pure.
- `artifice.ts` — **Crafting a Sealed Artifact (corpus: a Paragon "Artisan" Seq 6+ fuses a Beyonder Characteristic).** `craftCapability(session)` → `{canSelfCraft (Paragon `PARAGON_PATHWAY_ID`19 at Seq ≤`ARTISAN_SEQUENCE` 6), canCommission (always)}`; `gradeForCharacteristicSequence(seq)` is the user's band (9-7→3, 6-5→2, 4-3→1, 2-1→0, Seq 0 → `null` = no god-tier artifact); `parseCharacteristicItem(item)` reads pathway+sequence from a `main-ingredient` "Sequence N <Pathway> Beyonder Characteristic" (a creature-material returns `undefined`). `craftArtifact(session, input, now?)` gates (capability → characteristic → grade → name → funds → capacity), CONSUMES the Characteristic (`removeItemsByName`) + debits the fee (`craftFee`/`COMMISSION_FEE_BY_GRADE`/`SELF_CRAFT_MATERIALS_COST_BY_GRADE`), derives effects+drawback, registers via `custom-artifacts`, mints the carried item, seeds a memory fact — every `CraftOutcome` returned, nothing consumed on refusal. Pure + deterministic.
- `combat.ts` — Combat engine (issue #10). Pure, deterministic functions for hybrid combat: a mechanical **preparation** phase, an AI-narrated **exchange** of mid-fight decision points, and a **resolution**. `scorePreparation()` rates preparation 0-1 (capped per dimension; `AMBUSH_PREP_CAP` for ambushes); `computeSequenceGap()`, `computePathwayMatchup()` (Beyonder-vs-mundane edge, God-Almighty-vs-Eternal-Darkness opposition, kindred groups, same-pathway penalty), and `computeInjuryPenalty()` feed `computeBaseAdvantage()` together with a single injected `randomFactor`. `createEncounter()` → `applyPreparation()` → `chooseOption()` (×2-3) → `resolveEncounter()` walk the `CombatEncounter` mini state machine; `generateDecisionPoints()` builds pathway-flavoured options whose context-aware modifiers reward knowing your enemy and holding an edge. `resolveOutcome(input)` maps the final advantage + tactics + **encounter framing** to the full **outcome spectrum**; `computeConsequences()` produces injuries, sanity drain (via `sanityDelta`), items gained/lost, and dropped characteristics. **Encounter framing & opponent selection (issue #187, Phase 1):** `selectOpponents(state, opts)` surfaces an EXPLICIT, framed target list — present NPCs (an `ally` reframed via `coerceAllyFraming` to a reconcilable `mind-controlled`/etc., NEVER a plain foe — engine truth over the AI string), roster pursuers, and curated `bestiary` picks — retiring the silent `npcsPresent[0]` auto-pick; `deriveEncounter`/`deriveEncounterEnemy` are the single-pick fallbacks (present NPC/pursuer → generic framed fallback). `EncounterContext`/`EncounterFraming` ride on the encounter; `makeEncounterContext`/`isReconcilableFraming`/`framingLabel`/`describeFraming`/`bestiaryFoeToEnemy` are the helpers. **Outcome spectrum (Phase 4):** beyond win/lose, the player's chosen RESOLUTION LINE (tagged on `DecisionOption.resolutionLine`) yields `subdued`/`freed`/`talked-down`/`captured`/`spared` — a committed snap-free line frees a mind-controlled ally on a win (the React layer rosters them via `joinRoster`), a sustained de-escalation talks a reconcilable foe down; `isWinningOutcome` flags the win-shapes; loot/characteristic drop only on a lethal `victory`. **Ability kits (Phase 2):** `createEncounter` captures `combatKitFor` onto `availableKit`; `generateDecisionPoints` offers the costed kit options + the resolution lines + legible `effectTag`s. **Hunt quarry (issue #187 follow-ups):** `deriveHuntQuarry(state, opts)` derives an ingredient-hunt foe that matches WHAT is being hunted — the actual main ingredient (`opts.huntItem`, threaded by `startCombat` from `potionPreparationPlan`; `opts.pathwayId`/`opts.targetSeq` default to `state.pathwayId`/`state.sequenceLevel - 1`). The next potion's main ingredient is EITHER: (a) a **Beyonder Characteristic** → a Beyonder of the player's OWN pathway at the TARGET rung — the same tier as the potion (canon: a Seq-4 Bizarro Sorcerer potion takes a Bizarro Sorcerer Characteristic, fixed at the data source by `applyCanonMainIngredient` in `pathways.ts`), preferring a CATALOGUED carrier (`curatedCarrier`→`bestiaryForPathwaySequence`, e.g. the Devil Dog only for an ACTUAL Abyss hunter at a rung its band covers) else the synthesized bearer (`synthesizeBeyonderQuarry` — `hostile-beyonder`, canon role via `getSequence(...).name`, that rung's abilities); or (b) a **creature material** (the corpus-grounded canon material, e.g. "True Root of a Mist Treant", "Fire Salamander gland") → that CREATURE (`synthesizeBeastQuarry` — a `beast`, `isBeyonder:false`, never a pathway Beyonder, so it can't claim to drop a Characteristic it never had), the creature recovered by `creatureFromMaterial` (take the "{part} of a/an/the {creature}" tail / the head of a "{creature}'s {part}" possessive / strip a case-insensitive curated body-part suffix / keep a whole-entity name intact — handles the canon material forms). With no `huntItem`, falls back to the pathway+`targetSeq` bearer. It NEVER reads `npcsPresent`/the roster, so engaging a hunt can NEVER attack a present friend AND never fields an unrelated foe or a mismatched-pathway Characteristic (the reported bugs). Corpus-grounded: a Characteristic is of the SAME pathway, and a creature/material of that pathway+Sequence is the canonical source ("a Sequence 2 Beyonder characteristic of the same pathway"; "the main material for the Sequence 5 Mentor of Disorder, from a curly-haired baboon"). **Control meter (Phase 3):** a transient `controlStrain` (0..1) + `controlTier` (`steady`→`frayed`→`slipping`→`spiral`) seeded from sanity (`seedControlStrain`) and pushed/eased by each choice's `strainDelta` (risky aggression / relics / straining abilities push; defensive/evasive/de-escalation ease — recovery). `chooseOption` accumulates strain + the enemy-stance shift; at `slipping` the next point is FORCED RECKLESS (`restampUpcomingPoint` drops the evasive + snap-free/talk-down hatches); `resolveEncounter` folds a tier-scaled `CONTROL_PENALTY` into the final advantage and, at `spiral`, sets `result.lossOfControl` via the death engine's `evaluateFailure({cause:"loss-of-control", highRisk})` (a setback at low Sequence, transformation/death at high — the React layer routes it through `concludeChronicle`/`applySetback`, mirroring advancement). `highRisk` is passed from `game-loop` (`anchorHighRisk` || low sanity). `controlTierFor`/`controlTierLabel`/`describeControlTier` are the pure helpers. **Reactive enemy stances (Phase 3b):** `stanceForPoint(runningAdvantage)` → `pressing`/`guarded`/`reeling`/`desperate`, stamped on each `DecisionPoint.enemyStance` (re-stamped reactively in `chooseOption`); a bounded `STANCE_MODIFIER` shifts which tactic pays off; `stanceLabel`/`describeStance` format it. **Clarity surfaces (Phase 5):** `threatAssessment(encounter)` (danger tier + INTEL-GATED odds band) and `afterActionReport(encounter, result)` (itemized outcome / sanity-with-cause / injuries / items gained / spent / **KEPT-not-consumed** / control outcome / characteristic / world ripple) are pure formatters the UI renders. **Narration (Phase 6):** `combatNarrationContext` also appends the pursued resolution line + a fraying/slipping/spiralling control tier so narration matches mechanics. `applyCombatResult()` writes the result to `GameState` (sanity, inventory, `injuries`); `tickInjuries()` heals injuries each turn. **Design principle: preparation helps but never decides** — clever mid-fight choices can overturn a poor start and a single random factor keeps no fight certain. - `place-graph.ts` — Reachability graph (issue #101). Pure, derived on call. - `death.ts` — Death & failure engine (issue #12). The rules engine — never the AI — owns the consequence: `evaluateFailure({cause, sequenceLevel, highRisk?, catastrophic?})` resolves a failure (`loss-of-control` via the sanity ladder; `combat-defeat`/`ritual-failure` are setbacks unless `catastrophic`, which escalates on the same sequence-scaled ladder) into `setback | permadeath`. `applySetback(state, random?, turn?, epoch?)` = wake at 25% sanity, lose ~half the inventory, displaced to an unknown corner of the SAME locale, reputation memory fact. The displacement venue is **epoch-aware** (`DISPLACEMENT_PLACES_BY_EPOCH`, one pool per era from `EPOCHS` in `@/lib/lore`): the Fifth Epoch anchors to the character's current city (`"<place> in <city>"` — its cross-city travel is gated, so a breakdown must not teleport across cities), while earlier epochs draw a self-contained, era-appropriate place (no city suffix — those eras have no city model), so a setback never strands a character in another era's setting (the bug: an Iron-Age "abandoned warehouse" appearing in the Age of Chaos). The epoch is normalized once via `isFifthEpoch`/`DEFAULT_EPOCH_ID` (absent/unknown → Fifth), so the pool and the anchoring can't disagree; the game loop's `handleSetback` passes `session.gameState.epoch`. Permadeath: `buildLegacy(session, severity)` (transformation → loose **monster**; death → **legend** for Seq ≤6 else **cautionary tale**, with epitaph), `endSession()` sets `GameSession.ended` (the session is preserved as a historical record — inventory, memory, journal stay readable), `legaciesToFacts()` seeds a new same-timeline character's memory with the fallen (the world remembers), `fallbackDescentScene()` is the deterministic descent narration when no AI is configured, and strict legacy (de)serialization under `LEGACIES_KEY`. **Full restart** = wiping the legacy list (canonical baseline restored).
- `advancement.ts` — Sequence advancement (Seq 9 → 2), the rungs below the apotheosis endgame. The rules engine — never the AI — owns the climb. `advancementRequirements(session)`/`canAdvance` build a checklist from the **target** sequence's canon data (`getSequence`): a fully digested potion (Acting Method), the target potion's `prerequisiteItems` carried, anchors at the Saint tier (`anchorsRelevant`), and a sanity floor (`ADVANCEMENT_SANITY_RATIO`) — these are hard gates (`meetsRequirements`/`canAdvance`). **The Advancement Ritual from Sequence 5 onward is no longer a hard gate (issue #209):** the rite is a SOFT, fidelity-weighted factor — the "ritual" row is ADVISORY (`met: true`, `forthcoming: ritualFidelity < 1`, a "skipping is perilous" label), so the climb is always attemptable; how faithfully the rite was lived out (`ritualFidelity`, `ritual.ts`) does the work instead. `advancementSuccessChance` is high at low Sequences and falls per rung, **never reaching certainty**, steadied by anchor surplus + sanity, and **dragged down by an unfaithful rite** — it subtracts `RITUAL_INFIDELITY_PENALTY × (1 − ritualFidelity)` (a fully-forgone rite ≈ −0.5, canon "plummets to a dangerous point"; the floor keeps it possible but dire). `advancementHighRisk` is also true when the rite is rushed/forgone (`ritualFidelity < RITUAL_HIGH_RISK_FIDELITY`), escalating a loss of control by one step. `attemptAdvancement(session, random?)` returns `advanced` (engine decrements `sequenceLevel`, consumes the ingredients via `removeItemsByName`, re-seeds digestion via `createDigestionState`, drains sanity through `applySanityImpact`, clears the consumed `ritualState`, seeds memory facts — incl. a fidelity-aware note: the rite was "fully formed" / "only half-formed when the potion was drunk" / "forgone") or `lost-control` (an `evaluateFailure({cause:"loss-of-control"})` verdict — `highRisk` from an under-anchored high form / frayed mind / unfaithful rite — routed through the same setback/permadeath machinery as any failure). Canon source: novel + wiki (rituals begin at Seq 5). Pure + deterministic; the React layer narrates via the `"advancement"` instruction and **routes the committed climb through the normal turn loop via `ENGINE_RESOLUTION`** (narration-only response → `consequences` → memory turn record), so the narrator is aware of the advancement on the next turn rather than it appearing in a side-channel banner.
- `ritual.ts` — Advancement rituals — begun once, then matured naturally over play (issue #209; supersedes the per-step click-counting of issue #99 Part C). From Sequence 5 the Advancement Ritual is canon: it survives the surge of the new Beyonder characteristic at the moment of drinking. The rite is NOT a click-through step list and NOT a fixed number of turns. The player BEGINS it once (`beginRitual` — "Perform the rite"), which the React layer pairs with an IMMEDIATE narrated turn so it opens in the current scene; from then on it MATURES on its own each turn of play — there is no set length to wait out, and the player drinks (via the always-available climb panel) whenever they judge it ready. `RitualState {pathwayId, targetSeq, fidelity}` is a single optional sub-state on `GameSession.ritualState` (strictly validated via `isValidRitualStateShape`, preserved on the deserialize `...s` spread, never seeded — **no DB migration**; a legacy turn-based shape fails validation and drops, resetting the rite). `ritualStepsFor(session, targetSeq): RitualStep[]` returns the corpus-tagged `steps` (materials + lived conditions) when present, else the hand-authored fallback `requirements` treated as conditions — used to surface the canon materials/conditions, **never a clickable list**. **Circumstance** is the heart of it: `ritualCircumstanceFidelity(session)` scores how favourable the CURRENT scene is (1.0 private/safe; lowered by witnesses `RITUAL_WITNESS_PENALTY`, wounds `RITUAL_WOUNDED_PENALTY`, active pursuers `RITUAL_HUNTED_PENALTY`, stalling at 0 when several stack) — the "isolated vs crowd vs battle" signal the player weighs. `beginRitual` (idempotent per target, resets on re-target) seeds a first slice of progress scaled by that circumstance, adds an AI-visible quest label + an `event` fact. `advanceRitual(session)` is the per-turn tick (wired into `game-loop.tsx`'s commit chain beside `advanceFormulaPursuit`): each turn it closes part of the gap to a fully-formed rite (`accrueFidelity` — `(1−fidelity)×RITUAL_PROGRESS_RATE×circumstance`, so a private moment matures it fast and danger stalls it), keeps the label in sync, and no-ops once matured past `RITUAL_FIDELITY_CAP` (no churn). `ritualFidelity(session, targetSeq): number` in [0,1] (1 when no rite needed; 0 when not begun for this target = forgone; else the accrued `fidelity`) is the factor `advancement.ts` folds into the climb odds — a rite ≥ `RITUAL_FIDELITY_CAP` reads as fully formed. `ritualInProgress` backs the UI; `clearRitual` (consumed on a successful climb / a survived setback, drops the label). Pure. The React layer's `RitualPerformancePanel` (`game-loop.tsx`, between `PotionPreparationPanel` and `AdvancementPanel`) shows the pathway's canon description + materials/conditions + a single "Perform the rite" trigger (and a `role="progressbar"` fidelity meter once maturing) with a circumstance hint; the climb attempt (`AdvancementPanel`) is ALWAYS offered — drinking early at low fidelity is the player's call. `advancement.ts`'s ritual requirement is ADVISORY (never hard-blocks), and `attemptAdvancement` consumes the `ritualState` (clears it) on a successful climb so the next rung's rite starts fresh.
- `potion-preparation.ts` — Next-potion acquisition (issue #84), the missing step that makes `advancement.ts`'s strict ingredient gate reachable in normal play (inventory is not AI-mutable, so nothing else grants the next potion's reagents). Pure + engine-owned like advancement. `potionPreparationPlan(session)` builds the target potion's checklist (each `prerequisiteItem` with `owned`, allowed `methods`, and purchase `cost`); `acquisitionMethodsFor(item, targetSeq)` makes the Beyonder Characteristic (main ingredient) **hunt-or-buy at shallow rungs but hunt-only from `MAIN_INGREDIENT_HUNT_ONLY_AT_OR_BELOW` (6) onward**, while the formula and supplementary reagents are purchase-only; `acquisitionCost`/`huntLootReward` scale with `acquisitionDepthFactor` (deeper potions cost far more — "higher Sequences require extra effort"). `purchasePotionItem(session, name)` spends funds (`marketplace` economy) and delivers the item (refusing `not-required`/`already-owned`/`not-purchasable`/`unaffordable`); `deliverHuntedItem(session, name)` grants a hunted Characteristic plus depth-scaled spoils (the React layer runs the combat encounter and calls this only on victory). - `formula-pursuit.ts` — Formula pursuit (issue #171, "The Climb"), the "seek it through the story" route to the next potion's recipe — the alternative to trading for it (`purchasePotionItem`). The formula is the canon gate to advancement (a Beyonder gathers reagents only once they hold the recipe — enforced in `potion-preparation.ts`). This is a tracked, multi-turn QUEST to chase a recipe down, mirroring `hunt.ts` but for the `potion-formula` item: `FormulaPursuitState {targetItemName, targetSeq, turnsRemaining, totalTurns}` is a SINGLE optional sub-state on `GameSession.formulaPursuit` (only one formula is ever in play — the next potion's — so it mirrors `ritualState`, not the list-shaped `hunts`), strictly validated (`isValidFormulaPursuitShape`) and preserved on the deserialize `...s` spread, never seeded. `formulaPursuitTurns(targetSeq)` scales the search with the same `acquisitionDepthFactor` as the rest of the potion economy (deeper recipes are more jealously guarded, so longer) off `FORMULA_PURSUIT_BASE_TURNS`. `beginFormulaPursuit(session)` validates the next potion has a formula prerequisite not yet owned and that none is already under way (`not-required`/`already-owned`/`already-pursuing`); on success it records the pursuit, seeds a `quest-progress` fact, and adds the AI-visible `formulaPursuitQuestLabel` to `activeQuests`. `advanceFormulaPursuit(session)` decrements one turn each play turn (floored at 0 = ready) and idempotently re-adds a dropped quest label (engine truth is `formulaPursuit`, never the AI string). `isFormulaPursuitReady` gates the secure step; `secureFormulaThroughStory(session)` claims the formula at the end of a completed pursuit — earned through the story, so (unlike the trade route) it costs NO funds — delivering the item, seeding a memory fact, and clearing the pursuit (`secured`/`no-pursuit`/`not-ready`/`already-owned`); `clearFormulaPursuit` removes it + its label on success or abandonment. Pure; the React layer narrates the "recipe secured" beat through `ENGINE_RESOLUTION`, exactly like the trade route.

- `anchors.ts` — Anchors system (issues #35, #25). The tier-gated stabilising resource that opposes a high-Sequence Beyonder's godhood/mythical-form pressure. **Pure + additive — imports the sanity and death/failure ladders, never duplicates them.** `Anchor {id, kind: object|place|congregation, name, integrity 0-100}` + `AnchorState` (on `GameSession.anchorState`, optional + strictly validated via `isValidAnchorStateShape`, mirroring `identityState`). `requiresAnchors(seq)` (mandatory at Seq ≤ 2 — Angels) and `anchorsRelevant(seq)` (Saint tier, Seq ≤ 4). `effectiveSupport` (Σ integrity × `ANCHOR_KIND_WEIGHT`, congregation > place > object — "the faith of the mass" is strongest) vs `requiredSupport(seq)` (0 above Saint, constant for Saints, rising for Angels) → `supportShortfall`. **Integration helpers the game loop calls (game-loop.tsx is NOT modified here):** `anchorPressure(state, seq)` returns a non-positive per-turn sanity delta (fold into the same accumulation as `sanityDelta(event)`, then `applySanityImpact`); `anchorHighRisk(state, seq)` maps an under-anchored Beyonder onto the failure engine's `highRisk` flag (thread into `evaluateFailure`/`evaluateLossOfControl`) — this is canon anchor theft tipping an advancement into loss of control. `consecrateAnchor`/`damageAnchor`/`repairAnchor`/`loseAnchor` (pure, integrity clamped 0-100; `damageAnchor` keeps a zeroed anchor re-growable, `isLost` reports it). `canConsecrateCongregation(followingSize)` gates the strongest anchor — a congregation of believers — on the character actually having a following (society members + allies), so a Saint leans on personal marks (object/place) while believers require a gathered flock (canon). **Player-facing UI:** the character-sheet `AnchorsSection` (`character-sheet.tsx`) surfaces `consecrateAnchor`/`repairAnchor`/`loseAnchor` + the `effectiveSupport`-vs-`requiredSupport` meter — the acquisition path that makes the Saint-tier anchors gate in `advancementRequirements` actually satisfiable. ## Game Loop Phases

```
idle → situation → choices → resolution → choices (normal-turn loop)
                                       ↘ consequences → situation (engine-turn loop)
any active phase → error (on failure)
error → situation (retry) | idle (abandon)
```

A **normal player turn** loops `resolution → choices` via **`PRESENT_NEXT_CHOICES`**: the
resolution narrates the outcome AND presents the next decision in ONE beat, so the loop no
longer regenerates a separate forward-looking scene that could assume actions the player
never chose (the action-assumption fix — feedback that the AI "did another turn" past the
player's input). `situation`/`generateSituation` is now reached only for the **opening scene**
(turn 0) and to **resume after an engine-decided turn**. `consequences` is reached **only** by
an engine-decided turn (advancement / apotheosis / combat via `ENGINE_RESOLUTION`).

## Key Design Decisions

- **Pure state machine.** `transition()` is a pure function — no side effects, no async. The UI orchestrates AI calls and storage.
- **Separation of concerns.** Phase transitions and world state mutations are independent operations composed by the caller.
- **AI-mutable field allowlist.** Only `location`, `activeQuests`, `npcsPresent` can be changed via AI `worldStateChanges`. Pathway, sequence, and max sanity are rules-engine-only. `location` is additionally **reachability-gated** (issue #101): the narrator may move within a city but a cross-city teleport is refused unless flagged with an `involuntaryCause` (or the player turns the gate off). It is also **access-gated** (issue #130): a destination on an access-gated continent (the Forsaken Land) is refused unless the character `meetsAccessGate` (sequence + held capability flag) — a hard canon rule that holds even with an `involuntaryCause` and even when the movement gate is off. `GameState.accessFlags` (the held capabilities) is NOT AI-mutable.
- **Session serialization is pure.** `serializeSession`/`deserializeSession` operate on strings. localStorage access is in the UI layer.

## Conventions

- All functions are pure and synchronous (except AI calls in UI).
- Test patterns follow `src/lib/rules/rules.test.ts`.
- Session shape validation is strict — unknown JSON is rejected rather than coerced.
