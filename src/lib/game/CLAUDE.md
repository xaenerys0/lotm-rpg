# Game Loop Engine

Pure-function game loop state machine and world state management. The engine handles phase transitions, world state mutations, and session persistence. AI calls and UI rendering are handled by components in `src/components/game/`.

## Structure

- `types.ts` — Game loop type definitions: phases, actions, session, pillars, transition maps.
- `state-machine.ts` — Pure reducer: `transition(session, action)` produces the next session state. Phase transitions validated against `VALID_TRANSITIONS`. `InvalidTransitionError` for illegal transitions. **`ENGINE_RESOLUTION`** is the action for an engine-decided turn (advancement / apotheosis): the engine has already committed the outcome onto `gameState`, so it carries a narration-only `ValidatedAIResponse` straight from `choices` → `consequences` (the one extra edge in `VALID_TRANSITIONS.choices`), setting `currentNarrative`/`lastResolution`/`pendingPlayerAction` and clearing `selectedChoiceId`. The normal `handleContinue` then commits it like any turn — so the climb lands in the transcript AND a memory turn record (the narrator stays aware), with no side-channel banner. `pendingPlayerAction` (the player-action text for that turn) is cleared on `APPLY_CONSEQUENCES`.
- `world-state.ts` — World state mutation: `applyResolution()` orchestrates sanity, location, inventory, **funds**, digestion, acting-method discovery, and memory updates from a validated AI response. **Hybrid sanity (issue #95):** per-turn sanity = engine-scored `sanityDeltaForTags(response.sanityEventTags, pre-mutation sequence)` + the AI's residual `sanityImpact` clamped to ±`SANITY_RESIDUAL_CAP` (5); both fold into one `applySanityImpact`. It also takes a trailing `actingMethodState` param (caller passes `resolveActingMethodState(session.actingMethodState)`), runs `evaluateActingDiscovery` from the acting eval + `actingMethodTaught` flag, appends a diegetic `actingMethodDiscoveryFact` on discovery, and returns the extended shape `{ gameState, memory, digestionDelta, sanity: {tagDelta, residual, total}, actingMethodState, discovery: {discoveredThisTurn, trigger} }`. Only AI-mutable fields (`location`, `activeQuests`, `npcsPresent`) are applied from `worldStateChanges`; pathway/sequence/maxSanity are rules-engine-only. Money found/lost in the fiction arrives via the dedicated `AIResponse.fundsDiscovered` channel (NOT `worldStateChanges`, like `itemsDiscovered`): clamped to ±`FUNDS_DISCOVERED_CAP` per turn and applied through `adjustFunds` (floors the wallet at 0) — the wallet updates so found money is spendable on ingredients, bounded so a fortune can't be conjured in one turn. `applyDigestion()` advances/reverses potion digestion from the AI's acting evaluation. A normal turn also calls `tickInjuries()` (from `combat.ts`) so combat injuries heal over play. **AI-discovered items are partitioned (issue #90):** `partitionDiscoveredItems()` splits `itemsDiscovered` into the items the player may carry (`mundane` loot + the narrator-grantable `uniqueness` artifact) and the advancement-critical reagents (`main-ingredient`/`supplementary-ingredient`/`potion-formula`, via `isReagentCategory`) the AI may NOT mint — only the carried kinds reach inventory, and each blocked reagent becomes a `quest-progress` story lead via `discoveredItemLeadFact()` instead (so narration cannot bypass the purchase/hunt gates in `potion-preparation.ts`). The turn record is built from a response carrying only the carried items, so the memory fact extractor / recent-summary never report a blocked reagent as "discovered". The consequences panel (`game-loop.tsx`) renders the same partition. **`narrationOnly(response)`** strips an `AIResponse` down to its narrative (and optional journal flag) for an engine-decided turn (advancement / apotheosis) routed through `ENGINE_RESOLUTION` — the engine already applied every mechanical effect, so feeding the full response back through `applyResolution` would double-apply sanity/digestion/items/world changes; narration-only keeps the turn-loop integration (transcript + memory) without re-deciding anything.
- `hunt.ts` — Hunt quests (advancement/combat streamline). Hunting a Beyonder Characteristic for the next potion no longer jumps straight to combat: it starts a tracked, multi-turn QUEST to find the creature. `HuntState {targetItemName, targetSeq, turnsRemaining, totalTurns}` lives as a **list** on `GameSession.hunts` (several hunts can run at once), optional + strictly validated (`isValidHuntStateShape`/`isValidHuntsShape`) and preserved on the deserialize `...s` spread, exactly like `actingMethodState`. `huntTrackingTurns(targetSeq)` scales the search length with the same `acquisitionDepthFactor` as the potion economy (deeper rungs take longer) off `HUNT_BASE_TRACKING_TURNS`. `startHunt(session, itemName)` validates like `deliverHuntedItem` (missing, huntable prerequisite for the target rung; honors `MAIN_INGREDIENT_HUNT_ONLY_AT_OR_BELOW`) and refuses `not-required`/`already-owned`/`not-huntable`/`already-hunting`; on success it records the hunt, seeds a `quest-progress` fact, and adds the AI-visible `huntQuestLabel` to `activeQuests`. `advanceHunt` decrements one hunt (floored at 0 = ready); `advanceActiveHunts(session)` advances all each turn and idempotently re-adds dropped quest labels (engine truth is `hunts`, never the AI-mutable string). `isHuntReady` gates the engage step; `clearHunt(session, itemName)` removes a hunt + its label on a victorious kill or abandonment; `findHunt` looks one up. Pure; the React layer runs the combat encounter (existing `CombatEncounter.huntTarget` plumbing → `deliverHuntedItem` on victory) and persists.
- `combat.ts` — Combat engine (issue #10). Pure, deterministic functions for hybrid combat: a mechanical **preparation** phase, an AI-narrated **exchange** of mid-fight decision points, and a **resolution**. `scorePreparation()` rates preparation 0-1 (capped per dimension; `AMBUSH_PREP_CAP` for ambushes); `computeSequenceGap()`, `computePathwayMatchup()` (Beyonder-vs-mundane edge, God-Almighty-vs-Eternal-Darkness opposition, kindred groups, same-pathway penalty), and `computeInjuryPenalty()` feed `computeBaseAdvantage()` together with a single injected `randomFactor`. `createEncounter()` → `applyPreparation()` → `chooseOption()` (×2-3) → `resolveEncounter()` walk the `CombatEncounter` mini state machine; `generateDecisionPoints()` builds pathway-flavoured options whose context-aware modifiers reward knowing your enemy and holding an edge. `resolveOutcome()` maps the final advantage + tactics to `victory`/`defeat`/`escape`/`stalemate`; `computeConsequences()` produces injuries, sanity drain (via `sanityDelta`), items gained/lost, and dropped characteristics. `applyCombatResult()` writes the result to `GameState` (sanity, inventory, `injuries`); `tickInjuries()` heals injuries each turn. **Design principle: preparation helps but never decides** — clever mid-fight choices can overturn a poor start and a single random factor keeps no fight certain. The AI layer only narrates (instruction `"combat"`). **Dynamic mid-fight options:** `createEncounter` captures the player's `availableAbilities` (from `sequenceAbilities`) and `availableArtifacts` (inventory minus reagents) onto the encounter, and `generateDecisionPoints` appends, beyond the templated trio, up to `MAX_DYNAMIC_ABILITY_OPTIONS` named ability options and `MAX_DYNAMIC_ARTIFACT_OPTIONS_PER_POINT` carried-artifact options per point — selected deterministically by point index (no new randomness, so a serialized fight regenerates identical points). A chosen dynamic artifact carries its `artifactItem`; `computeConsequences` adds those to `itemsLost` distinctly from the sealed-prep `consumesArtifact` slice (no double-count). **`enemyIntel(enemy, intel, playerSequence)`** is the pure, intel-gated opponent presenter (`none` → name/description only; `partial` → coarse strength + pathway; `thorough` → exact sequence + `knownAbilities`); the combat UI formats it.
- `inventory.ts` — Shared inventory helpers. `removeItemsByName(inventory, items)` is the single pure "drop these items by name" operation reused by combat (`applyCombatResult`), advancement (consuming the next potion's ingredients), and any other subsystem that removes carried items — one place so name-vs-id matching stays consistent. `hasItem(inventory, name)` is the name-only carried check; `hasItemMatching(inventory, item)` matches **both category and name** — the advancement, potion-preparation, and apotheosis gates use it (issue #90) so a `mundane` item that merely shares a reagent's name can neither satisfy a prerequisite nor soft-lock its acquisition, staying consistent with the canonical category+name gate in `@/lib/rules` (`laws.ts`). `REAGENT_CATEGORIES`/`isReagentCategory(category)` is the single source of truth for "is this an advancement reagent?" — `world-state` blocks exactly these from AI item discovery and `marketplace` lists exactly these, so the engine-block and the market-sellable set can never drift.
- `digestion.ts` — Acting Method digestion engine (issue #8). Pure functions mapping the AI's acting `alignment` score (0-1) to 0-100 digestion progress for the current potion: `computeProgressDelta()`, `applyDigestionProgress()`, `createDigestionState()`, `isDigestionComplete()`, and deterministic in-world `digestionFeedback()`. In-character acting (alignment ≥ 0.5) advances digestion; contradictory acting (< 0.35) reverses it; non-contradictory acting always yields at least `MIN_PROGRESS_PER_SESSION` (anti-stagnation guardrail). **Pace slowed in issue #95** (`MAX_PROGRESS_PER_EVAL` 12→10, `MIN_PROGRESS_PER_SESSION` 2→1, flat across Sequences). `digestionFeedback(role, state, delta, knowsMethod)` takes a trailing **`knowsMethod`** flag (no default — the compiler flags every caller): pre-discovery it returns vague, cryptic prose that names no mechanic/number and does NOT branch on the progress band (one line per direction, so magnitude never leaks); post-discovery it returns the clearer progress-tiered prose.
- `acting-method.ts` — Acting-method discovery (issue #95). The Acting Method is LOTM _secret knowledge_ a Beyonder must earn through play; `ActingMethodState {knowsMethod, alignedStreak}` (on `GameSession.actingMethodState`, optional + strictly validated via `isValidActingMethodStateShape`, lazily resolved via `resolveActingMethodState`, mirroring `profileState` — NOT seeded by `createSession`). `evaluateActingDiscovery({state, alignment?, taughtFlag})` short-circuits once known, then discovers it via **taught** (an AI-flagged NPC teaching / lore reading — immediate) or **repetition** (the streak grows on aligned acting ≥ `ACTING_ALIGNED_THRESHOLD` 0.5, resets on a lapse, and crossing `ACTING_DISCOVERY_STREAK` 6 flips it). Pre-discovery the digestion meter/number stays hidden and feedback is vague; the tutorial does NOT grant it and a neglect scare is NOT a trigger. Pure.
- `session.ts` — Session lifecycle: `createSession()`, `createDefaultGameState()` (seeds a fresh `digestion` state; accepts an optional `prologueRecap` pinned onto `GameState` for the prologue → story bridge, and an optional `startScenario` from `@/lib/lore/start-scenarios` for **varied story openings** — sets `GameState.location` to the scenario's (randomly varied) starting place and `GameState.openingBeat` to its first-turn beat; both fall back to the epoch default when absent, so legacy callers/tests are unaffected), serialization/deserialization with shape validation (`isValidDigestionShape()`; digestion is seeded on deserialize for legacy sessions saved without it; the durable `memory.runningSummary` is optional but rejected when present and not a string). Pure functions — no localStorage dependency. Accepts optional `initialMemory` to pre-populate memory (used for prologue facts). Anchor state (issues #35, #25): `GameSession.anchorState` is optional and strictly validated (`isValidAnchorStateShape`), exactly like `identityState` — absent on saves that never consecrated an anchor, rejected when malformed. True-self profile (character-info storage): `GameSession.profileState` is optional and strictly validated (`isValidProfileStateShape`), same pattern — lazily seeded for legacy saves via `resolveProfileState`, preserved on the deserialize `...s` spread. Acting-method discovery (issue #95): `GameSession.actingMethodState` is optional and strictly validated (`isValidActingMethodStateShape`), same pattern — lazily resolved via `resolveActingMethodState`, preserved on the `...s` spread, never seeded by `createSession`. Hunt quests: `GameSession.hunts` (a list — multiple concurrent hunts) is optional and strictly validated (`isValidHuntsShape` from `hunt.ts`), same pattern — preserved on the `...s` spread, absent on saves with no active hunts. RAG fields (issue #63): `GameSession.canonPosition` (the player's shared-timeline position feeding the retrieval timeline gate; starts at `DEFAULT_CANON_POSITION`, advanced only forward via `advanceCanonPosition()` — monotonic) and `GameSession.embeddingModelId` (the embedding model **locked at character creation** — validated against `APPROVED_EMBEDDING_MODELS`, no mid-save switching; deserialize rejects unapproved locks and seeds defaults for legacy saves).
- `lore-retrieval.ts` — The pure half of wiring RAG retrieval into the game loop (issues #63/#64): `buildRetrievalQuery(session)` composes the per-turn query from the current location, the most recent narrative beat (or the opening beat on turn 0), active quests, and present NPCs (capped at `RETRIEVAL_QUERY_CHAR_CAP`); `toRetrievedLoreChunks(chunks)` narrows retrieval rows to the `RetrievedLoreChunk` slice the prompt assembler consumes. The network glue (embedder + Supabase RPC, best-effort) lives in `components/game/lore-retrieval-client.ts`, outside the coverage mandate.
- `travel.ts` — Travel mechanics (issue #23). Pure module: `CITIES` (Tingen + Backlund, Trier, Bayam, plus the varied-start regions Pritz Harbor, Enmat Harbor, and Feysac — each with id/name/kingdom/public blurb; the id is the lowercase first word so it matches the lore `city` field and `cityNarrationDirective`), a symmetric `travelDays` table, `getCity`, `cityIdFromLocation` (resolve a `location` string by leading word), `canTravelTo(state, cityId)` (known destination, not the current city; an unknown current location may set out for any city), and `travelTo(state, cityId, turnNumber?)` returning a NEW `GameState` (location set to the city name, `npcsPresent` cleared) plus a memory-fact payload — or `null` when not permitted. Location is on the AI-mutable allowlist; this is the deliberate _player_ travel path. The React layer (`map-panel.tsx`) folds the fact in via `addSessionFact` and persists.
- `sanity.ts` — Sanity/madness engine (issue #9). Re-exports the tier classifier from `@/lib/ai` (the single source of truth, since `GameState` and the prompt assembly live there) and builds the game-facing concerns on top: `SANITY_EFFECTS` (per-tier UI-effect profiles — CSS class, distortion intensity, and which narrative degradations are active) consumed by the React layer via `sanityEffects()`; `sanityDelta(event)` mapping a `SanityEvent` (ability-use, horror-encounter scaled to sequence gap, advancement, outer-deity, acting-success, rest, human-connection, routine, acting-neglect) to a deterministic sanity delta (engine-owned, not AI); and loss-of-control evaluation at zero sanity — `isLossOfControl(state)` and `evaluateLossOfControl(ctx)` returning `setback | transformation | fatal` (higher sequences fail more catastrophically; `highRisk` escalates one step). Ready to integrate with the death/failure system (issue #12). The caller applies deltas via `applySanityImpact` (world-state.ts), which clamps to `[0, maxSanity]`. **Hybrid sanity tags (issue #95):** `SanityEventTag` (`rest`|`human-connection`|`routine`|`ability-use`|`horror-encounter`), `SANITY_EVENT_TAGS`, `HORROR_TAG_GAP` (1), and the pure `sanityDeltaForTags(tags, playerSequence)` that sums `sanityDelta()` per tag — `ability-use` scaled to the player's Sequence, `horror-encounter` modeled as one rung above the player. This is the engine-owned magnitude behind the AI's per-turn `sanityEventTags`.
- `death.ts` — Death & failure engine (issue #12). The rules engine — never the AI — owns the consequence: `evaluateFailure({cause, sequenceLevel, highRisk?, catastrophic?})` resolves a failure (`loss-of-control` via the sanity ladder; `combat-defeat`/`ritual-failure` are setbacks unless `catastrophic`, which escalates on the same sequence-scaled ladder) into `setback | permadeath`. `applySetback(state, random?, turn?)` = wake at 25% sanity, lose ~half the inventory, displaced to an unknown corner of the city, reputation memory fact. Permadeath: `buildLegacy(session, severity)` (transformation → loose **monster**; death → **legend** for Seq ≤6 else **cautionary tale**, with epitaph), `endSession()` sets `GameSession.ended` (the session is preserved as a historical record — inventory, memory, journal stay readable), `legaciesToFacts()` seeds a new same-timeline character's memory with the fallen (the world remembers), `fallbackDescentScene()` is the deterministic descent narration when no AI is configured, and strict legacy (de)serialization under `LEGACIES_KEY`. **Full restart** = wiping the legacy list (canonical baseline restored).
- `advancement.ts` — Sequence advancement (Seq 9 → 2), the rungs below the apotheosis endgame. The rules engine — never the AI — owns the climb. `advancementRequirements(session)`/`canAdvance` build a checklist from the **target** sequence's canon data (`getSequence`): a fully digested potion (Acting Method), the target potion's `prerequisiteItems` carried, a **mandatory Advancement Ritual from Sequence 5 onward** (a hard gate — the pathway must define one; it is performed and narrated during the attempt), anchors at the Saint tier (`anchorsRelevant`), and a sanity floor (`ADVANCEMENT_SANITY_RATIO`). All requirements are hard gates (`meetsRequirements`/`canAdvance`). `advancementSuccessChance` is high at low Sequences and falls per rung, **never reaching certainty** (a permanent loss-of-control chance), steadied by anchor surplus + sanity. `attemptAdvancement(session, random?)` returns `advanced` (engine decrements `sequenceLevel`, consumes the ingredients via `removeItemsByName`, re-seeds digestion via `createDigestionState`, drains sanity through `applySanityImpact`, seeds memory facts) or `lost-control` (an `evaluateFailure({cause:"loss-of-control"})` verdict — `highRisk` from an under-anchored high form / frayed mind — routed through the same setback/permadeath machinery as any failure). Canon source: novel + wiki (rituals begin at Seq 5). Pure + deterministic; the React layer narrates via the `"advancement"` instruction and **routes the committed climb through the normal turn loop via `ENGINE_RESOLUTION`** (narration-only response → `consequences` → memory turn record), so the narrator is aware of the advancement on the next turn rather than it appearing in a side-channel banner.
- `potion-preparation.ts` — Next-potion acquisition (issue #84), the missing step that makes `advancement.ts`'s strict ingredient gate reachable in normal play (inventory is not AI-mutable, so nothing else grants the next potion's reagents). Pure + engine-owned like advancement. `potionPreparationPlan(session)` builds the target potion's checklist (each `prerequisiteItem` with `owned`, allowed `methods`, and purchase `cost`); `acquisitionMethodsFor(item, targetSeq)` makes the Beyonder Characteristic (main ingredient) **hunt-or-buy at shallow rungs but hunt-only from `MAIN_INGREDIENT_HUNT_ONLY_AT_OR_BELOW` (6) onward**, while the formula and supplementary reagents are purchase-only; `acquisitionCost`/`huntLootReward` scale with `acquisitionDepthFactor` (deeper potions cost far more — "higher Sequences require extra effort"). `purchasePotionItem(session, name)` spends funds (`marketplace` economy) and delivers the item (refusing `not-required`/`already-owned`/`not-purchasable`/`unaffordable`); `deliverHuntedItem(session, name)` grants a hunted Characteristic plus depth-scaled spoils (the React layer runs the combat encounter and calls this only on victory). Both append memory facts; the caller journals + persists. The combination is what lets the engine actually commit a sequence change (status bar updates) instead of the AI narrating an advancement the engine never made.
- `apotheosis.ts` — Sequence 0 endgame (issue #30). The True God ascent, owned by the rules engine. `apotheosisRequirements(session)`/`canAttemptApotheosis` gate the ritual on session-carried data only (Seq 1, fully-digested potion, the pathway `uniquenessItemFor` item in inventory — its **own `uniqueness` category** (not lumped with mundane loot or the reagents), the singular endgame artifact the narrator may grant through play but that is never sold by any channel; matched via `hasItemMatching`, anchor `effectiveSupport ≥ requiredSupport(0)`, ≥ half sanity); `apotheosisSuccessChance` is strong-but-capped (anchor surplus + sanity), `attemptApotheosis(session, random?)` returns `ascended` (engine writes Seq 0 — consumes the Uniqueness, resets digestion, seeds memory facts + the one-time `ABOVE_SEQUENCE_TEASE`) or `unmade` (a catastrophic `ritual-failure` verdict from the death engine — Seq 1 = fatal — routed through the same permadeath machinery). Post-apotheosis: `drawPetition` (a `PETITION_CHANCE` per-turn worshipper memory fact), `trueGodName`/`TRUE_GOD_NAMES`, and `TRUE_GOD_ABILITIES`/`TRUE_GOD_ACTING` (the narrator framing that stands in for a rules `Sequence`, which Seq 0 has none of). `sequenceAbilities(pathwayId, level)` is the one True-God-aware ability/acting derivation the prompt and combat UI share: at Seq 0 it returns the True-God framing; below it the **abilities are cumulative** (`getCumulativeAbilities` from `@/lib/rules`, earlier-rung powers suffixed `(enhanced)` so the narrator knows they are retained and strengthened), while acting requirements stay scoped to the current rung. Pure + deterministic under injected randomness; the godhood narration shift lives in `buildDemigodDirective` (ai/prompts.ts).
- `echoes.ts` — Cross-epoch timeline echoes (issue #31). When a chronicle ends, `mintArtifact(session, legacy)` mints a `TimelineArtifact` from the fallen character's signature possession (potion formula > first carried item > their journal as fallback), persisted under `ECHOES_KEY` next to the legacy list. A fresh character may discover echoes: `discoverableArtifacts` enforces the **paradox guard** (`originEpoch <= currentEpoch` — the past never inherits from the future; echoes are resonances, not unique objects, so multiple timelines may each find one), `pickStartingEcho` (≤1, ~50% chance, deterministic under injected randomness) seeds the inventory via `artifactToItem`, and `echoFacts` gives the narrator up to three foreshadow facts plus a concrete carried-item fact. Strict `serializeArtifacts`/`deserializeArtifacts`. Full timeline restart wipes `ECHOES_KEY` together with `LEGACIES_KEY`.
- `journal.ts` — Story journal engine (issue #11). Pure functions only: `validateJournalFlag` (the single validation point for the AI's optional `AIResponse.journalEntry` flag — malformed flags are dropped, never thrown), `deriveJournalEntries` (the AI flag plus deterministic detections from the state delta — advancement, travel, new-NPC encounters — deduped by event type with the AI's summary winning), annotation CRUD (`addAnnotation`/`editAnnotation`/`deleteAnnotation`; **player-only — never fed into prompt assembly**), `filterJournal` (text/type/npc/location/character), `journalToMarkdown` (arc-grouped export with annotations as blockquotes), and strict `serializeJournal`/`deserializeJournal`. Arc = the character's sequence label.
- `journal-sync.ts` — Journal persistence to Supabase (issue #11): `toEntryRow`/`toAnnotationRow` mappers and `syncEntries` (batched, after a turn) / `syncAnnotation` + `deleteAnnotationRemote` (immediate on edit) / `syncJournal`, over a minimal structural `JournalSyncClient`. `deleteSessionEntriesRemote(client, sessionId)` deletes every durable journal row for one character on removal (RLS scopes to the caller; annotations cascade via `entry_id`). Best-effort from the UI — the localStorage journal never blocks on the network. Tables: `journal_entries` / `journal_annotations` (migration `20260612160000`, owner-only RLS).
- `character-admin.ts` — Character management (deletion). `characterDeletionPlan(sessionId, sessionIndex)` is a PURE function returning the localStorage keys to remove (session save, journal, in-progress combat `COMBAT_KEY_PREFIX`, usage estimate `USAGE_KEY_PREFIX`) and the session index with that id stripped; the React layer executes the plan and the best-effort `deleteSessionEntriesRemote` (journal-sync). Cross-timeline `LEGACIES_KEY`/`ECHOES_KEY` are world memory and intentionally NOT removed — the separate full-timeline restart wipes those.
- `preferences.ts` — Player display preferences (issues #9, #13, #95). `GamePreferences` (`sanityMeterVisible` default `false`; `digestionMeterVisible` default `false` — issue #95, mirrors `sanityMeterVisible` but stays inert until the acting method is discovered; `highContrast` default `false` — legacy payloads without these remain valid), `DEFAULT_PREFERENCES`, strict shape validation, and pure `serializePreferences`/`deserializePreferences`/`mergePreferences` (fall back to defaults on missing/invalid data). localStorage access lives in the React layer (`PREFERENCES_KEY`).
- `achievements.ts` — Achievements & timeline divergence (issue #18). Pure evaluation over a `GameSession`: `ACHIEVEMENTS` (11 milestones derived ONLY from data the session carries — turns, sequence, digestion, injuries, identities, legacies in memory facts, funds, ended chronicles), `evaluateAchievements`, `divergenceScore` (an **honest heuristic** 0-100: story length, fallen predecessors remembered, leaving the canon start, identity webs, an ended chronicle — documented in-code, not a literal canon diff), `showcaseStats` (turns, combats won via memory facts, potions = rungs climbed, funds), `legacyCount`.
- `showcase-sync.ts` — Supabase glue for showcases/leaderboards (injectable structural client): `toShowcaseRow`, `publishShowcase` (explicit player action, upsert by user), `fetchLeaderboard` (public rows only — RLS hides private ones — ranked by sequence/achievements/divergence with a pathway filter). Table `player_showcases` (migration `20260613010000`).
- `marketplace.ts` — Item marketplace engine (issue #16). Introduces currency: `GameState.funds` in pence (optional for old saves; `getFunds` seeds `STARTING_FUNDS`), `adjustFunds` (floored at 0) and `canAfford`. `FUNDS_DISCOVERED_CAP` bounds the per-turn money the narrator may grant/take via `AIResponse.fundsDiscovered` (applied in `world-state.ts`). `validateListing` is the anti-exploit gate for **player-to-player** listings (item must actually be carried, its category an advancement reagent via `isReagentCategory` — `mundane` loot must not become an uncapped funds source on the open market, issue #90, and `uniqueness` is never sold; integer price under a sanity cap). `mundane` belongings are instead sold to a non-player vendor via `sellItemToVendor`/`vendorSaleValue` (`VENDOR_SALE_CATEGORIES`) — a local, modest-fixed-price "fence" path ("sellable, but not to users"), kept low so it cannot launder AI loot into deep-Sequence reagents. `removeItemForListing` escrows the item out of the save at listing time (cannot list twice or keep while listed), `addItemToInventory` delivers purchases/returns, `PRICE_GUIDANCE` per category, `filterListings` for browsing (active + unexpired only). Honest framing: the server guarantees marketplace-row legitimacy and atomic purchase; funds/escrow live client-side with the rest of the save, by architecture.
- `marketplace-sync.ts` — Supabase glue (injectable structural client): `fetchActiveListings`, `fetchOwnHistory` (seller+buyer ledger), `createListing`, `delist`, and `purchase` via the `purchase_listing` SECURITY DEFINER RPC (row-locked, self-purchase and stale listings rejected server-side; the ONLY path to 'sold'). Table `market_listings` (migration `20260613000000`).
- `world-messages.ts` — Shared world messages (issue #17). Safety is structural: a message is a TEMPLATE id + slot fills validated against a fixed vocabulary (`composeMessage` throws on anything else — arbitrary text cannot exist). `MESSAGE_TEMPLATES` covers warnings/tips/lore/humor; `selectMessagesForScene` does helpfulness-weighted sampling capped at `MAX_MESSAGES_PER_SCENE` (2) with heavily-downvoted spam buried, deterministic under an injected random source.
- `world-messages-sync.ts` — Supabase glue (injectable structural client): `fetchLocationMessages`, `placeMessage`, `rateMessage` (the `rate_world_message` SECURITY DEFINER RPC — one vote per player, repeats are silent no-ops). Tables `world_messages` / `world_message_votes` (migration `20260612230000`): shared authenticated read, owner insert/delete.
- `identity.ts` — Identity management (issue #22). Pure engine: `identityCapability(pathwayId, seq)` gates three tiers — **flawless** (`FLAWLESS_IDENTITY_THRESHOLDS` — Fool ≤6 "Faceless", Error ≤5 "Dream Stealer"): a persona that reads as a real, separate person (`flawless` flag, no `activeDisguise`, NO exposure accrual in `recordIdentityUse`, `checkExposure` early-returns so mundane NPCs never connect it, `identityPromptContext` drops the "presentation not truth" framing; `isFateProof`/`FATE_PROOF_THRESHOLDS` adds the Dream-Stealer "divination/fate-reading fails" clause via the `fateProof` flag); **full** (`FULL_IDENTITY_THRESHOLDS` — Error ≤8 Swindler line); and a one-disguise **basic** tier for everyone else (`activeDisguise`, frays 2×); `createIdentity` takes a trailing `fateProof` arg used only for flawless personas; `createIdentity`/`discardIdentity`/`switchIdentity` (deliberate, never automatic; `null` = true face); per-identity `reputation` (clamped −100..100) and `knownBy` NPC awareness (`recordIdentityUse` — NPCs learn the active face only, never cross-reference); exposure: risk accrues per use/new witness, and past `EXPOSURE_EVENT_THRESHOLD` a witness who knows two faces may connect them (`checkExposure`, injected randomness) — `applyExposure` links them, dents both reputations with that NPC, resets risk to a simmer. `identityPromptContext` renders the one narrator-facing presentation line (threaded through `GenerateOptions.identityContext` → an `## Active Identity` system layer); `GameSession.identityState` is optional + strictly validated. **Assuming a persona from the play surface** (the game-loop `IdentityActionPanel`): `hasPreparedIdentity(state)` gates whether there is any face to wear; `detectAssumeIdentityIntent(input)` flags a free-text attempt to assume one (an assume-verb — wear/put on/take on/slip into/adopt… — paired with a persona noun in the same breath; a bare "the identity of the thief" is investigation, not assumption, and is deliberately NOT caught) so the loop can steer it to the deliberate `switchIdentity` instead of letting the claim dissolve into narration the engine never commits; `UNPREPARED_IDENTITY_NARRATIVE` / `ASSUME_VIA_PANEL_NARRATIVE` are the in-world steers (narration, never error text — like `freeTextRejection`).
- `profile.ts` — True-self profile (character-info storage). The mutable, D&D-style record of who the character actually IS — distinct from `identity.ts` personas. The chosen NAME is NOT stored here; it mirrors `GameState.characterName` (single source of truth), and `renameTrueSelf` swaps it while pushing the old name onto `formerNames` (reputation carries over untouched — it is keyed to the stable `characterId`, never the name). `TrueSelfProfile` holds gender/pronouns/appearance/epithet/age/marks, `demeanor` trait tags (allure etc.), a player-authored `notes` log (NEVER fed to the AI), and `formerNames`. `ProfileState.recognition` is the **inverse of identity exposure**: a drastic change snapshots the present NPCs into a "must re-recognise" pool (`openRecognitionGap`) and the narrator is told they see a stranger; `recordRecognition` is a deliberate reveal, `pierceRecognition` a per-turn deduction beat (deterministic under injected randomness) — both close the gap when the pool empties. `isDrasticChange` is the hybrid pre-tick (gender changed OR demeanor changed OR `flawlessCapable`). `applyProfileChange` is the one edit path (rename + edits + optional gap) used by the sheet editor, the in-turn confirm, and the ritual scene; `proposalToChange` maps an AI `proposedSelfChange` (validated by `validateSelfChangeProposal`, the single drop-not-throw point). `profilePromptContext` (ground-truth self facts) and `recognitionPromptContext` thread into the prompt via `GenerateOptions.profileContext`/`recognitionContext` (the game loop suppresses recognition while a persona is worn). `resolveProfileState` is the lazy `?? createProfileState()` boundary; `GameSession.profileState` is optional + strictly validated (`isValidProfileStateShape`), exactly like `identityState`.
- `transformation-rites.ts` — Pure data: canon-weighty advancement moments that remake who a Beyonder is. `TRANSFORMATION_RITES` (seed: Demoness Seq 7 "Witch" — new name + allure + pre-ticked recognition gap); `transformationRiteFor(pathwayId, targetSeq)` returns the optional rite the character-sheet True-Self editor offers (never forces), pre-filling the editor with its suggestions.
- `society.ts` — Secret society / Tarot Club system (issue #32). `societyKindForPathway` (Fool founds the Tarot Club proper; Death/Darkness a Nighthawks squad, Sun a church division, Tyrant a pirate crew, others a scholars' circle), founding gated at Seq ≤7 (`canFoundSociety`). AI-NPC members carry code names, pathway hints, dispositions (−100..100) and their own slow arcs; `holdGathering` ("above the gray fog", `GATHERING_COOLDOWN_TURNS` apart) yields MECHANICAL consequences — intel **memory facts** the narrator/investigation pillar consume, occasional traded items, disposition drift, arc creep — deterministic under injected randomness; `resolveMemberArc` settles a fully-advanced arc for trust. **Member prose is never persisted**: a member stores stable indices (`pathwayHintId`, `arcId` — `RESOLVED_ARC_ID` for a settled arc) into the append-only `PATHWAY_HINTS`/`MEMBER_ARCS` catalogs, and the UI derives the text via `memberPathwayHint`/`memberArc`, so a copy/grammar edit reaches every existing save instead of being frozen in at recruit time. `migrateSocietyState` (called by `deserializeSession`) converts legacy saves that stored the rendered prose — including the pre-fix singular-verb arcs — back to ids; `isValidSocietyShape` accepts both the legacy prose and id member shapes. `GameSession.societyState` optional + strictly validated.
- `anchors.ts` — Anchors system (issues #35, #25). The tier-gated stabilising resource that opposes a high-Sequence Beyonder's godhood/mythical-form pressure. **Pure + additive — imports the sanity and death/failure ladders, never duplicates them.** `Anchor {id, kind: object|place|congregation, name, integrity 0-100}` + `AnchorState` (on `GameSession.anchorState`, optional + strictly validated via `isValidAnchorStateShape`, mirroring `identityState`). `requiresAnchors(seq)` (mandatory at Seq ≤ 2 — Angels) and `anchorsRelevant(seq)` (Saint tier, Seq ≤ 4). `effectiveSupport` (Σ integrity × `ANCHOR_KIND_WEIGHT`, congregation > place > object — "the faith of the mass" is strongest) vs `requiredSupport(seq)` (0 above Saint, constant for Saints, rising for Angels) → `supportShortfall`. **Integration helpers the game loop calls (game-loop.tsx is NOT modified here):** `anchorPressure(state, seq)` returns a non-positive per-turn sanity delta (fold into the same accumulation as `sanityDelta(event)`, then `applySanityImpact`); `anchorHighRisk(state, seq)` maps an under-anchored Beyonder onto the failure engine's `highRisk` flag (thread into `evaluateFailure`/`evaluateLossOfControl`) — this is canon anchor theft tipping an advancement into loss of control. `consecrateAnchor`/`damageAnchor`/`repairAnchor`/`loseAnchor` (pure, integrity clamped 0-100; `damageAnchor` keeps a zeroed anchor re-growable, `isLost` reports it). `DEMIGOD_TRAITS` = per-tier (Seq 4-1) Saint/Angel/King-of-Angels lifespan + mythical-form + prayer-response flavour data for the character sheet (pure data, not fed to the AI). Design doc: `docs/anchors-design.md`.
- `input-modes.ts` — Context-dependent input modes (issue #24). `detectInputMode(choices, narrative)` infers the scene's mode from the AI's own choice mix (plurality; narrative keywords only as tiebreaker): exploration/ritual/dialogue/investigation (combat is exogenous — an active encounter takes the surface). Guided modes COMPOSE one free-text action (`composeRitualAction` with `RITUAL_STEPS` + material selection, `composeDialogueAction`, `composeDeduction` over `gatherClues`) and submit through the validated issue-#19 pipeline — one resolution path serves every mode.
- `free-text.ts` — Free-text player actions (issue #19). Pure pipeline: `validateFreeText` (normalize; reject empty/over-280-chars/exploit declarations like "I become Sequence 0" — the hard guard is the engine's existing clamps: AI-mutable allowlist, sanity clamp, rules-engine-only advancement), `freeTextRejection` (in-world narration, never an error message), and `freeTextToChoice` (wraps valid text as the synthetic `FREE_TEXT_CHOICE_ID` choice with a pillar inferred from its verbs, so the unchanged SELECT_CHOICE → resolution machinery — sanity, acting, journal — handles it).
- `tutorial.ts` — Tutorial prologue data (issue #14): `TUTORIAL_SCENES`, four hand-authored scenes that teach Beyonders/secrecy, the potion risk, the acting method, and pathway choice **through fiction** with one plain lesson line each; `TUTORIAL_REQUIRED_TOPICS` is the coverage contract tests enforce. Pure data — skippable, no gameplay state, never fed to the AI.
- `prologue.ts` — Prologue engine: 4 static narrative scenes, pathway affinity scoring, recommendation algorithm, and `createPrologueMemory()` to seed session facts from character creation choices. Also the **live, deterministic decision path** for the AI prologue (issue #53), generic over any number of pathways (never assumes 4): `tallyAffinities(weightMaps)` sums per-pathway affinity weights across the player's picked choices; `rankPathways(tally)` sorts into descending `PathwayScore[]` (ties break by ascending id); `selectTopCandidates(tally, {count=3, min=2})` returns the ranked candidate ids for the finale (top `count`, all ties at the cutoff, with a `min` floor); `dominantAffinity(map)` is the argmax pathway id of a single affinity map. The engine — not the LLM — decides the candidate set; the player picks the final potion. `recommendPathway` is de-biased: with no affinity signal it returns `pathwayId: 0` (none) rather than defaulting to Seer/pathway 1. **`buildPrologueRecap({choices, finaleNarrative?, chosenPotion?})`** composes the compact, durable recap of the AI prologue (the defining choices, the finale encounter, the potion drunk; finale capped at `RECAP_FINALE_CHAR_CAP`) that character creation pins into `GameState.prologueRecap` — the prologue → story bridge, since the story narrator runs on a separate prompt that never sees the prologue scenes. Returns `""` when there is nothing to recap (the manual path). Exported via `index.ts`.

## Game Loop Phases

```
idle → situation → choices → resolution → consequences → situation (loop)
any active phase → error (on failure)
error → situation (retry) | idle (abandon)
```

## Key Design Decisions

- **Pure state machine.** `transition()` is a pure function — no side effects, no async. The UI orchestrates AI calls and storage.
- **Separation of concerns.** Phase transitions and world state mutations are independent operations composed by the caller.
- **AI-mutable field allowlist.** Only `location`, `activeQuests`, `npcsPresent` can be changed via AI `worldStateChanges`. Pathway, sequence, and max sanity are rules-engine-only.
- **Session serialization is pure.** `serializeSession`/`deserializeSession` operate on strings. localStorage access is in the UI layer.

## Conventions

- All functions are pure and synchronous (except AI calls in UI).
- Test patterns follow `src/lib/rules/rules.test.ts`.
- Session shape validation is strict — unknown JSON is rejected rather than coerced.
