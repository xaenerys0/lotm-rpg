# Components

## Organization

- `auth/` — Authentication forms (`login-form.tsx`, `signup-form.tsx`)
- `pwa/` — Progressive Web App helpers (mounted globally in the root layout):
  - `service-worker-registrar.tsx` (`ServiceWorkerRegistrar`) — registers `/sw.js`; renders nothing
  - `install-prompt.tsx` (`InstallPrompt`) — Android install button (via `beforeinstallprompt`) / iOS "Add to Home Screen" hint; self-hides when already installed
- `game/` — Game shell and gameplay components:
  - `game-sidebar.tsx` — Sidebar navigation, sign-out (collapsible on mobile)
  - `mobile-nav.tsx` (`MobileNav`) — Mobile bottom navigation (issue #27): fixed thumb-reach bar (Play/Character/Journal/Map, ≥48px targets, `aria-current`, safe-area inset padding) shown below `md`; the game layout adds `pb-20 md:pb-0` so content never hides behind it. Heavy sanity/fog animation is softened under 640px for low-power GPUs (`globals.css`).
  - `provider-config.tsx` — AI provider BYOK configuration (provider, API key, models). Model dropdowns prefer a live catalog fetched via `listProviderModels()` (cached in localStorage with a TTL, refreshable, auto-fetched after a successful connection test), falling back to the static `PROVIDER_MODELS` map. Completing issue #15: an always-visible **privacy disclosure** (key lives only in browser localStorage, sent only browser→provider, never our servers, nothing key-derived logged) and a **Remove key** button that wipes the key from the form AND the persisted config immediately (rotation/removal at any time). For Ollama Cloud, once a live catalog has been fetched, `findUnservedModels` (`@/lib/ai`) flags any selected routine/premium model missing from it with an inline `role="alert"` warning — pre-empting the mid-game 403 an unserved premium model would otherwise cause. Scoped to Ollama Cloud on purpose: there the live `/models` ids are exactly the chat ids and a missing model genuinely 403s; other providers can list ids in a different shape than the configured/default id (and a missing model 404s), where the dropdown's `withSelected()` already accommodates the mismatch. Providers flagged `needsAccessProbe` in the `PROVIDERS` metadata (currently only Ollama Cloud — a single flag like `needsBaseUrl`/`requiresAuth` rather than scattered id checks) also get a **"Check models"** action and an auto-check after a successful connection test that runs `probeModelAccess` (`@/lib/ai`) on the two selected models — ollama.com lists its whole cloud catalog regardless of plan, so a selectable model can still 403 ("requires a subscription") at chat time. The probe test-calls each model and lists any the account can't run (each with its own reason) in a `role="alert"` with an upgrade link, or shows a count-aware `role="status"` all-clear. A monotonic `probeIdRef` guards against a late-resolving probe overwriting state after the selection/provider changed; the result is cleared (and the id bumped) on any field edit or provider switch, and the auto-check skips when the current selection already has a result so re-testing doesn't re-spend probe calls.
  - `play-dashboard.tsx` — Play page dashboard (new game, continue, character creation, manage). Seeds each new chronicle with permadeath legacy facts (issue #12) and **timeline echoes** (issue #31): artifacts under `ECHOES_KEY` from fallen predecessors may start in the inventory (`pickStartingEcho` → `artifactToItem`, paradox-guarded to the chosen epoch or earlier) with narrator foreshadow facts via `echoFacts`. A **Manage characters** view (reached from the Character Summary header) lists every saved character with a two-step confirm **Delete** that runs `characterDeletionPlan` (wipes the local save, journal, in-progress combat, and usage estimate) plus a best-effort `deleteSessionEntriesRemote` (the durable Supabase journal rows); cross-timeline legacies/echoes are left intact by design.
  - `character-creation.tsx` — Multi-step character creation flow. Two paths: (1) AI-driven prologue: AI-generated scenes that each offer four affinity-tagged choices. The engine accumulates the picked choices' affinity weights via `tallyAffinities` and computes the finale candidate potions via `selectTopCandidates` (deterministic — the AI never decides the pathway). A separate `generatePrologueFinale` call renders those top candidates as potions; the player's pick (`dominantAffinity` of the chosen potion) sets the pathway. The picked finale potion is NOT fed back into the tally. An **epoch selector** (issues #26/#29) on mode-select chooses the starting era (all five epochs; Fifth default — it sets the narrator's epoch tone layer). A **start-location picker** (varied story openings) on the final first-potion step chooses where the chronicle opens: `startLocationsForEpoch(epoch)` lists the epoch's distinct places (Tingen/Backlund/Trier/Bayam plus the farther regions Pritz Harbor/Enmat Harbor/Feysac), with "Surprise me" (the default) keeping it random; places that thematically suit the chosen pathway are flagged as a suggestion only (never auto-applied). The chosen location (or `null` for random) threads through `onComplete` → `play-dashboard` → `selectStartScenario`/`selectStartScenarioForLocation` → `createDefaultGameState`, which sets `GameState.location` + `GameState.openingBeat`. (2) Manual path: direct pathway selection from all 9 playable pathways (`ALL_PATHWAYS`, ids 1-9) + character sheet. The AI prologue keeps its fixed 4 thematic affinities (issue #53 contract); only the manual selection lists all 9. Both paths end with the first-potion narrative scene. On "Begin Your Chronicle" the AI path also composes a durable **prologue recap** (`buildPrologueRecap` — defining choices + finale encounter + the potion drunk) and threads it through `onComplete` → `play-dashboard` → `createDefaultGameState` into `GameState.prologueRecap`, so the main narrator (which never sees the prologue prompt) continues the thread seamlessly; the manual path passes an empty recap. A skippable **tutorial introduction** (issue #14, `TUTORIAL_SCENES` from `@/lib/game/tutorial.ts`) is offered on mode-select for newcomers — four hand-authored scenes (Beyonders & secrecy, potions & the fatal risk, the acting method, pathways) that flow back into mode-select; veterans skip without losing anything.
  - `game-loop.tsx` — Core game loop UI (situation/choices/resolution/consequences phases). The whole surface is wrapped in `SanityEffects` so CSS distortion tracks the (hidden) sanity meter. The status bar shows the sequence/pathway, location, the **epoch label** (`getEpoch(...).name`, character epoch isolation), turn count, the potion-digestion meter and — only when the player has enabled it in Settings (`sanityMeterVisible`) — the numeric sanity meter; otherwise sanity is communicated purely through the visual effects. `buildAICallParams` threads the character's `epoch` AND `sequenceLevel` into `selectCuratedLore` so the narrator's curated lore is epoch-gated (no crossover) and sequence-gated (progressive disclosure — deeper pathway rungs unlock as the character advances). Each AI turn (`generateSituation`/`resolveChoice`) also calls `retrieveLoreForTurn` (`lore-retrieval-client.ts`) best-effort and passes the gated novel/wiki chunks as `generate({ retrievedChunks })`; retrieval never blocks a turn (resolves to `[]` when no embedding endpoint/corpus is available).
  - `lore-retrieval-client.ts` (`retrieveLoreForTurn`) — the network glue for gated RAG retrieval (issues #63/#64): resolves an embedding transport (operator endpoint if configured, else the player's Ollama) and the authenticated Supabase `match_source_chunks` RPC, runs the timeline/epoch/concealment-gated `retrieveChunks` (query + chunk shaping from `@/lib/game`; concealment ceiling from the character's sequence), and returns `[]` on any failure. Pure decisions live under the coverage mandate; this thin shell does not. The consequences panel previews the digestion change from the acting evaluation and surfaces a completion event when a potion is fully digested. When sanity bottoms out (`isLossOfControl`), a `FailurePanel` (issue #12) presents the verdict: a **setback** is endured in place (`applySetback` — sanity sliver, lost items, displacement, reputation memory fact; transient consequence report) while **permadeath** runs `handlePermadeath` — legacy persisted under `LEGACIES_KEY`, an artifact echo minted under `ECHOES_KEY` (issue #31), a `death` journal entry recorded, the descent narrated by the AI best-effort (deterministic `fallbackDescentScene` otherwise), and the session marked `ended`. An ended session renders the `DeathScreen` (the chronicle stays readable as a historical record) with two paths: _begin anew in this world_ (legacies persist — `play-dashboard` seeds them into the next character's memory via `legaciesToFacts`) or _restart the timeline completely_ (wipes `LEGACIES_KEY` + `ECHOES_KEY`, canonical baseline). A session **token-usage estimate** (issue #15) accumulates each `generate()` call's rough prompt/output token counts (`@/lib/ai` `usage.ts`; persisted under its own `lotm:usage:` localStorage key) and renders in the status bar with an order-of-magnitude cost. The choices phase is **mode-aware** (issue #24): `detectInputMode` infers exploration/ritual/dialogue/investigation from the scene and a `ModeAssist` block renders the matching guided interface (ritual steps + inventory material chips + petition; speak-with/ask-about; two-clue deduction over `gatherClues`) — every guided action composes into the validated free-text pipeline. The choices phase also offers an optional **free-text action box** (issue #19): input is validated by `validateFreeText` (exploit declarations rejected with in-world narration via a `role="status"` line), then wrapped via `freeTextToChoice` and dispatched through the normal SELECT_CHOICE machinery. Once the current potion is fully digested and the rung is advanceable (Seq 9-2) the choices phase offers two panels (gated on live state — `digestion.complete` + `isAdvanceableSequence`, so an existing save that drifted ahead of the engine recovers simply by loading, with no destructive migration). First a `PotionPreparationPanel` (issue #84): the next potion's `potionPreparationPlan` checklist where each missing prerequisite is **bought** with funds or — for the Beyonder Characteristic — **hunted** via a combat encounter (`handleHuntItem` → `startCombat(false, itemName)` stamps the objective onto the persisted `CombatEncounter.huntTarget`, so `handleCombatResult` grants it via `deliverHuntedItem` on victory even after a mid-hunt reload); deeper rungs cost more and the Characteristic becomes hunt-only, so this is the route that makes advancement actually attainable (inventory is not AI-mutable). When everything is in hand the panel hides and the `AdvancementPanel` takes over: the engine's `advancementRequirements` checklist (canon ingredients and, from Sequence 5, the Advancement Ritual) and a two-step arm-then-confirm attempt; `attemptAdvancement` either advances the sequence (the engine writes `sequenceLevel`, re-seeds digestion, and the `"advancement"` instruction narrates the rite into a status banner) or yields a `lost-control` verdict routed through the shared setback (`applySetback`) / permadeath (`concludeChronicle`) paths — a loss of control is always possible and worse at the higher rungs. At Sequence 1 the choices phase also offers an `ApotheosisPanel` (issue #30): the engine's requirement checklist (`apotheosisRequirements`) and a two-step arm-then-confirm attempt; on success the session jumps to Sequence 0 with an "above the sequences" banner, on failure `attemptApotheosis` yields a catastrophic verdict that runs through the shared `concludeChronicle` permadeath path. A Sequence 0 True God's per-turn worshipper petitions (`drawPetition`) fold into session memory in `handleContinue`, and `buildAICallParams` swaps in `TRUE_GOD_ABILITIES`/`TRUE_GOD_ACTING` (Seq 0 has no rules `Sequence`). A `CombatLauncher` in the choices phase starts a combat encounter (`createEncounter`, enemy derived from the scene); an active encounter (persisted under its own `lotm:combat:` localStorage key, separate from the session) takes over the surface via `CombatEncounterView`, and on resolution `applyCombatResult` writes sanity/inventory/injuries back to the session.
  - `combat-encounter.tsx` (`CombatEncounterView`) — Combat UI (issue #10): a preparation form (intelligence, terrain, ritual materials / sealed artifacts from inventory, readied abilities), an exchange of inline tactical decision points, and a resolution screen (outcome, injuries, sanity, items, dropped characteristic). All mechanics come from the pure engine in `@/lib/game/combat.ts`; the component layers optional pathway-flavoured AI narration (instruction `"combat"`) on top, falling back to the engine's own text when no provider is configured.
  - `journal-panel.tsx` (`JournalPanel`) — Story journal viewer (issue #11): per-session chronicle picker, arc-grouped timeline, expandable entry detail, player annotations (add/edit/delete — player-only, never in AI context; synced to Supabase immediately and best-effort), text/event-type filtering, and Markdown export (Blob download). Entries are captured by `game-loop.tsx` on APPLY_CONSEQUENCES via `deriveJournalEntries` (AI flag + deterministic detection), stored under `lotm-rpg-journal-<sessionId>` and batch-synced to `journal_entries`.
  - `sanity-effects.tsx` (`SanityEffects`) — wraps the game surface and applies the per-tier CSS class + vignette/corruption overlay (`globals.css` `.sanity-fx-*` / `.sanity-overlay`) based on `sanityEffects()`. The primary channel for the hidden sanity meter.
  - `society-panel.tsx` (`SocietyPanel`) — The Gathering (/society, issue #32): found the pathway-appropriate society (gated at Seq ≤7), extend invitations, convene above the gray fog (cooldown-gated) — outcomes write intel facts into session memory, traded items into the inventory, and a major-event journal entry; member cards show code name, hint, arc, trust bar, and an arc-resolution action when a member's matter comes to a head.
  - `showcase-panel.tsx` (`ShowcasePanel`) — Player profile (/profile, issue #18): achievements with descriptions, the divergence meter, public stats, and the publish flow — display name, a **public/private switch (private by default; private rows never reach the leaderboard — RLS)**, explicit Publish.
  - `leaderboard-panel.tsx` (`LeaderboardPanel`) — The Board (/leaderboard, issue #18): public showcases ranked by highest sequence / achievement count / divergence, filterable by pathway.
  - `market-panel.tsx` (`MarketPanel`) — The Night Market (/market, issue #16): browse with category/price/text filters, list-with-escrow (failed listings return the item), atomic purchase with funds settlement, delisting, the seller/buyer ledger, and price guidance. The player-listing form is scoped to reagents (`isReagentCategory`); a separate **"The fence"** section sells `mundane` belongings to a non-player vendor via `sellItemToVendor` (local, offline-capable — "sellable, but not to users", issue #90). Live-ish via a best-effort Supabase realtime subscription + refetch-on-action.
  - `world-messages.tsx` (`WorldMessages`) — Shared world messages (issue #17), mounted in the choices phase: up to two helpfulness-weighted notes from other timelines render as subtle occult-glow lines with Helpful/Unhelpful rating (one vote, then "The fog takes note."); "Leave a message" composes strictly from template + slot dropdowns (no free text by construction). All networking best-effort — signed-out/offline players just see nothing.
  - `scene-art.tsx` (`SceneArt`) + `scene-art-cache.ts` — Scene illustrations (issue #20): renders from the IndexedDB cache instantly; when absent and the player opted in with an OpenAI key, generates once via `generateSceneArt`, caches under `sceneArtKey(sessionId, turn)`, and fades in. Failures stay silent (art is garnish). Mounted in the consequences phase (triggered by the AI's journal flag) and, cache-only, in expanded journal entries.
  - `sanity-preferences.tsx` (`SanityPreferences`) — Settings toggles for display preferences: `sanityMeterVisible` (hidden by default), `sceneArtEnabled` (issue #20, opt-in AI art) and `highContrast` (issue #13 — sets `data-contrast="high"` on `<html>` via `applyContrastPreference`). Persisted to localStorage (`PREFERENCES_KEY`).
  - `preference-effects.tsx` (`PreferenceEffects`) — mounted once in the game layout; applies persisted preferences that live outside React (currently the high-contrast attribute). Renders nothing.
  - `character-sheet.tsx` (`CharacterSheet`) — Character sheet panel (issue #13): identity (name, sequence/pathway, an **epoch badge** — `"<name> · <era>"` from `getEpoch(state.epoch)`, character epoch isolation — background, ended-state badge), **cumulative abilities** grouped by originating rung (`getCumulativeAbilityGroups` — every power from the Sequences climbed, current rung first, earlier rungs flagged "Enhanced") + the current rung's acting requirements from the rules engine, condition (sanity as an in-world descriptor — never a number, preserving the hidden-meter design; digestion, injuries, quests), a **True Self** editor section (character-info storage) above Identities — edit name/gender/pronouns/appearance/epithet/age/marks + demeanor-trait chips, a "people who knew me won't recognise this face" recognition toggle (pre-ticked via `isDrasticChange`), a former-names line, a player-only personal log (add/delete), and a "Begin the rite of the {rite}" button when `transformationRiteFor` matches the current rung (pre-fills the editor); commits via `applyProfileChange` → persisted `profileState`+`gameState`, the **Identities** section (issue #22: craft/discard personas, deliberate face-switching, exposure-risk bars, per-identity standing and witnesses; gated flawless/full/basic tier via `identityCapability` — flawless personas show a flawless/fate-proof tag and no exposure bar), the inventory as parchment item cards grouped by category, and a **Delete Character** danger zone scoped to the selected Beyonder — a two-step confirm that runs the same `characterDeletionPlan` + best-effort `deleteSessionEntriesRemote` as the dashboard's Manage view (wipes the local save, journal, in-progress combat, and usage estimate; cross-timeline legacies/echoes are left intact). Game-loop side: each consequences turn runs `recordIdentityUse`/`checkExposure` (exposure events journal as major events and surface as an in-world notice) and — while the true face is shown — `pierceRecognition` (a present NPC who knew the prior self may deduce the connection); it threads `identityPromptContext`/`profilePromptContext`/`recognitionPromptContext` into every `generate()` call, and a `SelfChangeConfirm` banner surfaces any AI `proposedSelfChange` for a one-tap confirm (with an overridable recognition-gap checkbox) before the change is ever applied.
  - `glossary-panel.tsx` (`GlossaryPanel`) — In-game glossary (issue #14) with **progressive disclosure**: entries (from `@/lib/lore/glossary.ts`) unlock at sequence thresholds keyed to the active session, deep-game concepts stay sealed (shown only as a count), search included. The active session's **epoch** is read alongside its sequence and threaded into `glossaryForSequence`/`sealedTermCount` (character epoch isolation), so a non-Fifth character sees only universal mechanics plus its own era's terms — never the Fifth-Epoch nations/churches, and the sealed count never leaks their existence. Written as world-building, never a manual.
  - `character-actions.ts` (`purgeCharacter`) — shared character-deletion side-effects used by both the character sheet's danger zone and the dashboard's Manage view: wipes the local save/journal/combat/usage via `characterDeletionPlan` + a best-effort `deleteSessionEntriesRemote` (durable Supabase journal rows). Callers own their own list/selection state update. Cross-timeline legacies/echoes are left intact.
  - `first-time-hint.tsx` (`FirstTimeHint`) — one-shot dismissible pointer per screen (issue #14); dismissal persists per hint id under `lotm-rpg-hint-<id>`. Used on the journal, map, and character pages.
  - `map-panel.tsx` (`MapPanel`) — Epoch-keyed gazetteer (issue #13, character epoch isolation). The atlas comes from `gazetteerForEpoch(session.gameState.epoch)` in `@/lib/lore`: the Fifth Epoch shows Tingen's districts plus a "Farther afield" travel section (Backlund/Trier/Bayam, issue #23) with `travelTo` from `@/lib/game`; **earlier epochs show their own era-appropriate region list with the travel section hidden** (the CITIES model is Fifth-Epoch only). Blurbs are hand-written street-level PUBLIC knowledge (the curated lore is narrator-only and must not leak into the atlas); the active session's location pins a "You are here" badge only when it names a district specifically.

## Conventions

- Server Components by default. Only use `"use client"` when the component needs browser APIs, hooks, or event handlers.
- Auth forms are client components — they manage form state and call Supabase Auth methods directly.
- Game sidebar is a client component — uses `usePathname` for active route highlighting and `useState` for mobile toggle.
- Provider config, play dashboard, and game loop are client components — manage localStorage for BYOK keys and session persistence.
- Game loop orchestrates AI calls via `generate()` from `@/lib/ai` and state transitions via `transition()` from `@/lib/game`.
- No component libraries. Build with Tailwind utility classes using the Victorian steampunk theme tokens.
- Atmosphere utilities (issue #13, `globals.css`): `.parchment` (paper-grain narrative panels), `.gaslit` (warm halo behind headings), `.fog-overlay` (drifting fog banks, mounted once on the game layout), `.candle-flicker` (uneven living glow). All decorative, reduced-motion-safe, and muted under high-contrast mode.
- Forms handle loading state, error display, and success redirects internally.
- Use `useSyncExternalStore` for initial localStorage reads — avoid `setState` inside `useEffect` bodies (React 19 lint rule).
- **Accessibility is required (WCAG 2.2 AA).** Follow `docs/rules/accessibility.md` for every component: accessible names, ARIA state, `role="status"`/`alert`, decorative `aria-hidden`, no low-opacity meaningful text, and ≥24px targets. Add/extend the axe suite in `src/test/a11y.test.tsx` when adding a screen or significant interactive component.
