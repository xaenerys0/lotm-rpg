# Components

## Organization

- `auth/` — Authentication forms (`login-form.tsx`, `signup-form.tsx`)
- `pwa/` — Progressive Web App helpers (mounted globally in the root layout):
  - `service-worker-registrar.tsx` (`ServiceWorkerRegistrar`) — registers `/sw.js`; renders nothing
  - `install-prompt.tsx` (`InstallPrompt`) — Android install button (via `beforeinstallprompt`) / iOS "Add to Home Screen" hint; self-hides when already installed
- `game/` — Game shell and gameplay components:
  - `game-sidebar.tsx` — Sidebar navigation, sign-out (collapsible on mobile), and the global **active-character switcher** (`CharacterSwitcher`, active-character sync): one control visible on every page that writes the shared active-character pointer (`saveActiveSessionId`) the Map/Journal/Character/Glossary/Market/Society/Profile all read. Renders nothing with no saves, a static name with one, a `<select>` to switch between several (reactive via `useSessionSummaries`/`useActiveSessionId`).
  - `mobile-nav.tsx` (`MobileNav`) — Mobile bottom navigation (issue #27): fixed thumb-reach bar (Play/Character/Journal/Map, ≥48px targets, `aria-current`, safe-area inset padding) shown below `md`; the game layout adds `pb-20 md:pb-0` so content never hides behind it. Heavy sanity/fog animation is softened under 640px for low-power GPUs (`globals.css`).
  - `image-provider-config.tsx` (`ImageProviderConfig`) — **Image model selection** for scene art (issue #20), configured INDEPENDENTLY of the text narrator: a provider grid (OpenAI / Ollama local / local Stable Diffusion — **Ollama Cloud is deliberately absent: ollama.com serves no image-generation endpoint, so it could never produce art**), an API key field (only when the provider needs one — `imageProviderRequiresKey`, i.e. OpenAI), a base URL field (Ollama / local-SD — `imageProviderNeedsBaseUrl`), and a model picker from `IMAGE_PROVIDER_MODELS` with a "Custom model…" escape hatch (free-form id for the diffusion backends; local-SD is a single optional checkpoint field since the WebUI uses its loaded model). Persisted to its OWN localStorage key (`IMAGE_PROVIDER_CONFIG_KEY`, never synced — BYOK); a stale saved `ollama-cloud` provider falls back to the OpenAI default on load. A Remove-key button mirrors `provider-config.tsx`. SSR-safe via the same `useSyncExternalStore` + `formOverride` snapshot pattern. So a player narrating on Ollama can still drop in an OpenAI key only for images (or none). Mounted in Settings → Scene art; gameplay reads the saved config in `game-loop.tsx` and passes it to `SceneArt`.
  - `provider-config.tsx` — AI provider BYOK configuration (provider, API key, models). Model dropdowns prefer a live catalog fetched via `listProviderModels()` (cached in localStorage with a TTL, refreshable, auto-fetched after a successful connection test), falling back to the static `PROVIDER_MODELS` map. Completing issue #15: an always-visible **privacy disclosure** (key lives only in browser localStorage, sent only browser→provider, never our servers, nothing key-derived logged) and a **Remove key** button that wipes the key from the form AND the persisted config immediately (rotation/removal at any time). For Ollama Cloud, once a live catalog has been fetched, `findUnservedModels` (`@/lib/ai`) flags any selected routine/premium model missing from it with an inline `role="alert"` warning — pre-empting the mid-game 403 an unserved premium model would otherwise cause. Scoped to Ollama Cloud on purpose: there the live `/models` ids are exactly the chat ids and a missing model genuinely 403s; other providers can list ids in a different shape than the configured/default id (and a missing model 404s), where the dropdown's `withSelected()` already accommodates the mismatch. Providers flagged `needsAccessProbe` in the `PROVIDERS` metadata (currently only Ollama Cloud — a single flag like `needsBaseUrl`/`requiresAuth` rather than scattered id checks) also get a **"Check models"** action and an auto-check after a successful connection test that runs `probeModelAccess` (`@/lib/ai`) on the two selected models — ollama.com lists its whole cloud catalog regardless of plan, so a selectable model can still 403 ("requires a subscription") at chat time. The probe test-calls each model and lists any the account can't run (each with its own reason) in a `role="alert"` with an upgrade link, or shows a count-aware `role="status"` all-clear. A monotonic `probeIdRef` guards against a late-resolving probe overwriting state after the selection/provider changed; the result is cleared (and the id bumped) on any field edit or provider switch, and the auto-check skips when the current selection already has a result so re-testing doesn't re-spend probe calls.
  - `play-dashboard.tsx` — Play page dashboard (new game, character creation, resume, manage). The home view is a single **Start New Game** CTA above one **"Your Characters"** roster (no separate "Continue" box — the old Continue + Character Summary sections were redundant and both capped at 3): every saved character is a clickable card (`handleContinue` → `saveActiveSessionId` makes it the active character everywhere, then opens the `GameLoop`), **paginated** at `PAGE_SIZE` (6) per page with Previous/Next controls and a `role="status"` "Page X of Y" indicator; `safePage` clamps the page during render so a shrinking roster (e.g. after a delete) never points past the end. Seeds each new chronicle with permadeath legacy facts (issue #12) and **timeline echoes** (issue #31): artifacts under `ECHOES_KEY` from fallen predecessors may start in the inventory (`pickStartingEcho` → `artifactToItem`, paradox-guarded to the chosen epoch or earlier) with narrator foreshadow facts via `echoFacts`. **Start archetypes (issue #131):** the creation flow passes a single discriminated `StartSelection` (`random` | `location` | `archetype` | `custom`); `handleStartNewGame` resolves a curated preset via `getStartArchetype` or a player-authored circle via `buildCustomArchetype(circle, epoch)`, threads it into `createDefaultGameState(..., archetype)` (location/opening/grounding), then applies `seedArchetype` to the created session (tracked-ally roster, optional society pre-membership, relationship facts); the plain location pick is skipped when a circle is chosen. A **Manage characters** view (reached from the roster header) lists every saved character with a two-step confirm **Delete** that runs `characterDeletionPlan` (wipes the local save, journal, in-progress combat, and usage estimate) plus a best-effort `deleteSessionEntriesRemote` (the durable Supabase journal rows); cross-timeline legacies/echoes are left intact by design. When `devToolsEnabled()` (`NEXT_PUBLIC_DEV_TOOLS`), the home view also shows a **"Seed test character"** button (`createTestCharacter` → persist + activate) for fast scene-art testing; that character (fixed `TEST_CHARACTER_ID`) is protected from deletion — the Manage view shows a "Protected" label instead of Delete, and the character sheet's danger zone hides for it (`isUndeletableCharacter`).
  - `character-creation.tsx` — Multi-step character creation flow. Two paths: (1) AI-driven prologue: AI-generated scenes that each offer 3–5 choices, each LEANING toward 1–2 neighboring pathways within one canonical region (issue #119 — never a clean 1:1 "the X option"), with each scene branching across ≥2 regions. The engine accumulates the picked choices' blended affinity weights via `tallyAffinities` and computes the finale candidate potions via `selectTopCandidates` (deterministic — the AI never decides the pathway; the pathway emerges from the pattern of picks and can land on any of the nine playable pathways). A separate `generatePrologueFinale` call renders those top candidates as potions; the player's pick (`dominantAffinity` of the chosen potion) sets the pathway. The picked finale potion is NOT fed back into the tally. An **epoch selector** (issues #26/#29) on mode-select chooses the starting era (all five epochs; Fifth default — it sets the narrator's epoch tone layer). A **start-location picker** (varied story openings) on the final first-potion step chooses where the chronicle opens: `startLocationsForEpoch(epoch)` lists the epoch's distinct places (Tingen/Backlund/Trier/Bayam plus the farther regions Pritz Harbor/Enmat Harbor/Feysac), with "Surprise me" (the default) keeping it random; places that thematically suit the chosen pathway are flagged as a suggestion only (never auto-applied). The chronicle-start choice threads through `onComplete` as a single discriminated `StartSelection` (exported from this file: `random` | `location` | `archetype` | `custom`) → `play-dashboard` → `createDefaultGameState`, which sets `GameState.location` + `GameState.openingBeat`. **Start archetypes (issue #131):** the picker is one `<select>` (single `startChoice` state, reset on epoch change so a pick can't leak across epochs) offering, beyond plain places: a **"Begin in someone's circle (examples)"** `<optgroup>` of `startArchetypesForEpoch(epoch)` curated PRESETS (value `archetype:<id>`), and a **"Describe your own circle"** option (value `custom`) that reveals an inline form — a free-text **tie**, a **companions** add-control (chips; a `circleNpcSuggestions(epoch)` datalist suggests canon NPCs but ANY invented name is accepted), and a **begins-in** location. The custom form's fields build a `CustomStartCircle` → `buildCustomArchetype` in play-dashboard, so creativity is never capped by the preset list; curated archetypes are just examples. (2) Manual path: direct pathway selection from `ALL_PATHWAYS` + character sheet. Both the AI prologue and the manual path can now reach all nine playable pathways (issue #119); the becoming-scene content (`FIRST_POTION_NARRATIVE`/`POTION_HEADINGS`) covers ids 1–9 (extending to all 22 is tracked in #120). Both paths end with the first-potion narrative scene. On "Begin Your Chronicle" the AI path also composes a durable **prologue recap** (`buildPrologueRecap` — defining choices + finale encounter + the potion drunk) and threads it through `onComplete` → `play-dashboard` → `createDefaultGameState` into `GameState.prologueRecap`, so the main narrator (which never sees the prologue prompt) continues the thread seamlessly; the manual path passes an empty recap. A skippable **tutorial introduction** (issue #14, `TUTORIAL_SCENES` from `@/lib/game/tutorial.ts`) is offered on mode-select for newcomers — four hand-authored scenes (Beyonders & secrecy, potions & the fatal risk, the acting method, pathways) that flow back into mode-select; veterans skip without losing anything.
  - `game-loop.tsx` — Core game loop UI (situation/choices/resolution/consequences phases). The whole surface is wrapped in `SanityEffects` so CSS distortion tracks the (hidden) sanity meter. The status bar shows the sequence/pathway, location, the **epoch label** (`getEpoch(...).name`, character epoch isolation), turn count, the potion-digestion meter (issue #95: shown only once the acting method is discovered AND `digestionMeterVisible` is enabled — `resolveActingMethodState(session.actingMethodState).knowsMethod`) and — only when the player has enabled it in Settings (`sanityMeterVisible`) — the numeric sanity meter; otherwise sanity/digestion are communicated purely through the visual effects and vague prose. The consequences panel mirrors this: pre-discovery only vague `digestionFeedback` prose shows (alignment %, digestion delta, and the AI's reasoning are gated on `knowsMethod && digestionMeterVisible`), the sanity number follows `sanityMeterVisible` (qualitative descriptor otherwise) from a locally-recomputed hybrid breakdown, and `handleContinue` threads `resolveActingMethodState(...)` into the new `applyResolution` signature, persists `result.actingMethodState`, and announces a discovery via a `role="status"` notice. It also threads `gameState.epoch`, `resolveTrackedNpcState(session.trackedNpcState)`, and `preferences.movementGateEnabled` into `applyResolution` (issue #101) so the movement gate runs each turn — a refused cross-city teleport leaves the location unchanged and lands its redirect fact in memory, with no special UI. `buildAICallParams` threads the character's `epoch` AND `sequenceLevel` into `selectCuratedLore` so the narrator's curated lore is epoch-gated (no crossover) and sequence-gated (progressive disclosure — deeper pathway rungs unlock as the character advances). Each AI turn (`generateSituation`/`resolveChoice`) also calls `retrieveLoreForTurn` (`lore-retrieval-client.ts`) best-effort and passes the gated novel/wiki chunks as `generate({ retrievedChunks })`; retrieval never blocks a turn (resolves to `[]` when no embedding endpoint/corpus is available).
  - `lore-retrieval-client.ts` (`retrieveLoreForTurn`) — the network glue for gated RAG retrieval (issues #63/#64): resolves an embedding transport (the player's own Ollama first when it's their chat provider — localhost embeds free/private; else a self-hosted operator endpoint; else the Cloudflare Workers AI proxy as the general fallback when `NEXT_PUBLIC_CLOUDFLARE_EMBEDDING` is set) and the authenticated Supabase `match_source_chunks` RPC, runs the timeline/epoch/concealment-gated `retrieveChunks` (query + chunk shaping from `@/lib/game`; concealment ceiling from the character's sequence), and returns `[]` on any failure. Pure decisions live under the coverage mandate; this thin shell does not. The consequences panel previews the digestion change from the acting evaluation and surfaces a completion event when a potion is fully digested. When sanity bottoms out (`isLossOfControl`), a `FailurePanel` (issue #12) presents the verdict: a **setback** is endured in place (`applySetback` — sanity sliver, lost items, displacement, reputation memory fact; transient consequence report) while **permadeath** runs `handlePermadeath` — legacy persisted under `LEGACIES_KEY`, an artifact echo minted under `ECHOES_KEY` (issue #31), a `death` journal entry recorded, the descent narrated by the AI best-effort (deterministic `fallbackDescentScene` otherwise), and the session marked `ended`. An ended session renders the `DeathScreen` (the chronicle stays readable as a historical record) with two paths: _begin anew in this world_ (legacies persist — `play-dashboard` seeds them into the next character's memory via `legaciesToFacts`) or _restart the timeline completely_ (wipes `LEGACIES_KEY` + `ECHOES_KEY`, canonical baseline). A session **token-usage estimate** (issue #15) accumulates each `generate()` call's rough prompt/output token counts (`@/lib/ai` `usage.ts`; persisted under its own `lotm:usage:` localStorage key) and renders in the status bar with an order-of-magnitude cost. The choices phase is **mode-aware** (issue #24): `detectInputMode` infers exploration/ritual/dialogue/investigation from the scene and a `ModeAssist` block renders the matching guided interface (ritual steps + inventory material chips + petition; speak-with/ask-about; two-clue deduction over `gatherClues`) — every guided action composes into the validated free-text pipeline. The choices phase also offers an optional **free-text action box** (issue #19): input is validated by `validateFreeText` (exploit declarations rejected with in-world narration via a `role="status"` line), then wrapped via `freeTextToChoice` and dispatched through the normal SELECT_CHOICE machinery. **Assuming a persona** (issue #22) is a deliberate engine action, not narration: an `IdentityActionPanel` in the choices phase lets the player slip into / out of any prepared face via `switchIdentity` (the same call the character sheet uses, persisted through `updateSession`; the narrator picks up the worn persona via `identityPromptContext` next turn). It renders nothing when no face has been prepared (no per-turn nag). The "you have prepared none" steer surfaces only when the player actually reaches for one: `handleFreeText` guards the box — a typed assume attempt (`detectAssumeIdentityIntent`) is short-circuited to a `role="status"` line (`ASSUME_VIA_PANEL_NARRATIVE` when faces exist, `UNPREPARED_IDENTITY_NARRATIVE` otherwise) instead of being sent to the AI, so it can't silently fail to associate and only pollute the context. Once the current potion is fully digested and the rung is advanceable (Seq 9-2) the choices phase offers two panels (gated on live state — `digestion.complete` + `isAdvanceableSequence`, so an existing save that drifted ahead of the engine recovers simply by loading, with no destructive migration). First a `PotionPreparationPanel` (issue #84): the next potion's `potionPreparationPlan` checklist where each missing prerequisite is **bought** with funds or — for the Beyonder Characteristic — **hunted**. Hunting now starts a tracked, multi-turn QUEST rather than jumping to combat: `handleHuntItem` → `startHunt` registers a `HuntState` on `session.hunts` (several hunts can run at once) and adds an AI-visible quest label; each turn `handleContinue` calls `advanceActiveHunts` to close the distance (scaled by rung depth), and once cornered the player engages the unified combat system (`handleEngageHunt` → `startCombat(false, itemName)`, which stamps `CombatEncounter.huntTarget` so `handleCombatResult` grants it via `deliverHuntedItem` on victory and then `clearHunt`s the quest). deeper rungs cost more and the Characteristic becomes hunt-only, so this is the route that makes advancement actually attainable (inventory is not AI-mutable). A general **`QuestLogPanel`** in the choices phase surfaces ALL tracked quests in one place — the AI-narrated story quests in `gameState.activeQuests` plus the structured hunts with a tracking `role="progressbar"`, an "Engage the quarry" button when ready, and "Abandon the hunt". When everything is in hand the prep panel hides and the `AdvancementPanel` takes over: the engine's `advancementRequirements` checklist (canon ingredients and, from Sequence 5, the Advancement Ritual) and a two-step arm-then-confirm attempt; `attemptAdvancement` either advances the sequence (the engine writes `sequenceLevel`, re-seeds digestion, and the climb is **routed through the normal turn loop via `transition(..., { type: "ENGINE_RESOLUTION", ... })`** — the `"advancement"` narration becomes the inline `consequences` scene and a memory turn record, so the narrator stays aware, with no side-channel banner) or yields a `lost-control` verdict routed through the shared setback (`applySetback`) / permadeath (`concludeChronicle`) paths — a loss of control is always possible and worse at the higher rungs. At Sequence 1 the choices phase also offers an `ApotheosisPanel` (issue #30): the engine's requirement checklist (`apotheosisRequirements`) and a two-step arm-then-confirm attempt; on success the ascent is routed through the same `ENGINE_RESOLUTION` turn (the tease renders inline + is remembered), on failure `attemptApotheosis` yields a catastrophic verdict that runs through the shared `concludeChronicle` permadeath path. A Sequence 0 True God's per-turn worshipper petitions (`drawPetition`) fold into session memory in `handleContinue`, and `buildAICallParams` swaps in `TRUE_GOD_ABILITIES`/`TRUE_GOD_ACTING` (Seq 0 has no rules `Sequence`). A `CombatLauncher` in the choices phase starts a combat encounter (`createEncounter`, enemy derived from the scene); an active encounter (persisted under its own `lotm:combat:` localStorage key, separate from the session) takes over the surface via `CombatEncounterView`, and on resolution `applyCombatResult` writes sanity/inventory/injuries back to the session.
  - `combat-encounter.tsx` (`CombatEncounterView`) — Combat UI (issue #10): an intel-gated **`EnemyPanel`** dossier (name + description always, plus rough strength / pathway / exact sequence / known abilities revealed by the player's intelligence level via the pure `enemyIntel`), a preparation form (intelligence, terrain, ritual materials / sealed artifacts from inventory, readied abilities), an exchange of inline tactical decision points — now including **dynamic in-the-moment options** for the player's actual learned abilities and carried artifacts (engine-generated; an artifact option consumes the item on use) — and a resolution screen (outcome, injuries, sanity, items, dropped characteristic, plus an optional `SceneArt` illustration of the fight — issue #20, gated on the player's opt-in + image provider). All mechanics come from the pure engine in `@/lib/game/combat.ts`; the component layers optional pathway-flavoured AI narration (instruction `"combat"`) on top, falling back to the engine's own text when no provider is configured. Receives `sessionId`/`imageConfig`/`sceneArtEnabled` from `game-loop.tsx` for the resolution illustration.
  - `journal-panel.tsx` (`JournalPanel`) — Story journal viewer (issue #11): per-session chronicle picker, arc-grouped timeline, expandable entry detail, player annotations (add/edit/delete — player-only, never in AI context; synced to Supabase immediately and best-effort), text/event-type filtering, and Markdown export (Blob download). Entries are captured by `game-loop.tsx` on APPLY_CONSEQUENCES via `deriveJournalEntries` (AI flag + deterministic detection), stored under `lotm-rpg-journal-<sessionId>` and batch-synced to `journal_entries`.
  - `sanity-effects.tsx` (`SanityEffects`) — wraps the game surface and applies the per-tier CSS class + vignette/corruption overlay (`globals.css` `.sanity-fx-*` / `.sanity-overlay`) based on `sanityEffects()`. The primary channel for the hidden sanity meter.
  - `society-panel.tsx` (`SocietyPanel`) — The Gathering (/society, issue #32): found the pathway-appropriate society (gated at Seq ≤7), extend invitations, convene above the gray fog (cooldown-gated) — outcomes write intel facts into session memory, traded items into the inventory, and a major-event journal entry; member cards show code name, hint, arc, trust bar, and an arc-resolution action when a member's matter comes to a head. The hint/arc PROSE is derived at render via `memberPathwayHint`/`memberArc` (the member persists only stable ids), so copy/grammar edits apply to existing saves too.
  - `showcase-panel.tsx` (`ShowcasePanel`) — Player profile (/profile, issue #18): achievements with descriptions, the divergence meter, public stats, and the publish flow — display name, a **public/private switch (private by default; private rows never reach the leaderboard — RLS)**, explicit Publish.
  - `leaderboard-panel.tsx` (`LeaderboardPanel`) — The Board (/leaderboard, issue #18): public showcases ranked by highest sequence / achievement count / divergence, filterable by pathway.
  - `market-panel.tsx` (`MarketPanel`) — The Night Market (/market, issue #16): browse with category/price/text filters, list-with-escrow (failed listings return the item), atomic purchase with funds settlement, delisting, the seller/buyer ledger, and price guidance. The player-listing form is scoped to reagents (`isReagentCategory`); a separate **"The fence"** section sells `mundane` belongings to a non-player vendor via `sellItemToVendor` (local, offline-capable — "sellable, but not to users", issue #90). Live-ish via a best-effort Supabase realtime subscription + refetch-on-action.
  - `world-messages.tsx` (`WorldMessages`) — Shared world messages (issue #17), mounted in the choices phase: up to two helpfulness-weighted notes from other timelines render as subtle occult-glow lines with Helpful/Unhelpful rating (one vote, then "The fog takes note."); "Leave a message" composes strictly from template + slot dropdowns (no free text by construction). All networking best-effort — signed-out/offline players just see nothing.
  - `scene-art.tsx` (`SceneArt`) + `scene-art-cache.ts` — Scene illustrations (issue #20): renders from the IndexedDB cache instantly; when absent and the player opted in AND configured an image provider, generates once via `generateImage` (image model selection — the provider/model are decoupled from the text narrator, configured in `image-provider-config.tsx`), caches under `sceneArtKey(sessionId, turn)`, and fades in. Takes an `imageConfig: ImageProviderConfig | null` prop (replacing the old text-`config`) and gates on `imageArtSupported(imageConfig)`. Failures stay silent for the player but are now **logged to `console.error`** (provider/model/reason, never the key) so a misconfigured/unsupported image model is diagnosable instead of vanishing. Mounted in the consequences phase (triggered by the AI's journal flag — advancement/apotheosis/pillar attach a deterministic flag), in the **combat resolution screen** (`combat-encounter.tsx`, key `${sessionId}:combat:${encounter.id}`), on the **DeathScreen** (`game-loop.tsx`, key `${sessionId}:death` — permadeath replaces the consequences phase, so it mounts its own), as a dedicated **identity-exposure card** in the choices phase (`game-loop.tsx`, key `${sessionId}:exposure:${turn}` — an engine "major-event" computed at Continue-time, illustrated in the following scene), and, cache-only, in expanded journal entries (`imageConfig={null}`). These cover every `ART_TRIGGERS` eventType (advancement/death/combat/major-event) so each illustrates deterministically rather than relying on narrator luck.
  - `dev-scene-art-harness.tsx` (`DevSceneArtHarness`) — **Dev-only scene-art testbench** (mounted at `/dev/scene-art`, gated by `devToolsEnabled()`/`NEXT_PUBLIC_DEV_TOOLS`). Exercises the REAL image path — saved BYOK image config → `generateImage` → transport → parse → `<img>` — once per trigger eventType, with NO game progression or AI text turn. Deliberately does NOT use `SceneArt` (which replays from IndexedDB): it calls `generateImage` directly and renders the data-URL, so every run re-generates and never pollutes the real cache. Surfaces failures in a `role="alert"`.
  - `sanity-preferences.tsx` (`SanityPreferences`) — Settings toggles for display preferences: `sanityMeterVisible` (hidden by default), `digestionMeterVisible` (issue #95 — mirrors the sanity toggle; inert until the acting method is discovered), `sceneArtEnabled` (issue #20, opt-in AI art — the image provider/model is configured separately in `image-provider-config.tsx`), `movementGateEnabled` (issue #101 — "Movement realism", a `role="switch"` toggle on by default that blocks the narrator from teleporting the character across the map) and `highContrast` (issue #13 — sets `data-contrast="high"` on `<html>` via `applyContrastPreference`). Persisted to localStorage (`PREFERENCES_KEY`).
  - `preference-effects.tsx` (`PreferenceEffects`) — mounted once in the game layout; applies persisted preferences that live outside React (currently the high-contrast attribute). Renders nothing.
  - `cloud-sync.ts` + `cloud-sync-registrar.tsx` + `cloud-hydration-gate.tsx` — Cross-device save sync (cross-device sync). The full `GameSession`, world memory (legacies/echoes), and preferences/hint-dismissals previously lived ONLY in localStorage; Supabase is now the durable, authoritative copy. `cloud-sync.ts` is the browser-only networking shell (pure row-mapping + last-write-wins `reconcile` live in `@/lib/game` under the coverage mandate, like `lore-retrieval-client.ts`): `startCloudSync()` registers a best-effort **sink** into the session store (so every `persistSession`/`saveActiveSessionId` mirrors to the cloud), subscribes to Supabase `onAuthStateChange` (so a sign-out/account-switch in the same tab resets the cached identity and re-hydrates under the new user instead of writing the old one's id), and runs `hydrate()` — reconcile local+cloud saves/world-memory/preferences on entry and write both directions. A local **sync ledger** (`CLOUD_SYNCED_KEY`: the ids/keys seen in the cloud) lets reconcile distinguish a brand-new local record (push) from one deleted on another device (drop), so deletes and full-timeline restarts propagate instead of being resurrected. Best-effort pushers (`pushWorldMemoryToCloud`, `pushPreferencesToCloud`, `deleteSessionFromCloud`) are called from the legacy/echo writes (`game-loop.tsx`), `savePreferences` (`preferences-store.ts`), hint dismissal (`first-time-hint.tsx`), and character deletion (`character-actions.ts`). API keys are NEVER synced (BYOK boundary). `CloudSyncRegistrar` mounts the start-once effect in the game layout; renders nothing. **`CloudHydrationGate`** wraps the game layout's `<main>` and shows an accessible "Restoring your chronicles…" `role="status"` panel until the first hydrate settles (`useCloudSyncStatus()`), gating EVERY surface (not just the dashboard) so no stale local save can be resumed/edited and clobber a newer cloud copy before the pull completes; it blocks only on first load/reload (status then stays "synced"), and resolves near-instantly for signed-out/offline players. Signed-out/offline players keep playing against localStorage — nothing here throws into a render.

## Conventions

- Server Components by default. Only use `"use client"` when the component needs browser APIs, hooks, or event handlers.
- Auth forms are client components — they manage form state and call Supabase Auth methods directly.
- Game sidebar is a client component — uses `usePathname` for active route highlighting and `useState` for mobile toggle.
- Provider config, play dashboard, and game loop are client components — manage localStorage for BYOK keys and session persistence.
- Game loop orchestrates AI calls via `generate()` from `@/lib/ai` and state transitions via `transition()` from `@/lib/game`.
- No component libraries. Build with Tailwind utility classes using the Victorian steampunk theme tokens.
- Atmosphere utilities (issue #13, `globals.css`): `.parchment` (paper-grain narrative panels), `.gaslit` (warm halo behind headings), `.fog-overlay` (drifting fog banks, mounted once on the game layout), `.candle-flicker` (uneven living glow). All decorative, reduced-motion-safe, and muted under high-contrast mode.
- Forms handle loading state, error display, and success redirects internally.
- Use `useSyncExternalStore` for initial localStorage reads — avoid `setState` inside `useEffect` bodies (React 19 lint rule).
- **Active character is a single shared pointer** (`@/lib/react/session-store`, active-character sync). Every page shows the SAME character: read it with `useActiveSession()` (full session, reactive to switches AND in-place edits — a mutation just `persistSession`s and the hook re-reads, no local override), `useActiveSessionId()`, or `useSessionSummaries()` (switcher labels). Change it with `saveActiveSessionId(id)` — the sidebar `CharacterSwitcher`, the Character sheet's "Active Beyonder" selector, the Journal's "Chronicle" picker, and `play-dashboard` (auto-activates on new game / Continue) all write this one pointer. `loadActiveSessionId` self-heals a stale/deleted id to the newest save. Do NOT reintroduce per-component "first-in-index" or local-dropdown character selection — that was the desync bug this replaced.
- **Accessibility is required (WCAG 2.2 AA).** Follow `docs/rules/accessibility.md` for every component: accessible names, ARIA state, `role="status"`/`alert`, decorative `aria-hidden`, no low-opacity meaningful text, and ≥24px targets. Add/extend the axe suite in `src/test/a11y.test.tsx` when adding a screen or significant interactive component.
