# Components

## Organization

- `auth/` — Authentication forms (`login-form.tsx`, `signup-form.tsx`)
- `pwa/` — Progressive Web App helpers (mounted globally in the root layout):
  - `service-worker-registrar.tsx` (`ServiceWorkerRegistrar`) — registers `/sw.js`; renders nothing
  - `install-prompt.tsx` (`InstallPrompt`) — Android install button (via `beforeinstallprompt`) / iOS "Add to Home Screen" hint; self-hides when already installed
- `game/` — Game shell and gameplay components:
  - `game-sidebar.tsx` — Sidebar navigation, sign-out
  - `provider-config.tsx` — AI provider BYOK configuration (provider, API key, models). Model dropdowns prefer a live catalog fetched via `listProviderModels()` (cached in localStorage with a TTL, refreshable, auto-fetched after a successful connection test), falling back to the static `PROVIDER_MODELS` map. Completing issue #15: an always-visible **privacy disclosure** (key lives only in browser localStorage, sent only browser→provider, never our servers, nothing key-derived logged) and a **Remove key** button that wipes the key from the form AND the persisted config immediately (rotation/removal at any time).
  - `play-dashboard.tsx` — Play page dashboard (new game, continue, character creation)
  - `character-creation.tsx` — Multi-step character creation flow. Two paths: (1) AI-driven prologue: AI-generated scenes that each offer four affinity-tagged choices. The engine accumulates the picked choices' affinity weights via `tallyAffinities` and computes the finale candidate potions via `selectTopCandidates` (deterministic — the AI never decides the pathway). A separate `generatePrologueFinale` call renders those top candidates as potions; the player's pick (`dominantAffinity` of the chosen potion) sets the pathway. The picked finale potion is NOT fed back into the tally. (2) Manual path: direct pathway selection from all 9 playable pathways (`ALL_PATHWAYS`, ids 1-9) + character sheet. The AI prologue keeps its fixed 4 thematic affinities (issue #53 contract); only the manual selection lists all 9. Both paths end with the first-potion narrative scene. A skippable **tutorial introduction** (issue #14, `TUTORIAL_SCENES` from `@/lib/game/tutorial.ts`) is offered on mode-select for newcomers — four hand-authored scenes (Beyonders & secrecy, potions & the fatal risk, the acting method, pathways) that flow back into mode-select; veterans skip without losing anything.
  - `game-loop.tsx` — Core game loop UI (situation/choices/resolution/consequences phases). The whole surface is wrapped in `SanityEffects` so CSS distortion tracks the (hidden) sanity meter. The status bar shows the potion-digestion meter and — only when the player has enabled it in Settings (`sanityMeterVisible`) — the numeric sanity meter; otherwise sanity is communicated purely through the visual effects. The consequences panel previews the digestion change from the acting evaluation and surfaces a completion event when a potion is fully digested. When sanity bottoms out (`isLossOfControl`), a `FailurePanel` (issue #12) presents the verdict: a **setback** is endured in place (`applySetback` — sanity sliver, lost items, displacement, reputation memory fact; transient consequence report) while **permadeath** runs `handlePermadeath` — legacy persisted under `LEGACIES_KEY`, a `death` journal entry recorded, the descent narrated by the AI best-effort (deterministic `fallbackDescentScene` otherwise), and the session marked `ended`. An ended session renders the `DeathScreen` (the chronicle stays readable as a historical record) with two paths: _begin anew in this world_ (legacies persist — `play-dashboard` seeds them into the next character's memory via `legaciesToFacts`) or _restart the timeline completely_ (wipes `LEGACIES_KEY`, canonical baseline). A session **token-usage estimate** (issue #15) accumulates each `generate()` call's rough prompt/output token counts (`@/lib/ai` `usage.ts`; persisted under its own `lotm:usage:` localStorage key) and renders in the status bar with an order-of-magnitude cost. The choices phase is **mode-aware** (issue #24): `detectInputMode` infers exploration/ritual/dialogue/investigation from the scene and a `ModeAssist` block renders the matching guided interface (ritual steps + inventory material chips + petition; speak-with/ask-about; two-clue deduction over `gatherClues`) — every guided action composes into the validated free-text pipeline. The choices phase also offers an optional **free-text action box** (issue #19): input is validated by `validateFreeText` (exploit declarations rejected with in-world narration via a `role="status"` line), then wrapped via `freeTextToChoice` and dispatched through the normal SELECT_CHOICE machinery. A `CombatLauncher` in the choices phase starts a combat encounter (`createEncounter`, enemy derived from the scene); an active encounter (persisted under its own `lotm:combat:` localStorage key, separate from the session) takes over the surface via `CombatEncounterView`, and on resolution `applyCombatResult` writes sanity/inventory/injuries back to the session.
  - `combat-encounter.tsx` (`CombatEncounterView`) — Combat UI (issue #10): a preparation form (intelligence, terrain, ritual materials / sealed artifacts from inventory, readied abilities), an exchange of inline tactical decision points, and a resolution screen (outcome, injuries, sanity, items, dropped characteristic). All mechanics come from the pure engine in `@/lib/game/combat.ts`; the component layers optional pathway-flavoured AI narration (instruction `"combat"`) on top, falling back to the engine's own text when no provider is configured.
  - `journal-panel.tsx` (`JournalPanel`) — Story journal viewer (issue #11): per-session chronicle picker, arc-grouped timeline, expandable entry detail, player annotations (add/edit/delete — player-only, never in AI context; synced to Supabase immediately and best-effort), text/event-type filtering, and Markdown export (Blob download). Entries are captured by `game-loop.tsx` on APPLY_CONSEQUENCES via `deriveJournalEntries` (AI flag + deterministic detection), stored under `lotm-rpg-journal-<sessionId>` and batch-synced to `journal_entries`.
  - `sanity-effects.tsx` (`SanityEffects`) — wraps the game surface and applies the per-tier CSS class + vignette/corruption overlay (`globals.css` `.sanity-fx-*` / `.sanity-overlay`) based on `sanityEffects()`. The primary channel for the hidden sanity meter.
  - `showcase-panel.tsx` (`ShowcasePanel`) — Player profile (/profile, issue #18): achievements with descriptions, the divergence meter, public stats, and the publish flow — display name, a **public/private switch (private by default; private rows never reach the leaderboard — RLS)**, explicit Publish.
  - `leaderboard-panel.tsx` (`LeaderboardPanel`) — The Board (/leaderboard, issue #18): public showcases ranked by highest sequence / achievement count / divergence, filterable by pathway.
  - `market-panel.tsx` (`MarketPanel`) — The Night Market (/market, issue #16): browse with category/price/text filters, list-with-escrow (failed listings return the item), atomic purchase with funds settlement, delisting, the seller/buyer ledger, and price guidance. Live-ish via a best-effort Supabase realtime subscription + refetch-on-action.
  - `world-messages.tsx` (`WorldMessages`) — Shared world messages (issue #17), mounted in the choices phase: up to two helpfulness-weighted notes from other timelines render as subtle occult-glow lines with Helpful/Unhelpful rating (one vote, then "The fog takes note."); "Leave a message" composes strictly from template + slot dropdowns (no free text by construction). All networking best-effort — signed-out/offline players just see nothing.
  - `scene-art.tsx` (`SceneArt`) + `scene-art-cache.ts` — Scene illustrations (issue #20): renders from the IndexedDB cache instantly; when absent and the player opted in with an OpenAI key, generates once via `generateSceneArt`, caches under `sceneArtKey(sessionId, turn)`, and fades in. Failures stay silent (art is garnish). Mounted in the consequences phase (triggered by the AI's journal flag) and, cache-only, in expanded journal entries.
  - `sanity-preferences.tsx` (`SanityPreferences`) — Settings toggles for display preferences: `sanityMeterVisible` (hidden by default), `sceneArtEnabled` (issue #20, opt-in AI art) and `highContrast` (issue #13 — sets `data-contrast="high"` on `<html>` via `applyContrastPreference`). Persisted to localStorage (`PREFERENCES_KEY`).
  - `preference-effects.tsx` (`PreferenceEffects`) — mounted once in the game layout; applies persisted preferences that live outside React (currently the high-contrast attribute). Renders nothing.
  - `character-sheet.tsx` (`CharacterSheet`) — Character sheet panel (issue #13): identity (name, sequence/pathway, background, ended-state badge), abilities + acting requirements from the rules engine, condition (sanity as an in-world descriptor — never a number, preserving the hidden-meter design; digestion, injuries, quests), the **Identities** section (issue #22: craft/discard personas, deliberate face-switching, exposure-risk bars, per-identity standing and witnesses; gated full vs basic tier via `identityCapability`), and the inventory as parchment item cards grouped by category. Game-loop side: each consequences turn runs `recordIdentityUse`/`checkExposure` (exposure events journal as major events and surface as an in-world notice) and threads `identityPromptContext` into every `generate()` call.
  - `glossary-panel.tsx` (`GlossaryPanel`) — In-game glossary (issue #14) with **progressive disclosure**: entries (from `@/lib/lore/glossary.ts`) unlock at sequence thresholds keyed to the active session, deep-game concepts stay sealed (shown only as a count), search included. Written as world-building, never a manual.
  - `first-time-hint.tsx` (`FirstTimeHint`) — one-shot dismissible pointer per screen (issue #14); dismissal persists per hint id under `lotm-rpg-hint-<id>`. Used on the journal, map, and character pages.
  - `map-panel.tsx` (`MapPanel`) — Tingen district gazetteer (issue #13). Blurbs are hand-written street-level PUBLIC knowledge (the curated lore is narrator-only and must not leak into the atlas); the active session's location pins a "You are here" badge only when it names a district specifically.

## Conventions

- Server Components by default. Only use `"use client"` when the component needs browser APIs, hooks, or event handlers.
- Auth forms are client components — they manage form state and call Supabase Auth methods directly.
- Game sidebar is a client component — uses `usePathname` for active route highlighting and `useState` for mobile toggle.
- Provider config, play dashboard, and game loop are client components — manage localStorage for BYOK keys and session persistence.
- Game loop orchestrates AI calls via `generate()` from `@/lib/ai` and state transitions via `transition()` from `@/lib/game`.
- No component libraries. Build with Tailwind utility classes using the Victorian steampunk theme tokens.
- Atmosphere utilities (issue #13, `globals.css`): `.parchment` (paper-grain narrative panels), `.gaslit` (warm halo behind headings), `.fog-overlay` (drifting fog banks, mounted once on the game layout), `.candle-flicker` (uneven living glow). All decorative, reduced-motion-safe, and muted under high-contrast mode.
- Forms handle loading state, error display, and success redirects internally.
- Use `useSyncExternalStore` for initial localStorage reads — avoid `setState` inside `useEffect` bodies (React 19 lint rule).
- **Accessibility is required (WCAG 2.2 AA).** Follow `docs/rules/accessibility.md` for every component: accessible names, ARIA state, `role="status"`/`alert`, decorative `aria-hidden`, no low-opacity meaningful text, and ≥24px targets. Add/extend the axe suite in `src/test/a11y.test.tsx` when adding a screen or significant interactive component.
