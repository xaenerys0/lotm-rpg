# Components

## Organization

- `auth/` — Authentication forms (`login-form.tsx`, `signup-form.tsx`)
- `pwa/` — Progressive Web App helpers (mounted globally in the root layout):
  - `service-worker-registrar.tsx` (`ServiceWorkerRegistrar`) — registers `/sw.js`; renders nothing
  - `install-prompt.tsx` (`InstallPrompt`) — Android install button (via `beforeinstallprompt`) / iOS "Add to Home Screen" hint; self-hides when already installed
- `game/` — Game shell and gameplay components:
  - `game-sidebar.tsx` — Sidebar navigation, sign-out (collapsible on mobile), and the global **active-character switcher** (`CharacterSwitcher`, active-character sync): one control visible on every page that writes the shared active-character pointer (`saveActiveSessionId`) the Map/Journal/Character/Glossary/Market/Society/Profile all read. Renders nothing with no saves, a static name with one, a `<select>` to switch between several (reactive via `useSessionSummaries`/`useActiveSessionId`). Takes an optional **`isAdmin`** prop (resolved server-side in the `(game)` layout via `@/lib/supabase/admin`); when true it appends a "Dev Admin" nav link to `/dev/admin` (the page itself 404s a non-admin, so this is purely a convenience affordance).
  - `admin-tools-panel.tsx` (`AdminToolsPanel`) — **Admin / dev test-utilities console** (the `/dev/admin` body, gated server-side on `profiles.is_admin`). Two regions over the pure `@/lib/game` admin builder: **Forge a test character** (pathway/sequence pickers, digestion start/end, an **AI identity generator** — a region picker + "Generate name & background" that calls the player's BYOK provider via `generateCharacterIdentity` (`@/lib/ai`), names following the chosen LOTM region's corpus-grounded naming register, with loading `role="status"` + error `role="alert"`; falls back to a Settings-CTA error when no provider is configured — editable name/background fields, funds/sanity, acting-method/advancement-ready/sample-power/sample-anchor toggles, a grade-grouped sealed-artifact multi-select, and an endgame select — apotheosis/pillar) that `buildAdminCharacter`s, `persistSession`s, indexes, and activates the save; and **Modify the active character** (`useActiveSession`) — set sanity/funds, grant an artifact by code, make advancement-ready, and a two-step **Trigger loss of control** showing the `lossOfControlPreview` severity before `forceLossOfControl`. Themed with the shared `PageHeader` + `.parchment`/`.gaslit`/`.gilt-rule` atmosphere; WCAG-AA (labelled controls, `aria-pressed` toggles, `role="status"`/`alert`). A `.tsx` component — not in `coverage.include`; the pure logic lives in `@/lib/game/admin-tools.ts` (tested) and it is verified via the axe suite + the authenticated `e2e/dev-admin.spec.ts` gate spec.
  - `mobile-nav.tsx` (`MobileNav`) — Mobile bottom navigation (issue #27): fixed thumb-reach bar (Play/Character/Journal/Map, ≥48px targets, `aria-current`, safe-area inset padding) shown below `md`; the game layout adds `pb-20 md:pb-0` so content never hides behind it. Heavy sanity/fog animation is softened under 640px for low-power GPUs (`globals.css`).
  - `image-provider-config.tsx` (`ImageProviderConfig`) — **Image model selection** for scene art (issue #20), configured INDEPENDENTLY of the text narrator: a provider grid (OpenAI / Ollama local / local Stable Diffusion — **Ollama Cloud is deliberately absent: ollama.com serves no image-generation endpoint, so it could never produce art**), an API key field (only when the provider needs one — `imageProviderRequiresKey`, i.e. OpenAI), a base URL field (Ollama / local-SD — `imageProviderNeedsBaseUrl`), and a model picker from `IMAGE_PROVIDER_MODELS` with a "Custom model…" escape hatch (free-form id for the diffusion backends; local-SD is a single optional checkpoint field since the WebUI uses its loaded model). Persisted to its OWN localStorage key (`IMAGE_PROVIDER_CONFIG_KEY`, never synced — BYOK); a stale saved `ollama-cloud` provider falls back to the OpenAI default on load. A Remove-key button mirrors `provider-config.tsx`. SSR-safe via the same `useSyncExternalStore` + `formOverride` snapshot pattern. So a player narrating on Ollama can still drop in an OpenAI key only for images (or none). Mounted in Settings → Scene art; gameplay reads the saved config in `game-loop.tsx` and passes it to `SceneArt`.
  - `provider-config.tsx` — AI provider BYOK configuration (provider, API key, models). Model dropdowns prefer a live catalog fetched via `listProviderModels()` (cached in localStorage with a TTL, refreshable, auto-fetched after a successful connection test), falling back to the static `PROVIDER_MODELS` map. Completing issue #15: an always-visible **privacy disclosure** (key lives only in browser localStorage, sent only browser→provider, never our servers, nothing key-derived logged) and a **Remove key** button that wipes the key from the form AND the persisted config immediately (rotation/removal at any time). For Ollama Cloud, once a live catalog has been fetched, `findUnservedModels` (`@/lib/ai`) flags any selected routine/premium model missing from it with an inline `role="alert"` warning — pre-empting the mid-game 403 an unserved premium model would otherwise cause. Scoped to Ollama Cloud on purpose: there the live `/models` ids are exactly the chat ids and a missing model genuinely 403s; other providers can list ids in a different shape than the configured/default id (and a missing model 404s), where the dropdown's `withSelected()` already accommodates the mismatch. Providers flagged `needsAccessProbe` in the `PROVIDERS` metadata (currently only Ollama Cloud — a single flag like `needsBaseUrl`/`requiresAuth` rather than scattered id checks) also get a **"Check models"** action and an auto-check after a successful connection test that runs `probeModelAccess` (`@/lib/ai`) on the two selected models — ollama.com lists its whole cloud catalog regardless of plan, so a selectable model can still 403 ("requires a subscription") at chat time. The probe test-calls each model and lists any the account can't run (each with its own reason) in a `role="alert"` with an upgrade link, or shows a count-aware `role="status"` all-clear. A monotonic `probeIdRef` guards against a late-resolving probe overwriting state after the selection/provider changed; the result is cleared (and the id bumped) on any field edit or provider switch, and the auto-check skips when the current selection already has a result so re-testing doesn't re-spend probe calls.
  - `play-dashboard.tsx` — Play page dashboard (new game, character creation, resume, manage). The home view is a single **Start New Game** CTA above one **"Your Characters"** roster (no separate "Continue" box — the old Continue + Character Summary sections were redundant and both capped at 3): every saved character is a clickable card (`handleContinue` → `saveActiveSessionId` makes it the active character everywhere, then opens the `GameLoop`), **paginated** at `PAGE_SIZE` (6) per page with Previous/Next controls and a `role="status"` "Page X of Y" indicator; `safePage` clamps the page during render so a shrinking roster (e.g. after a delete) never points past the end. Seeds each new chronicle with permadeath legacy facts (issue #12) and **timeline echoes** (issue #31): artifacts under `ECHOES_KEY` from fallen predecessors may start in the inventory (`pickStartingEcho` → `artifactToItem`, paradox-guarded to the chosen epoch or earlier) with narrator foreshadow facts via `echoFacts`. **Start archetypes (issue #131):** the creation flow passes a single discriminated `StartSelection` (`random` | `location` | `archetype` | `custom`); `handleStartNewGame` resolves a curated preset via `getStartArchetype` or a player-authored circle via `buildCustomArchetype(circle, epoch)`, threads it into `createDefaultGameState(..., archetype)` (location/opening/grounding), then applies `seedArchetype` to the created session (tracked-ally roster, optional society pre-membership, relationship facts); the plain location pick is skipped when a circle is chosen. **Canon-character takeover (issue #92):** `handleStartNewGame` resolves a canon preset from EITHER the GUIDED variant `start.kind === "canon-takeover"` (via `getCanonCharacter(canonCharacterId)`, which also threads the canon-faithful prologue's `prologueRecap` as the recap override) OR the implicit end-of-creation `matchCanonCharacter(characterName, pathwayId)` match (epoch-gated, the manual-path fallback) — on a hit it builds the save via `createCanonCharacterSession(preset, …, recapOverride)` and returns early, SUPERSEDING the generic creation path entirely (canon pathway/sequence/location/timeline + prefilled durable backstory + personality) while still layering the same world-continuity facts (legacies via `withLegacyFacts`, echoes + carried artifact); on a miss the existing path is unchanged. A **Manage characters** view (reached from the roster header) lists every saved character with a two-step confirm **Delete** that runs `characterDeletionPlan` (wipes the local save, journal, in-progress combat, and usage estimate) plus a best-effort `deleteSessionEntriesRemote` (the durable Supabase journal rows); cross-timeline legacies/echoes are left intact by design. When `devToolsEnabled()` (`NEXT_PUBLIC_DEV_TOOLS`), the home view also shows a **"Seed test character"** button (`createTestCharacter` → persist + activate) for fast scene-art testing; that character (fixed `TEST_CHARACTER_ID`) is protected from deletion — the Manage view shows a "Protected" label instead of Delete, and the character sheet's danger zone hides for it (`isUndeletableCharacter`).
  - `page-header.tsx` (`PageHeader`) — the shared standard header for every game route: an eyebrow kicker, a serif title, an optional description, an actions slot, and the hairline rule. Used across the game pages (e.g. `play-dashboard.tsx`) so each screen reads in one consistent voice.
  - `creation-ui.tsx` (`StepHeader`, `ChoiceCard`) — the shared presentational primitives for the character-creation flow (pure `.tsx`, outside the coverage gate; verified via the axe suite). `StepHeader` is the one consistent step header (a single "← Back" affordance + the shared dot/`Step X / Y` progress model) used by every step, replacing the per-step ad-hoc back buttons and the two competing progress indicators. `ChoiceCard` is the one "pick one of these" primitive (title + description + optional `badge`/`meta`/`accent`, `selected` → `aria-pressed` toggle, `tone="occult"` for sealed-origin/advanced choices) used across the whole flow — path select, era select, the grouped pathway select, and the start-opening chooser — so every choice reads in one visual language instead of a mix of bespoke cards, plain buttons, and native `<select>` dropdowns whose options couldn't show their own descriptions.
  - `character-creation.tsx` — Multi-step character creation flow. Two paths: (1) AI-driven prologue: AI-generated scenes that each offer 3–5 choices, each LEANING toward 1–2 neighboring pathways within one canonical region (issue #119 — never a clean 1:1 "the X option"), with each scene branching across ≥2 regions. The engine accumulates the picked choices' blended affinity weights via `tallyAffinities` and computes the finale candidate potions via `selectTopCandidates` (deterministic — the AI never decides the pathway; the pathway emerges from the pattern of picks and can land on any of the twenty-two playable pathways). A separate `generatePrologueFinale` call renders those top candidates as potions; the player's pick (`dominantAffinity` of the chosen potion) sets the pathway. The picked finale potion is NOT fed back into the tally. An **epoch picker** (issues #26/#29) on mode-select — a browsable row of era `ChoiceCard`s (not a dropdown), so each era's summary is visible before selecting — chooses the starting era (all five epochs; Fifth default — it sets the narrator's epoch tone layer). A **start-opening chooser** (varied story openings) on the final first-potion step chooses where the chronicle opens: `startLocationsForEpoch(epoch)` lists the epoch's distinct places (Tingen/Backlund/Trier/Bayam plus the farther regions Pritz Harbor/Enmat Harbor/Feysac); places that thematically suit the chosen pathway are flagged with a visible "Suits the {pathway}" badge — a suggestion only (never auto-applied). The chronicle-start choice threads through `onComplete` as a single discriminated `StartSelection` (exported from this file: `random` | `location` | `archetype` | `custom`) → `play-dashboard` → `createDefaultGameState`, which sets `GameState.location` + `GameState.openingBeat`. **Start chooser UI (redesign — `ChoiceCard`/`StepHeader` from `creation-ui.tsx`):** the start-opening control is a **two-level card chooser**, not a single crammed `<select>`. A UI-only `openingMode` (`random` | `place` | `circle` | `custom` | `origin`) row of family tiles reveals the matching browsable option cards; `startChoice` (the single `startChoice` string state, reset on epoch change so a pick can't leak across epochs) is still the one source of truth `handleBeginChronicle` resolves — the cards just set the same encoded values the dropdown used to (`""` random, bare `location`, `archetype:<id>`, `origin-scenario:<id>`, `"custom"`). **Start archetypes (issue #131):** the **"Within someone's circle"** family lists `startArchetypesForEpoch(epoch)` curated PRESETS as cards (label + blurb + suits-badge, value `archetype:<id>`); the **"Author your own circle"** family (value `custom`) reveals an inline form — a free-text **tie**, a **companions** add-control (chips; a `circleNpcSuggestions(epoch)` datalist suggests canon NPCs but ANY invented name is accepted), and a **begins-in** location. The custom form's fields build a `CustomStartCircle` → `buildCustomArchetype` in play-dashboard, so creativity is never capped by the preset list; curated archetypes are just examples. **Origin starts (issue #132):** an opt-in checkbox ("Begin as a native of a sealed, far-off land") reveals a gated occult-tinted **"A sealed origin"** family of `forsakenLandStartsForEpoch`/`forsakenLandArchetypesForEpoch` cards (the City of Silver, value `origin-scenario:<id>` / `archetype:<id>`), adding the `origin-scenario` variant to `StartSelection`; unchecking clears a now-hidden origin pick. The manual-path pathway list is **grouped by pathway family** (`PATHWAY_GROUPS`) so all twenty-two are scannable. Every "pick one of these" surface (path / era / pathway / start-opening) shares the one `ChoiceCard` primitive, and each step shares one `StepHeader` (single back affordance + one progress model). play-dashboard resolves an origin scenario via `getStartScenario`, so `createDefaultGameState` seeds the dream-world passage. (2) Manual path: direct pathway selection from `ALL_PATHWAYS` + character sheet. Both the AI prologue and the manual path reach **all twenty-two** playable pathways (issues #119/#120); the becoming-scene content (`FIRST_POTION_NARRATIVE`/`POTION_HEADINGS`) and the manual-card `PATHWAY_DESCRIPTIONS` (all imported from `@/lib/game`'s `prologue.ts`) cover the full 1–22 id set. Both paths end with the first-potion narrative scene. On "Begin Your Chronicle" the AI path also composes a durable **prologue recap** (`buildPrologueRecap` — defining choices + finale encounter + the potion drunk) and threads it through `onComplete` → `play-dashboard` → `createDefaultGameState` into `GameState.prologueRecap`, so the main narrator (which never sees the prologue prompt) continues the thread seamlessly; the manual path passes an empty recap. **Canon-character takeover prologue (issue #92):** in `character-setup`, when the typed name matches a known novel figure (`matchCanonCharacterByName`), an opt-in "Take over {name}'s story" affordance appears; accepting locks `selectedPathwayId` + `epoch` from the preset and routes to a new `canon-prologue` step that runs the canon-faithful, AI-driven prologue (`generateCanonPrologueScene`/`generateCanonPrologueFinale` from `@/lib/ai`, seeded with the figure's background/pathway/`personalityTraits`/`becomesOnScreen`). Scene choices are plain (the pathway is fixed); the finale ends on the becoming potion (becomes-on-screen figures) or an arrive-at-first-appearance hand-off (already-Beyonder figures). Finishing threads the prologue's `createAIPrologueMemory` + `buildPrologueRecap` through the existing `onComplete` `initialMemory`/`prologueRecap` params with `start = { kind: "canon-takeover", canonCharacterId }`. A skippable **tutorial introduction** (issue #14, `TUTORIAL_SCENES` from `@/lib/game/tutorial.ts`) is offered on mode-select for newcomers — four hand-authored scenes (Beyonders & secrecy, potions & the fatal risk, the acting method, pathways) that flow back into mode-select; veterans skip without losing anything. The freeform **background** textareas (both paths) are capped at a file-local `MAX_BACKGROUND_LENGTH` (1200 — raised from 280 in issue #92 so a richer, prefilled-style backstory fits; canon-takeover presets supply their own much longer durable backgrounds out of band).
  - `game-loop.tsx` — Core game loop UI (situation/choices/resolution/consequences phases). **One narration per player action (action-assumption fix):** a normal turn resolves the action AND presents the resolution's OWN choices inline as the next decision — `resolveChoice` commits the turn via the shared `applyAndCommitTurn` (every `applyResolution` + bookkeeping side-effect) then transitions `resolution → choices` (`PRESENT_NEXT_CHOICES`), so there is NO separate forward-looking `generateSituation` scene that could advance the story or act for the player without a prompt. On the choices screen, when `session.lastResolution` is set, a `ResolutionRecap` (the outcome narrative + sanity/acting-digestion/world/items summary + key-moment scene art, keyed to the committed turn) renders above the choices, and `ChoicesPhase` drops its own narrative block; a fresh scene (turn 0 / after an engine turn) has no `lastResolution` and reads as before. `generateSituation` now runs only for the opening scene and to resume after an engine turn. The "Continue" `ConsequencesPhase` (= `ResolutionRecap` + Continue) is reached ONLY by an engine-decided turn (advancement / apotheosis / combat via `ENGINE_RESOLUTION`, committed through the same `applyAndCommitTurn`). A `StoryChronicle` (`story-chronicle.tsx`) of recent memory beats renders above the live scene so the turn reads as a continuous chat; combat is routed back through `ENGINE_RESOLUTION` (`handleCombatResult`, like `handleAdvancement`) so a fight's narration becomes the consequences scene + a memory turn (no stale pre-combat scene, no disconnected side-screen). The whole surface is wrapped in `SanityEffects` so CSS distortion tracks the (hidden) sanity meter. The status bar shows the sequence/pathway, location, the **epoch label** (`getEpoch(...).name`, character epoch isolation), turn count, the potion-digestion meter (issue #95: shown only once the acting method is discovered AND `digestionMeterVisible` is enabled — `resolveActingMethodState(session.actingMethodState).knowsMethod`) and — only when the player has enabled it in Settings (`sanityMeterVisible`) — the numeric sanity meter; otherwise sanity/digestion are communicated purely through the visual effects and vague prose. The `ResolutionRecap` mirrors this: pre-discovery only vague `digestionFeedback` prose shows (alignment %, digestion delta, and the AI's reasoning are gated on `knowsMethod && digestionMeterVisible`), the sanity number follows `sanityMeterVisible` (qualitative descriptor otherwise) from a locally-recomputed hybrid breakdown. The shared `applyAndCommitTurn` (called by `resolveChoice` for a normal turn, `handleContinue` for an engine turn) threads `resolveActingMethodState(...)` into the `applyResolution` signature, persists `result.actingMethodState`, and announces a discovery via a `role="status"` notice. It also threads `gameState.epoch`, `resolveTrackedNpcState(session.trackedNpcState)`, and `preferences.movementGateEnabled` into `applyResolution` (issue #101) so the movement gate runs each turn — a refused cross-city teleport leaves the location unchanged and lands its redirect fact in memory, with no special UI. `buildAICallParams` threads the character's `epoch` AND `sequenceLevel` into `selectCuratedLore` so the narrator's curated lore is epoch-gated (no crossover) and sequence-gated (progressive disclosure — deeper pathway rungs unlock as the character advances). Each AI turn (`generateSituation`/`resolveChoice`) also calls `retrieveLoreForTurn` (`lore-retrieval-client.ts`) best-effort and passes the gated novel/wiki chunks as `generate({ retrievedChunks })`; retrieval never blocks a turn (resolves to `[]` when no embedding endpoint/corpus is available).
  - `lore-retrieval-client.ts` (`retrieveLoreForTurn`) — the network glue for gated RAG retrieval (issues #63/#64): resolves an embedding transport (the player's own Ollama first when it's their chat provider — localhost embeds free/private; else a self-hosted operator endpoint; else the Cloudflare Workers AI proxy as the general fallback when `NEXT_PUBLIC_CLOUDFLARE_EMBEDDING` is set) and the authenticated Supabase `match_source_chunks` RPC, runs the timeline/epoch/concealment-gated `retrieveChunks` (query + chunk shaping from `@/lib/game`; concealment ceiling from the character's sequence), and returns `[]` on any failure. Pure decisions live under the coverage mandate; this thin shell does not. The consequences panel previews the digestion change from the acting evaluation and surfaces a completion event when a potion is fully digested. When sanity bottoms out (`isLossOfControl`), a `FailurePanel` (issue #12) presents the verdict: a **setback** is endured in place (`applySetback` — sanity sliver, lost items, displacement, reputation memory fact; transient consequence report) while **permadeath** runs `handlePermadeath` — legacy persisted under `LEGACIES_KEY`, an artifact echo minted under `ECHOES_KEY` (issue #31), a `death` journal entry recorded, the descent narrated by the AI best-effort (deterministic `fallbackDescentScene` otherwise), and the session marked `ended`. An ended session renders the `DeathScreen` (the chronicle stays readable as a historical record) with two paths: _begin anew in this world_ (legacies persist — `play-dashboard` seeds them into the next character's memory via `legaciesToFacts`) or _restart the timeline completely_ (wipes `LEGACIES_KEY` + `ECHOES_KEY`, canonical baseline). A session **token-usage estimate** (issue #15) accumulates each `generate()` call's rough prompt/output token counts (`@/lib/ai` `usage.ts`; persisted under its own `lotm:usage:` localStorage key) and renders in the status bar with an order-of-magnitude cost. The choices phase is **mode-aware** (issue #24): `detectInputMode` infers exploration/ritual/dialogue/investigation from the scene and a `ModeAssist` block renders the matching guided interface (ritual steps + inventory material chips + petition; speak-with/ask-about; two-clue deduction over `gatherClues`) — every guided action composes into the validated free-text pipeline. The choices phase also offers an optional **free-text action box** (issue #19): input is validated by `validateFreeText` (exploit declarations rejected with in-world narration via a `role="status"` line), then wrapped via `freeTextToChoice` and dispatched through the normal SELECT_CHOICE machinery. **Assuming a persona** (issue #22) is a deliberate engine action, not narration: an `IdentityActionPanel` in the choices phase lets the player slip into / out of any prepared face via `switchIdentity` (the same call the character sheet uses, persisted through `updateSession`; the narrator picks up the worn persona via `identityPromptContext` next turn). It renders nothing when no face has been prepared (no per-turn nag). The "you have prepared none" steer surfaces only when the player actually reaches for one: `handleFreeText` guards the box — a typed assume attempt (`detectAssumeIdentityIntent`) is short-circuited to a `role="status"` line (`ASSUME_VIA_PANEL_NARRATIVE` when faces exist, `UNPREPARED_IDENTITY_NARRATIVE` otherwise) instead of being sent to the AI, so it can't silently fail to associate and only pollute the context. Once the current potion is fully digested and the rung is advanceable (Seq 9-2) the choices phase offers the **unified `TheClimb` flow** (issue #171, gated on live state — `digestion.complete` + `isAdvanceableSequence`, so an existing save that drifted ahead of the engine recovers simply by loading, with no destructive migration): one illuminated header ("Ascend to {role}") + a stage stepper (Secure the recipe → Gather the ingredients → Perform the rite → Make the climb, current stage `aria-current="step"`) framing the three stage panels below it as ONE progressive ascent rather than three disconnected cards. First a `PotionPreparationPanel` (issue #84): the next potion's `potionPreparationPlan` checklist where each missing prerequisite is **bought** with funds or — for the Beyonder Characteristic — **hunted**. The **formula is the canon gate** (issue #171): until the recipe is in hand the ingredients render **sealed** (locked), and the formula itself is secured one of two corpus-true ways — **trade for it** (`handlePurchaseItem`, funds-gated) or **seek it through the story** (`handleSeekFormula` → `beginFormulaPursuit`, a tracked multi-turn pursuit on `session.formulaPursuit` advanced each turn in `handleContinue` via `advanceFormulaPursuit`; once ready, `handleSecureFormula` → `secureFormulaThroughStory` claims it for free). Either route narrates the gating "recipe secured" beat through the shared `narrateFormulaSecured` helper — routed through `ENGINE_RESOLUTION`, stamped `kind: "advancement"` so the chronicle styles it as a climb beat. Hunting now starts a tracked, multi-turn QUEST rather than jumping to combat: `handleHuntItem` → `startHunt` registers a `HuntState` on `session.hunts` (several hunts can run at once) and adds an AI-visible quest label; each turn `handleContinue` calls `advanceActiveHunts` to close the distance (scaled by rung depth), and once cornered the player engages the unified combat system (`handleEngageHunt` → `startCombat(false, itemName)`, which stamps `CombatEncounter.huntTarget` so `handleCombatResult` grants it via `deliverHuntedItem` on victory and then `clearHunt`s the quest). deeper rungs cost more and the Characteristic becomes hunt-only, so this is the route that makes advancement actually attainable (inventory is not AI-mutable). A general **`QuestLogPanel`** in the choices phase surfaces ALL tracked quests in one place — the AI-narrated story quests in `gameState.activeQuests` plus the structured hunts with a tracking `role="progressbar"`, an "Engage the quarry" button when ready, and "Abandon the hunt". When everything is in hand the prep panel hides and the `AdvancementPanel` takes over: the engine's `advancementRequirements` checklist (canon ingredients and, from Sequence 5, the Advancement Ritual) and a two-step arm-then-confirm attempt; `attemptAdvancement` either advances the sequence (the engine writes `sequenceLevel`, re-seeds digestion, and the climb is **routed through the normal turn loop via `transition(..., { type: "ENGINE_RESOLUTION", ... })`** — the `"advancement"` narration becomes the inline `consequences` scene and a memory turn record, so the narrator stays aware, with no side-channel banner) or yields a `lost-control` verdict routed through the shared setback (`applySetback`) / permadeath (`concludeChronicle`) paths — a loss of control is always possible and worse at the higher rungs. At Sequence 1 the choices phase also offers an `ApotheosisPanel` (issue #30): the engine's requirement checklist (`apotheosisRequirements`) and a two-step arm-then-confirm attempt; on success the ascent is routed through the same `ENGINE_RESOLUTION` turn (the tease renders inline + is remembered), on failure `attemptApotheosis` yields a catastrophic verdict that runs through the shared `concludeChronicle` permadeath path. A Sequence 0 True God's per-turn worshipper petitions (`drawPetition`) fold into session memory in `handleContinue`, and `buildAICallParams` swaps in `TRUE_GOD_ABILITIES`/`TRUE_GOD_ACTING` (Seq 0 has no rules `Sequence`). A `CombatLauncher` in the choices phase starts a combat encounter (`createEncounter`, enemy derived from the scene); an active encounter (persisted under its own `lotm:combat:` localStorage key, separate from the session) takes over the surface via `CombatEncounterView`, and on resolution `applyCombatResult` writes sanity/inventory/injuries back to the session.
  - `combat-encounter.tsx` (`CombatEncounterView`) — Combat UI (issue #10): an intel-gated **`EnemyPanel`** dossier (name + description always, plus rough strength / pathway / exact sequence / known abilities revealed by the player's intelligence level via the pure `enemyIntel`), a preparation form (intelligence, terrain, two disjoint inventory pickers — **ritual materials** for ordinary items/gear/weapons and **sealed artifacts** scoped to the `sealed-artifact` category only, so loot can't be readied as an artifact to game the prep score — and readied abilities), an exchange of inline tactical decision points — now including **dynamic in-the-moment options** for the player's actual learned abilities and carried artifacts (engine-generated; an artifact option consumes the item on use) — and a resolution screen (outcome, injuries, sanity, items, dropped characteristic, plus an optional `SceneArt` illustration of the fight — issue #20, gated on the player's opt-in + image provider). **Clarity surfaces (issue #187, Phase 5):** the `EnemyPanel` adds a framing card (`framingLabel`/`describeFraming`) and a `ThreatCard` (`threatAssessment` — danger tier + INTEL-GATED odds band); the exchange renders a `ControlMeter` (`role="progressbar"`, the loss-of-control meter from `controlTier`/`controlStrain`), the reactive enemy stance (`stanceLabel`/`describeStance`), and per-option effect + control-strain tags; the resolution renders an itemized after-action report from the pure `afterActionReport` (sanity-with-cause, injuries, items gained/spent/**kept-not-consumed**, control outcome, world ripple). ## Conventions

- Server Components by default. Only use `"use client"` when the component needs browser APIs, hooks, or event handlers.
- Auth forms are client components — they manage form state and call Supabase Auth methods directly.
- Game sidebar is a client component — uses `usePathname` for active route highlighting and `useState` for mobile toggle.
- Provider config, play dashboard, and game loop are client components — manage localStorage for BYOK keys and session persistence.
- Game loop orchestrates AI calls via `generate()` from `@/lib/ai` and state transitions via `transition()` from `@/lib/game`.
- No component libraries. Build with Tailwind utility classes using the Victorian steampunk theme tokens.
- Atmosphere utilities (issue #13, `globals.css`): `.parchment` (paper-grain narrative panels), `.gaslit` (warm halo behind headings), `.fog-overlay` (drifting fog banks, mounted once on the game layout), `.candle-flicker` (uneven living glow). All decorative, reduced-motion-safe, and muted under high-contrast mode.
- Forms handle loading state, error display, and success redirects internally.
- Use `useSyncExternalStore` for initial localStorage reads — avoid `setState` inside `useEffect` bodies (React 19 lint rule).
- **Active character is a single shared pointer** (`@/lib/react/session-store`, active-character sync). Every page shows the SAME character: read it with `useActiveSession()` (full session, reactive to switches AND in-place edits — a mutation just `persistSession`s and the hook re-reads, no local override), `useActiveSessionId()`, or `useSessionSummaries()` (switcher labels). Change it with `saveActiveSessionId(id)` — the sidebar `CharacterSwitcher`, the Character sheet's "Active Beyonder" selector, the Journal's "Chronicle" picker, and `play-dashboard` (auto-activates on new game / Continue) all write this one pointer. `loadActiveSessionId` self-heals a stale/deleted id to the newest save. Do NOT reintroduce per-component "first-in-index" or local-dropdown character selection — that was the desync bug this replaced.
- **Accessibility is required (WCAG 2.2 AA).** Follow `docs/rules/accessibility.md` for every component: accessible names, ARIA state, `role="status"`/`alert`, decorative `aria-hidden`, no low-opacity meaningful text, and ≥24px targets. Add/extend the axe suite in `src/test/a11y.test.tsx` when adding a screen or significant interactive component.

## Testing

- **Logic modules** (the `*.ts` files here, not the `*.tsx` components) are under the coverage gate and have colocated `*.test.ts` tests: `preferences-store`, `scene-art-cache` (an in-test IndexedDB fake), `character-actions`, `lore-retrieval-client`, and `cloud-sync` (a fresh-module-per-test harness; its `@/lib/game`/session-store/Supabase deps are mocked). When you add a new `components/game/*.ts` logic shell, add a colocated test **and** list the file in `coverage.include` (`vitest.config.mts`). The pure decisions these shells call still belong in `src/lib/**`.
- **Components** (`*.tsx`) are not unit-covered by the gate; verify them through the jsdom axe suite (`src/test/a11y.test.tsx`) and, for user-facing flows, an `e2e/` Playwright spec.
